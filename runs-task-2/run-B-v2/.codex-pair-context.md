# Project context for codex-pair (task 2)

This is a small Node.js + TypeScript + Express HTTP service implementing a **URL shortener** that:

- Accepts arbitrary user-supplied URLs via `POST /shorten`
- Generates 6-char base62 short codes with collision retry
- Redirects via `GET /:code` (302) and tracks visit counts per code
- Returns visit stats via `GET /:code/stats`
- Persists state to a JSON file (`shortener.json`)
- Rate-limits `POST /shorten` to 10 requests per IP per minute

**Stated requirements from the spec:**
- Express 5 + TypeScript strict mode + Zod input validation
- `tsc --noEmit` must pass
- `vitest run` must pass
- File writes must be safe under concurrent requests (no lost data)

**Deployment shape to assume when reviewing:**
- Public-facing HTTP service: ANY user can submit any URL, will be persisted, will redirect arbitrary visitors to it
- Concurrent requests are real (multiple users shortening simultaneously, popular short codes being visited concurrently)
- The visit counter on `GET /:code` is a hot path — every visit reads+increments
- Code generation may collide as the table fills; ungraceful exhaustion is a real failure mode
- Rate limiting must actually rate-limit (boundary bypasses defeat the purpose)
- URL inputs are UNTRUSTED user input — any URL scheme accepted by the validator becomes a redirect destination for future clicks

**Threat surface worth checking even if the spec doesn't mention it:**
- Open redirect / scheme injection (`javascript:`, `data:`, `file:`) — a malicious shortener URL becomes a phishing/XSS vehicle
- Rate-limit window bypasses (fixed-window edge: 20 requests in 2s by straddling the boundary)
- Code-generation exhaustion or denial of service via repeated POST
- Counter race conditions distinct from the storage-write race
