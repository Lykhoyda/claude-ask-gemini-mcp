# Architectural Decisions

## ADR-081: `codex-pair` Hook — `.codex-pair-ignore` Gitignore-Style Globs for Granular Per-Project Opt-Out
- **Date:** 2026-05-18
- **Status:** Accepted (extends ADR-077, ADR-079, ADR-080; v0.6.0 phase 2 item #7)
- **Context:** Before this change, the only opt-out mechanism for codex-pair was `CODEX_PAIR_DISABLED=1` — a process-wide kill switch. That granularity is wrong for the common case: a developer working on a project where 95% of edits should be reviewed but `experiments/scratch.ts` should not, must either toggle the env var per shell session or accept noise on scratch work. The skip-patterns list (`/node_modules/`, `/dist/`, lockfiles, etc., expanded in item #3) is hardcoded into the hook and not user-configurable — adding `/experiments/` to the global list would suppress that path in every project, which is wrong. The right abstraction is the well-precedented gitignore pattern: a `.codex-pair-ignore` file in the marker directory with gitignore-style globs, `#` comments, `!` negation. Familiar to every developer, version-controllable, per-project, and stackable with the existing skip patterns. The matcher must be hand-rolled (zero workspace imports per ADR-078; no `picomatch` / `minimatch` dependency). On match, the hook should exit silently — no `systemMessage` (preserving the silent-gating UX for opted-out files) — but still log the skip with the matching pattern so post-hoc analysis can correlate noise reduction with the ignore rules. Three architectural choices to make: (A) where to read the ignore file from — marker directory only, vs walking up like the marker itself, vs project root via git rev-parse. Marker directory is simplest and matches the user's mental model (the ignore file lives next to the marker). (B) When to apply the match — before the file-read (cheap, but means we can't show file-not-found errors for ignored missing files) or after. Before is correct: the entire point of opt-out is to skip work, including the file read. (C) Whether to surface a systemMessage on match — yes (visible "skipped via ignore") or no (silent). The goal command explicitly says NO systemMessage, preserving the silent-gating UX for files the user has consciously opted out of. Logging the skip is enough for forensics.
- **Decision:** Three components. **(1) Ignore-file reader** — `readIgnoreFile(markerDir)` reads `.codex-pair-ignore` from the marker directory if present, returns an array of `{ negate, pattern }` rule objects in order. Lines starting with `#` are comments; empty lines are skipped; lines starting with `!` are negation rules (re-include after a broader exclude). No reading of nested ignore files (Git supports this for `.gitignore` in subdirs; codex-pair's marker is the project anchor so a single ignore file is enough). Errors reading the file (not present, permission denied, etc.) silently return an empty rule list — the file is optional, absence is the common case. **(2) Gitignore-style matcher** — `matchesIgnoreRule(filePath, markerDir, rules)` returns the matching rule (or `null` if no rule matches). Walks rules in order; for each, computes whether the file path matches the glob, then applies the rule's `negate` flag (a matching `!` rule un-ignores). Glob semantics: `*` matches any character except `/`, `**` matches across directory separators, `?` matches a single character, `[abc]` is a character class, leading `/` anchors to the marker directory, trailing `/` matches directory contents. Hand-rolled — pattern is converted to a JS RegExp by escaping regex metacharacters then substituting glob tokens (`*` → `[^/]*`, `**` → `.*`, etc.). The matcher does NOT need to handle every edge case of git's full gitignore spec (the spec includes weird precedence rules around `**/dir/**` patterns and re-include order); the user-facing contract is "the common cases work, the weird cases are documented." Plain extensions (`*.test.ts`), directory excludes (`experiments/`), and inline negations (`!experiments/important.ts`) all work. **(3) main() integration** — between the SKIP_PATTERNS check and the marker-file frontmatter parse, the hook calls `readIgnoreFile + matchesIgnoreRule`. On match: `appendLog` with `verdict: "skipped"` and `reason: "matched .codex-pair-ignore: <pattern>"`, then `process.exit(0)` WITHOUT calling `emitSystemMessage`. This is the only path in the hook where a skip doesn't surface a systemMessage — intentional, the user opted out and doesn't want to see the result. **Tests**: structural (file reader exists, matcher exists, no-systemMessage on ignore match) + runtime (simple glob match, negation re-include, no-match passes through, missing ignore file passes through). Plugin test count expected ~167 → ~174.
- **Consequences:** Granular per-file/per-directory opt-out without per-shell-session env juggling. The ignore file is version-controllable (commit it to the repo) so teammates inherit the rules. The silent-on-match UX preserves "no noise from opted-out files" — exactly what users expect from gitignore-shaped tools. **What this costs**: every reviewed file now goes through an additional glob-match pass against the ignore rules. Per-file overhead is microseconds (regex matching against a typical handful of rules), negligible against the codex call cost. **What this doesn't change**: the hardcoded SKIP_PATTERNS list (node_modules, dist, lockfiles, fonts, etc.) still runs first — it's the cross-project baseline. The ignore file is on top of that, project-specific. ADR-077's threshold-in-hook design is unaffected. **What the matcher intentionally does NOT support**: (a) the full gitignore spec — `**/dir/**` quirks, complex precedence cycles between `!` and broad excludes — these are documented as edge cases where users should use a clearer pattern; (b) reading nested `.codex-pair-ignore` files in subdirectories (one file per project, anchored at the marker dir); (c) the `--no-pager` git-style escape (no need — codex-pair isn't a piped tool); (d) integration with the user's existing `.gitignore` (different concept — gitignore excludes from version control, codex-pair-ignore excludes from review; users who want symmetry can reference their .gitignore manually). **Forward compatibility for item #8 (cache)**: ignored files never reach the codex call, so they're never cached either — no cache pollution.

## ADR-080: `codex-pair` Hook — Adaptive Context Strategy Replaces Silent Skip at File-Size Cap
- **Date:** 2026-05-18
- **Status:** Accepted (extends ADR-077, ADR-078, ADR-079; v0.6.0 phase 2 item #6)
- **Context:** ADR-077's hook design capped per-file review at `MAX_FILE_BYTES = 20_000` (env-overridable, frontmatter-overridable per ADR-079) and *silently skipped* anything over the cap. The cap was the right cost-control move at design time — codex calls are expensive and unbounded file sizes break the cost model — but the silent-skip part was a coverage gap. Real source files at the limit (a typical React component, a moderate util module, the v0.5.0 hook itself at ~440 lines) hit the cap often. Those files got zero review, with no signal to the user that review had been skipped (until ADR-079's structured verdicts made the SKIP visible). Even with the SKIP signal, the user can't act on it — bumping the cap risks running codex on a 2MB file. The right framing isn't "should we increase the cap" but "what do we send when the file exceeds the cap." A partial-view review with explicit context-loss disclosure beats no review on every dimension that matters: codex still gets the imports/types header (where most type-system concerns live), still sees the diff against HEAD (where the actual edit intent lives), and is explicitly told not to speculate about omitted code. Three context-construction strategies were considered: (A) **bare diff** — `git diff -U20 HEAD` only, no other context. Token-cheap but loses imports, top-level types, and the enclosing function for many hunks. Codex would either hallucinate concerns about undefined symbols or skip the type check entirely. (B) **diff + surrounding context** — `git diff -U20` plus the first ~80 lines of the file as a "header" preview. Captures imports and type definitions, the diff captures the change intent. Net token spend is usually well under the cap. (C) **full file plus diff annotation** — send the full file AND a diff annotation. Strictly more tokens than full-file alone; doesn't help with the over-cap case (the point of the exercise). Option B was selected as the primary strategy. For untracked files (no HEAD to diff against), or when git itself is unavailable (binary missing, not a repo), the fallback is **head-150 + omission marker + tail-80** — file structure is usually densest at the top (imports/exports/declarations) and bottom (default exports, last-defined functions), so head+tail captures both bookends. A third strategy ("truncated") handles the rare case of a file with few lines but huge bytes (e.g., one massive minified line) by sending a hard-truncated slice. **A separate ADR for this item is warranted because** it changes the hook's coverage semantics — the over-cap skip was a documented behavior since ADR-077, and the new adaptive behavior changes user-visible logging (new `level: "info"` over-cap entries, no more `verdict: "skipped"` for size). It also introduces a new spawn surface (the hook now shells out to `git`, with its own timeout and error handling) that future items must understand.
- **Decision:** Three components shipped together in the hook. **(1) `runGitDiff({ filePath, contextLines, cwd, timeoutMs })`** — spawns `git diff -U<n> HEAD -- <filePath>` via `spawn` (not exec — no shell interpretation, no command-injection surface) with `stdio: ["ignore", "pipe", "pipe"]`. Hard 5-second timeout enforced via `setTimeout` + `SIGTERM`. Returns the diff string on success (exit 0 with non-empty stdout) or `null` on any failure path: spawn error (`ENOENT` when git binary missing), non-zero exit (not a repo, file untracked, etc.), timeout, empty output. Never throws — `null` is the only failure signal. The `cwd` is the marker directory, not `process.cwd()`, because the marker dir is where the user's project root lives (the marker file is the project anchor). **(2) `buildAdaptiveContext({ filePath, fileContent, markerDir, maxFileBytes })`** — async function with three strategies. First tries `runGitDiff` with 20 context lines. If diff is non-null and under `maxFileBytes` bytes, returns `{ strategy: "diff", content: <header_first_80_lines + diff_against_head> }`. Otherwise checks line count: if `<= 230` lines (i.e., file is small in lines but big in bytes — minified asset, JSON dump, etc.), returns `{ strategy: "truncated", content: <first maxFileBytes bytes + truncation marker> }`. Otherwise returns `{ strategy: "head-tail", content: <head_first_150_lines + omission_marker + tail_last_80_lines> }`. The 80/150/80 line counts are tuned for typical JS/TS files: 80 lines captures imports plus the file's top-level declarations in 95%+ of real files; 150 captures the same plus the start of the largest top-level function; 80 lines from the tail captures default exports and trailing helper functions. Numbers are pinned by structural test and should not be tuned without a benchmark. **(3) `buildPrompt` accepts `partialView: boolean`** — when true, prepends a "this is a partial view" instruction block to the prompt: *"The file is larger than the configured size cap. Only a slice is shown below (file header + git diff against HEAD, OR head + tail). Flag concerns ONLY if they are visible in this slice — do NOT speculate about omitted code. If you can't see enough to judge, prefer NONE over manufactured concerns."* This instruction is the load-bearing piece that prevents the speculation failure mode: LLMs trained to be helpful will, given a truncated payload, infer that the omitted parts probably have problems. The explicit instruction structurally constrains codex to flag only what it can see. **`main()` integration**: replaces the previous "over-cap → silent skip" branch with an "over-cap → buildAdaptiveContext → partialView=true → proceed to codex" branch. Logs an `level: "info"` entry with the chosen strategy ("diff", "head-tail", or "truncated") so post-hoc analysis can correlate codex's response with the context strategy used. **Tests**: 6 new (5 structural + 1 runtime: untracked-file head-tail strategy via a no-git tempdir). The previous item #5 runtime test asserting `verdict: "skipped"` on a 50-byte cap was upgraded to assert the new `level: "info"` over-cap entry with the strategy mentioned. Plugin test count 161 → 167. Runtime test uses `PATH = dirname(process.execPath)` to isolate node from git/codex — node is still findable so the test runner can launch the hook, but git/codex aren't, so the hook's spawns fail fast via ENOENT instead of timing out. Runtime tests stay under 1s.
- **Consequences:** The hook now reviews files that were previously skipped. Coverage strictly increases — every over-cap file gets at least a header+diff review instead of nothing — without raising the per-call token ceiling (the partial view is bounded by `maxFileBytes`). The partial-view instruction prevents the obvious failure mode (speculation about omitted code) by structurally telling codex what NOT to do. **What this costs**: (a) every over-cap edit now spawns `git diff` in addition to (eventually) `codex`. With the 5s git timeout and modern git's diff speed (<100ms for typical files), this is negligible. The git timeout is intentionally short — if git is slow, that's a signal the repo is huge or git is broken, and falling back to head-tail is safer than waiting. (b) The over-cap path now logs an extra `level: "info"` entry per run — slight extra log volume but bounded by the same rotation (ADR-079 / item #1, 1000-entry cap). (c) Codex sometimes produces concerns that don't apply to the visible slice (despite the partial-view instruction). These are categorized as "noise" in the log but don't block the user — they're surfaced via the same systemMessage channel, and the user can dismiss them as "not visible in the slice you sent". **What this doesn't change**: the under-cap path (full file, no diff, no partial-view instruction) is byte-identical to before — ADR-077's quality bar for normal-sized files is preserved exactly. The 20KB default cap is unchanged; users who want a larger cap set `maxFileBytes` in frontmatter (item #5). The hook still always exits 0 (`main().catch` wrapper unchanged); `runGitDiff` and `buildAdaptiveContext` never throw. **Forward compatibility for items #7-#8**: item #7's `.codex-pair-ignore` runs at a different layer (before the file is read at all), so it composes cleanly. Item #8's content-hash cache will key on the FINAL prompt content (which includes the partial-view marker), so partial and full reviews of the same file hash differently — that's correct, since they're different requests. **Out of scope**: (a) symbolic-reference resolution (e.g., "the diff references `User`; here's what `User` is from `./types.ts`") — would require AST parsing or LSP integration, both outside the zero-workspace-imports constraint; (b) configurable context-lines count for the diff (the `-U20` value) — `-U20` was chosen as a good default; if benchmarks show `-U30` is meaningfully better, we add a `diffContextLines` frontmatter key in a future item; (c) using `--ignore-space-change` or `-w` on the git diff to ignore whitespace — could improve signal-to-noise but might also hide real whitespace-significant bugs (Python, YAML); deferred; (d) per-edit-type strategy selection (e.g., "for MultiEdit on a small change, send only the diff; for Write of a new file, send the head") — could be more efficient but adds complexity; current strategy uses file size + git tracking as the routing signal, which is simpler and covers the dominant cases.

## ADR-079: `codex-pair` Hook — YAML Frontmatter Config in Marker File, Hand-Rolled Zero-Dep Parser, `surfaceThreshold` as Sole ADR-077 Opt-Up
- **Date:** 2026-05-18
- **Status:** Accepted (extends ADR-077 and ADR-078; v0.6.0 phase 2 item #5)
- **Context:** Before this change, every codex-pair knob was an env var: `ASK_CODEX_MODEL`, `ASK_CODEX_FALLBACK_MODEL`, `ASK_CODEX_TIMEOUT_MS`, `CODEX_PAIR_MAX_FILE_BYTES`, `CODEX_PAIR_DISABLED`, `ASK_CODEX_LOAD_USER_CONFIG`. Two problems with env-only configuration in a marketplace plugin: (1) per-project pinning requires per-shell-session exports — a developer who wants a 200KB cap on a docs repo and the 20KB default on their other projects has to remember to set the env var on entering and unset on leaving; (2) the configuration is invisible to teammates — the marker file `.codex-pair-context.md` is committed to the repo (and serves as the team's review rubric per the v0.6.0 dogfooding pattern), but the config that drives codex behavior is held in each developer's shell. The two are out of sync by design. A sidecar `.codex-pair-config.yaml` was considered but adds a second committed file, doubles the "did you remember to update both?" risk, and the parser would still need to be hand-rolled (zero-workspace-imports constraint from ADR-078). YAML frontmatter at the top of the existing marker file is the dominant alternative: one committed file carries both the freeform project context (read by codex as prompt input) AND the per-project knobs (read by the hook to drive its own behavior). Pattern is well-precedented (Hugo, Jekyll, Astro, MDX, etc.). The parser must be zero-dependency: the hook can't `import yaml from "yaml"` for the same `ERR_MODULE_NOT_FOUND` reason ADR-078 calls out. A separate ADR for this item is warranted because the change introduces a new configuration *surface* (not just another knob), defines a closed config schema that future items #6, #7, #8 will reference, and establishes the `surfaceThreshold` key as the *only* sanctioned opt-up to the ADR-077 LOW-suppression rule.
- **Decision:** Three components shipped together in the hook. **(1) `parseFrontmatter(content)`** — a ~40-LOC hand-rolled parser using string ops only. Recognizes `---` on line 1 as opener and `^---\s*$/m` as closer; lines between are parsed as flat `key: value` pairs (no nested structures, no arrays, no multi-line values, no `>` or `|` block scalars). Comments are `#` at start of trimmed line OR `\s+#.*$` inline after a value. Values are coerced: `true`/`false` → boolean, integer/decimal regex → number, otherwise string with optional surrounding `"` or `'` stripped. Returns `{ frontmatter, body, malformed }` — `malformed: true` flips only when an opener exists with no closer (user intent looked like frontmatter but failed to parse). **(2) `resolveConfig(frontmatter)`** — applies precedence `frontmatter > env > default` via `typeof` guards on each key (invalid types silently fall through, never throw). Closed config schema: `{ model, fallbackModel, timeoutMs, maxFileBytes, surfaceThreshold }`. Module-level constants provide the env-or-default layer; `resolveConfig` overlays frontmatter on top. **(3) Threshold-aware `buildVerdictMessage`** — accepts `surfaceThreshold` and gates LOW concerns behind `threshold === "low"`. HIGH always surfaces; MED surfaces at `"med"` or `"low"`; LOW surfaces only at `"low"`. Default `DEFAULT_SURFACE_THRESHOLD = "med"` preserves ADR-077's "LOW in log only" rule for projects that don't opt in. **Why `surfaceThreshold: low` is the *only* sanctioned exception**: ADR-077 established threshold-in-hook (not threshold-in-prompt) to keep codex's calibration consistent across projects — every codex call grades HIGH/MED/LOW with the same rubric; the hook decides what to *surface*. Per-project `surfaceThreshold` preserves that — the prompt stays identical, the filter changes. Allowing other surfacing changes (e.g., `surfaceFormat`, `surfaceColor`, custom regex filters) would re-introduce per-project drift in what the user sees, breaking the consistent-calibration property. Schema is intentionally closed; future items #6 (adaptive context) and #7 (`.codex-pair-ignore`) use the marker dir but don't extend this schema. **Tests**: 9 new tests covering structural (parser exists, threshold table, resolveConfig precedence, main() wires both, buildVerdictMessage gates LOW) and runtime (no-frontmatter path unchanged, full-frontmatter `maxFileBytes` overrides default, malformed-frontmatter logs warning + falls back). The previous test asserting `concerns.low.map(...)` must not appear was upgraded to assert the gating structure (`if (threshold === "low") { ... concerns.low ... }`) — the prior regex-only check had been giving false confidence (a `for...of` loop bypassed the `.map` pattern while still violating the spirit). Plugin test count 152 → 161.
- **Consequences:** The marker file is now both the team's review rubric and the per-project codex-pair config — single committed artifact, version-controlled with the codebase. Env-var sprawl drops from six to two (the kill switch `CODEX_PAIR_DISABLED` and the `ASK_CODEX_LOAD_USER_CONFIG` escape hatch); everything else flows through the schema. Future hooks should follow this pattern: frontmatter in the marker, not yet-another-env-var. **What this costs**: the hand-rolled parser is ~40 LOC of subtle string-handling code, intentionally narrow (no nested structures, no arrays, no block scalars). Anything more would require a real YAML library (violates ADR-078) or risks parser bugs. If a future item needs nested config, the answer is a flat key with a dotted name (e.g., `cache.ttl: 600`) handled inline at `resolveConfig`, not extending the parser. The malformed-warning path is intentionally non-fatal: a typo in the marker silently falls back to defaults plus a log entry; ADR-077's "never break Claude's flow" invariant is preserved. **Schema versioning**: not encoded yet. The current minimal set is the v0.6.0 baseline; items #6-#8 may add keys (e.g., `adaptiveContext: true`, `cache.enabled: false`) and they'll just be ignored by older plugin versions (forward-compat by construction since unrecognized keys fall through `resolveConfig`'s `typeof` guards). **Drift with `codex-pair-defaults.json`**: the JSON ships canonical defaults for `model` and `fallbackModel` (per ADR-078's drift guard); frontmatter overrides them at runtime. The two surfaces stack: JSON is the cross-project default, env is the per-shell override, frontmatter is the per-project pin. The structural test linking JSON to `codex-mcp/src/constants.ts:MODELS` continues to enforce that invariant; frontmatter is per-project and not pinned by any test (intentional — users can pin to any valid codex model). **Marker-file dogfooding**: in the same v0.6.0 batch, `.codex-pair-context.md` was committed to the ask-llm repo itself with project context calibrating codex against this codebase's invariants (zero-workspace-imports, exit-0, ADR-077, sync-blocking). The hook now fires on every Edit/Write/MultiEdit in this repo — the v0.6.0 work itself is reviewed by the production v0.6.0 hook. **Forward compatibility for items #6-#8**: each will read the same `frontmatter` object but use it for additional behaviors (item #6 may add `adaptiveContext: true|false`; item #7 reads `.codex-pair-ignore` separately, no schema change; item #8 may add `cache.ttl` and `cache.maxEntries`). The schema-closed-but-extensible property is the asymmetry that makes this work: today's parser ignores unknown keys (forward-compat), tomorrow's `resolveConfig` adds new defaults (backward-compat). **Out of scope**: (a) nested schema with proper YAML semantics — would require a real parser; (b) explicit `schemaVersion` key — premature for v0.6.0; (c) JSON Schema validation — overkill for ~5 keys; (d) per-file or per-glob config overrides within one marker — that's the territory of `.codex-pair-ignore` (item #7) for its narrow use case but not generalized to config; (e) live config reload — the hook is short-lived per invocation, every run re-reads the marker, no reload mechanism needed.

## ADR-078: `codex-pair` Hook Inlines `codex exec` Spawn — Marketplace Install Doesn't Resolve Workspace Imports
- **Date:** 2026-05-18
- **Status:** Accepted (fixes a P0 regression introduced by ADR-077)
- **Context:** ADR-077 shipped the `codex-pair-watch.mjs` hook with `import { executeCodexCLI } from "ask-codex-mcp/executor"` for the stated reason "single source of truth for codex invocation." That reasoning was correct in isolation but missed how the plugin actually reaches users. The plugin ships via `marketplace.json` as `source: git-subdir, path: packages/claude-plugin` — Claude Code clones the repo and uses only that subdirectory of the monorepo. **No `npm install` runs**, no `node_modules/` is materialized, and the workspace symlinks that make the import work in local dev don't exist on a marketplace install. Live-reproduced from the actual installed copy at `~/.claude/plugins/cache/ask-llm-plugins/ask-llm/0.4.0/scripts/codex-pair-watch.mjs`: every Edit/Write/MultiEdit triggered `Error [ERR_MODULE_NOT_FOUND]: Cannot find package 'ask-codex-mcp'` and `exit=1`. Critical detail: top-level ESM `import` statements execute at module-load time, **before** `main()` runs and **before** `main().catch(...)` installs its safety net. So the hook's exit-0-on-every-failure-path contract was structurally unenforceable for this class of error — Node's module loader threw before any user code could intercept. The blast radius covered every project on every marketplace install, including projects that never opted in by creating the `.codex-pair-context.md` marker file, because the failing import precedes the gate check. Severity: every edit by Claude surfaces a Node stack trace as a hook failure to the user. The same latent issue exists in the plugin's `dist/codex-run.js`, `dist/run.js`, `dist/ollama-run.js` CLI binaries (all import workspace executors), but those are user-invoked and the plugin `dist/` directory isn't even committed (root `.gitignore` excludes it), so they ship as no-ops rather than active failures. The auto-firing hook is the first place the bug actually matters in production.
- **Decision:** Inline a minimal `child_process.spawn("codex", [...])` directly in `scripts/codex-pair-watch.mjs` with zero workspace imports. The hook becomes a self-contained `.mjs` file that runs after `git-subdir` extraction with no install step. Implementation faithfully mirrors `packages/codex-mcp/src/utils/codexExecutor.ts`: (a) `buildCodexArgs(model)` returns `["exec", "--skip-git-repo-check", "--ephemeral", "--ignore-user-config", "--ignore-rules", "--sandbox", "workspace-write", "--json", "-m", model]` — same flag set as the MCP executor (`ASK_CODEX_LOAD_USER_CONFIG=1` opt-out preserved). (b) Prompt is **always** sent via stdin, never as positional argv — this is load-bearing: codex without an explicit prompt arg AND with an open stdin probes stdin until EOF; with a positional arg AND an open stdin it hangs at `"Reading additional input from stdin..."` indefinitely (first-hand observation: a 5-minute test produced exactly 39 bytes of output, then nothing). The fix is the pattern from `commandExecutor.ts`: `stdio: ["pipe", "pipe", "pipe"]`, `child.stdin.on("error", () => {})` to swallow EPIPE, write prompt, call `child.stdin.end()` to send EOF. Cannot use `"ignore"` (connects stdin to `/dev/null`) because some codex versions error during their stdin probe. (c) JSONL parsing inlined from `parseCodexJsonlOutput`: walks lines, picks the last `item.completed` whose `item.type === "agent_message"`, throws on `type: "error"` events when no agent_message arrived. (d) Quota fallback inlined: `isQuotaError(err)` checks the message against `["rate_limit_exceeded", "quota_exceeded", "429", "insufficient_quota"]` (same signal list as `ERROR_MESSAGES.QUOTA_SIGNALS`), and on hit retries once with `FALLBACK_MODEL` (default `gpt-5.5-mini`); the result records `fellBack: true` in the log entry. (e) Timeout: `ASK_CODEX_TIMEOUT_MS` env (default 800_000ms — same default as the MCP executor), enforced via `setTimeout` → `SIGTERM` → `setTimeout(5000)` → `SIGKILL` escalation — same pattern as `commandExecutor.ts` so we don't leave codex zombies on timeout. The `main().catch(...)` wrapper now legitimately catches every reachable failure path because there are no top-level workspace imports left to fail at load time. **Tests**: removed the structural assertion "imports executeCodexCLI from ask-codex-mcp/executor" — that contract has flipped sign. Added five new structural assertions: (i) zero workspace imports anywhere in the script (matches `ask-codex-mcp`, `ask-gemini-mcp`, `ask-ollama-mcp`, `@ask-llm` — any of which would re-introduce the bug); (ii) inline `spawn("codex", [...])` with the expected flag set; (iii) stdin write+end+error pattern present; (iv) quota fallback preserved (FALLBACK_MODEL + rate_limit_exceeded signals); (v) SIGTERM/SIGKILL/ASK_CODEX_TIMEOUT_MS all referenced. All 6 runtime-behavior tests continue to pass unchanged (they were already runtime-spawn tests that exercised the gate logic, not the executor pathway). Total plugin tests 136 → 140 (net +4 from this change). **Live end-to-end verification**: copied just the new `.mjs` to a fresh tmpdir (no node_modules, simulates marketplace install), created a marker file describing a money-handling service, edited a tiny float-money TypeScript file, invoked the hook with a synthesized payload. Result: codex ran in 19s, returned a HIGH concern correctly identifying the float-money violation against the project-context requirement, the hook surfaced it to stderr in the `[codex-pair]` format, wrote the structured log entry, and exited 0. Cost ~$0.05 for the run.
- **Consequences:** The codex-pair feature actually works on marketplace installs now. The previous v0.4.0 was structurally broken on every marketplace install (the auto-firing hook crashed on every edit even for projects that hadn't opted in, because the import error precedes the gate check). v0.4.1 bumps the plugin minor-patch and ships the fix; users on v0.4.0 should upgrade or the feature is unusable. **What the inline-spawn approach costs**: the "single source of truth for codex invocation" claim from ADR-077 is no longer true — the hook now duplicates ~50 LOC of executor logic (buildArgs, parseJSONL, quota fallback, timeout management). Drift risk: if `codexExecutor.ts` changes the flag set or JSONL shape, the hook must be updated in parallel. Mitigation: structural tests pin the flag list and the stdin pattern, and the comment block above each inlined function points at the source it mirrors so future maintainers know what to copy across. **What the inline-spawn approach gives up** that bundling (esbuild) or vendoring would have preserved: the shared `responseCache`, `Logger`, structured `UsageStats` reporting via `@ask-llm/shared`. The hook doesn't need any of these — it's a fire-and-forget reviewer that writes its own JSONL log; per-call cache wouldn't fit the use case (every file edit is its own prompt); and the rich UsageStats are MCP-server-facing, not user-facing. The simplification was net-positive in this case. **Out of scope for this fix**: (a) **The latent same-bug in the CLI binaries** (`run.js`, `codex-run.js`, `ollama-run.js`) — they have the same workspace-import pattern but ship as no-ops because the plugin `dist/` isn't committed, so they're not a production failure today. Filed as follow-up: either drop the `bin:` entries to remove false marketing, or convert all three to standalone `.mjs` scripts shipped in source like the hook now is. (b) **Bundling with esbuild as a build step** (Option A) — would keep the executor as single source of truth, but adds a bundler dep and requires the marketplace install to either run the build or ship pre-built artifacts. The hook's spawn requirements are simple enough that the direct inline is clearer, and the precedent set here is "Claude-Code-side scripts in the plugin should be self-contained `.mjs` files" — same shape as `pre-commit-review.sh` for the gemini hook. (c) **Vendoring `codexExecutor.ts`** (Option C) — strictly worse than inlining: it still creates two copies, but with a more elaborate file layout to maintain. (d) **Asking Claude Code to run `npm install`** (Option D) — would require an unknown amount of platform work; the inline-spawn fix doesn't depend on it. (e) **Telemetry of how often the quota-fallback path actually fires** — would inform whether to keep the fallback complexity in the hook at all; privacy-deferred to a future ADR. **Process-level lesson**: ADR-077's testing strategy verified the gate logic in isolation but never tested the script from outside the workspace symlink tree. The `live-extracts-to-tmp` verification step that proved this fix correct should become standard for any future plugin-shipped Node script that touches workspace imports — if it can't run from `/tmp/`, it can't run from `~/.claude/plugins/cache/`. The structural test "zero workspace imports anywhere in the script" pins this for the hook going forward.

## ADR-077: `codex-pair` PostToolUse Hook — Recall-First Complement to `/codex-review`, Opt-In via Marker File
- **Date:** 2026-05-16
- **Status:** Accepted (extends `@ask-llm/plugin`; closes the four-task POC documented on branch `experiment/codex-pair-poc`)
- **Context:** The existing `/codex-review` skill + `codex-reviewer` agent provide on-demand Codex review of diffs. Its prompt is calibrated for high precision (confidence ≥ 80, "don't flag style/linter-catchable") — appropriate for routine PR review where false alarms cause alert fatigue. A four-task benchmark over two days (todo CRUD, URL shortener, JSON Patch RFC 6902, shopping cart) found this calibration structurally *suppresses* a specific class of bug: domain-level "wrong but won't crash" issues. The single most damning data point came from task 4's three-arm comparison: against the same Run-A natural-Claude shopping-cart implementation, `/codex-review` (one call, 23s, ~$0.04) caught 7 of 10 differential probes; a recall-first per-file hook caught 10 of 10 (five calls, 92s, ~$0.20). The three probes `/codex-review` missed were exactly: float-money precision (`$1.99 × 3 + 8% tax = $6.4475999999…` — every transaction silently mispriced), `setQuantity` validation bypass (the tightened schema didn't cover a separate write path in `cart.ts`), and discount clamping (fixed discount $200 against $100 cart produced `tax=-$8`). Each is a "this is wrong but the runtime won't crash" issue that the agent's "confidence ≥ 80" filter and "don't flag style preferences" filter together classify as below threshold — yet each would ship to production unnoticed. The pair-mode prompt's HIGH/MED/LOW threshold (with the surface threshold living in the hook, not the prompt) catches them. Critically, the diagnostic of which arm is "right" is wrong-framed: they catch *different classes* of bug, not the same class with different completeness. `/codex-review` catches compile errors, obvious logic bugs, and high-confidence security; codex-pair adds domain-level correctness, cross-cutting validation gaps, and subtle edge cases. They are complementary, not competing. Three architectural shapes were considered: (A) **new package** `packages/codex-pair/` — clean separation but duplicates infrastructure (codex executor, project-context pattern, JSON output parsing) and adds a marketplace entry; (B) **extend the codex-reviewer agent with a `mode` parameter** — single agent file, two prompt branches — but muddles the two contracts and prevents the per-edit hook trigger (agents run in Claude Code's subagent context, hooks run as separate processes); (C) **new hook in the existing plugin** alongside the existing review skill — ship both surfaces side-by-side with documented selection criteria. Option C was selected: the new functionality is a hook, hooks are a Claude Code feature (not part of the MCP protocol), so it belongs in `packages/claude-plugin/` rather than in `packages/codex-mcp/` (the MCP server stays a pure cross-client tool surface). The hook coexists with `/codex-review` rather than replacing it. Documentation honest about when each helps.
- **Decision:** Three additions to `packages/claude-plugin/`, zero changes to any MCP package. (1) **Hook script `scripts/codex-pair-watch.mjs`** — a `.mjs` Node script invoked by PostToolUse on Edit/Write/MultiEdit. Imports `executeCodexCLI` from `ask-codex-mcp/executor` (reuses the shared timeout + JSONL parsing + fallback-model handling rather than shelling out — single source of truth for codex invocation). The script's first action after parsing the hook payload is `findMarkerUp(cwd)` — walks up from cwd through 20 levels looking for `.codex-pair-context.md`. **No marker found → exit 0 silently, no codex call, no log entry, no cost.** This is the load-bearing gate: the hook is always loaded in the plugin (so any plugin install ships with the capability), but acts only when the user opts in by creating the marker file. The marker file's content is read as project context and prepended to codex's prompt (single artifact, two purposes — gate signal AND context source). HIGH and MED concerns from codex's response are surfaced to stderr (Claude Code passes hook stderr to Claude on the next turn as a system reminder); LOW concerns are logged but suppressed from surfacing (alert-fatigue control). All calls — including NONE verdicts, skips, and errors — are logged to `.codex-pair-log.jsonl` alongside the marker file for post-hoc analysis. Configuration knobs via env vars: `CODEX_PAIR_DISABLED=1` (kill switch beats marker), `CODEX_PAIR_MAX_FILE_BYTES` (default 20_000 — files larger skip). Skip patterns hard-coded for the common cases (node_modules, dist, .git, lockfiles, images). The script's `main().catch(...)` wrapper guarantees `process.exit(0)` on every failure path — the hook must never break Claude's tool flow. (2) **`hooks.json` PostToolUse entry** matching `Edit|Write|MultiEdit` and invoking the hook via `node ${CLAUDE_PLUGIN_ROOT}/scripts/codex-pair-watch.mjs`. The matcher uses Claude Code's pipe-OR syntax; coexists with the existing PreToolUse Bash hook from the pre-commit Gemini review. (3) **New skill `skills/codex-pair/SKILL.md`** with `user_invocable: false` — it's documentation of the hook's behavior, not an invocable command. Describes the enable mechanism, the when-to-use-each comparison table vs `/codex-review`, the cost characteristics, the output format example, and the config knobs. The existing `skills/codex-review/SKILL.md` gets a new "When to use this vs codex-pair" section linking to the sibling skill so users routing on selection criteria find both. **Tests**: 19 new tests in `src/__tests__/codex-pair-watch.test.ts` — 13 structural invariants (script is executable, imports from ask-codex-mcp/executor, references the marker file, has the kill-switch env var, parses HIGH/MED/LOW, surfaces HIGH+MED but suppresses LOW from stderr, always exit(0)) plus 6 runtime-behavior tests that spawn the script as a subprocess with synthesized stdin payloads and verify the gate logic without invoking real codex (malformed JSON → exit 0, non-watched tool → exit 0, missing marker → exit 0, CODEX_PAIR_DISABLED=1 → exit 0 even with marker, node_modules → skip, unreadable target file → log skipped, marker found by walking up from subdirectory → log lands at marker dir). `skills-and-agents.test.ts` updated to include `codex-pair` in the expected skill set (test count 131 → 134 with the it.each expansion). **Out of scope**: (a) any change to `packages/codex-mcp/` or `packages/llm-mcp/` — they remain client-agnostic MCP servers; the hook belongs in the Claude-Code-specific plugin layer; (b) modifying the `codex-reviewer` agent's prompt — its precision-first calibration is the correct shape for diff review and stays unchanged; codex-pair is a complementary surface with a different prompt; (c) automatic enable based on heuristics (file types, project structure) — explicit per-project opt-in via marker file is intentional to avoid surprise costs and to make the opt-in version-controllable; (d) telemetry of codex-pair catches across user sessions — privacy-sensitive, deferred; (e) per-validator-slot architecture (multiple validators fanning out from one hook) — codex is the first instance, the structure scales but isn't built yet; (f) cross-file dedup of repeated concerns (the same race condition flagged at storage.ts and routes.ts) — a real refinement noted in the experiment findings but not blocking v0.1.
- **Consequences:** The plugin gains a continuous background validator that catches a measurably different class of bug than the existing `/codex-review` skill. For projects that opt in by creating `.codex-pair-context.md`, every Edit/Write/MultiEdit fires a Codex review against the just-edited file with project context. HIGH and MED concerns appear to Claude as system reminders on the next turn, naturally closing the loop without manual `/codex-review` invocation. Cost characteristics for opted-in projects: ~$0.04-0.07 per file reviewed, ~13-50s per call, ~$2-3.50 plus ~10-40 minutes of cumulative codex latency per 50-edit session — meaningful spend, justified for hot-path code (money handling, security, spec-implementing) where one shipped float-money or open-redirect bug costs more than a year of pair calls. Projects that don't create the marker file pay nothing — the hook fires but exits silently after one `fs.access()` call. **Backward compatibility**: every existing skill, agent, and the pre-commit Gemini hook continues to work unchanged. The plugin's package shape is the same (no new bin entries, no new MCP servers, no marketplace listing change). Plugin version bumps minor: 0.3.1 → 0.4.0 via the existing changesets release flow (ADR-076). **Test count**: plugin 131 → 134 (existing) + 19 new = 150 passing. **What is empirically defended by the four-task benchmark** (full audit trail at branch `experiment/codex-pair-poc`): tasks 1-3 demonstrate the v2 prompt design (HIGH/MED/LOW + project-context-file pattern + threshold-in-hook) catches real defects across structurally different task shapes (web CRUD, security-heavy web service, pure-algorithm spec conformance) with N=3 differential improvements (1/6→6/6, 2/11→11/11, etc.) including bonus catches like live prototype pollution and IP-spoof via trust-proxy. Task 4's three-arm comparison is the missing piece that justifies *this specific architectural choice over the simpler "extend /codex-review"* — by showing the two prompts catch different classes of bug, it confirms the value of shipping both surfaces side-by-side rather than collapsing them. **What this is NOT positioned as**: a replacement for `/codex-review`. Documentation throughout (in the new SKILL.md, the updated codex-review SKILL.md, the plugin README) frames them as complementary with explicit selection criteria. The marker file's *presence* is the opt-in signal because explicit opt-in is correct given the cost profile — a default-on pair-validator would surprise users with token bills on their next routine refactor session. **Open design items for future iteration** (NOT blocking ship): cross-file dedup of repeated concerns; per-code-path locking for the validator slot architecture; cost-budgeting in-session (warn at $X spent); multi-validator fan-out from one hook (codex + semgrep + custom rule packs); telemetry on catches per session if privacy concerns are addressed. **Forensic record of the design arc**: my initial recommendation was to ship as a new package (`packages/codex-pair/`). The user (Lykhoyda) correctly pushed back that hooks belong in the plugin layer because they're a Claude Code feature, not MCP. Earlier in the same arc I had also recommended "graduate the POC as the new code review" — until the three-arm task-4 experiment showed `/codex-review` already catches most of what pair catches with a different precision profile. Both course corrections shaped the final architecture: complement, not replacement; hook in plugin, not new package; opt-in via marker, not new config schema. The N=4 experimental evidence + the architectural arguments together support this exact shape.

## ADR-076: Adopt Changesets + `changesets/action` for Monorepo Versioning and Release Automation
- **Date:** 2026-05-14
- **Status:** Accepted (motivated by manual-bump friction observed across v1.6.3 and v1.6.4 releases)
- **Context:** The previous release flow was tag-driven and required the maintainer to manually edit 3–5 `package.json` files per release (gemini, codex, ollama, llm, shared — sometimes plugin), remember the bundled-dependencies cascade ("if `@ask-llm/shared` changed, every package that `bundledDependencies` it needs a republish — otherwise the new shared code never ships"), align the v-prefix unified tag with the lead-version (gemini's), and push the tag to fire `release.yml`. This worked but had three observable failure modes in the recent release cycle: (1) **forgotten bumps** — during the v1.6.3 release I correctly bumped 5 packages, but the cognitive load of remembering which packages bundle which is a tripwire; one missed bump means a user `npx -y ask-foo-mcp` installs a tarball with stale bundled code; (2) **SSH-agent flakiness on tag push** — the v1.6.4 release got stuck for a week because the 1Password agent dropped between the `main` push (which succeeded) and the tag push (which needed signing), and the maintainer-bot kept filing duplicate audit issues (#52, #54) reporting the unreleased fix as "still broken in production"; (3) **no CHANGELOG anywhere** — release notes were auto-generated by `gh release create --generate-notes`, which scrapes commit messages but doesn't capture "this PR affects gemini's response shape" or "this is a patch vs minor" intent. Changesets addresses all three: the version bump is mechanical, releases trigger on PR merges (no tag push needed), and contributors write the changelog-worthy summary at PR time. The `changesets/action` pattern is the canonical GitHub-Actions integration — it opens a `chore: version packages` PR aggregating queued changesets, and merging that PR fires the publish. Three implementation shapes were considered: (A) **changesets/action canonical** — bot-opened Version Packages PR, merge triggers publish, no manual tag push; (B) **changesets for versioning only, tag-driven publish kept** — contributors `yarn changeset` per PR but maintainer still runs `yarn changeset:version` locally and pushes a tag; (C) **hybrid** — bot-opened Version Packages PR, but the merge only lands the bump, manual tag push still required to publish. Option A was selected (after the maintainer chose it from an explicit AskUserQuestion comparing all three). The 1Password-agent stuck-release failure mode specifically motivated dropping the tag-push dependency — a release path that depends on an interactive credential agent being unlocked at the right moment is a release path that occasionally gets stuck for days. Concurrency was a concern with Option A ("what if I'm not ready to release yet?") — but the Version Packages PR is itself the deferral mechanism. Pending changesets accumulate until the maintainer merges the bot's PR; *that* merge is the release trigger, not the upstream feature-PR merges. So the maintainer keeps timing control without depending on SSH-agent state.
- **Decision:** Install `@changesets/cli` + `@changesets/changelog-github` as workspace-root dev deps. Run `yarn changeset init` to scaffold `.changeset/`, then customize `.changeset/config.json` with four load-bearing settings: (1) **`changelog: ["@changesets/changelog-github", { repo: "Lykhoyda/ask-llm" }]`** — generates rich changelogs with PR/commit hyperlinks rather than the default plain-text format; (2) **`access: "public"`** — all our published MCP server packages are public on npm; (3) **`ignore: ["@ask-llm/plugin"]`** — the plugin is distributed via `.claude-plugin/marketplace.json` not npm, so it's on its own version track outside changesets' control; (4) **`privatePackages: { version: true, tag: false }`** — `@ask-llm/shared` is `"private": true` and bundled into MCP packages rather than published, but its version still needs to be tracked internally because the bundled-tarball signature changes when shared changes. The most important config — **`updateInternalDependents: "always"`** under the `___experimentalUnsafeOptions_WILL_CHANGE_IN_PATCH` block — forces bundlers (gemini/codex/ollama/llm) to receive a patch bump whenever `@ask-llm/shared` bumps, regardless of whether the declared `workspace:*` range is "broken." Without this, the default `out-of-range` policy would NEVER fire the cascade because `*` is always-in-range, and shared changes would never reach published tarballs. Verified empirically with a rehearsal changeset bumping shared: produced the expected 5-package patch-bump cascade (shared 0.3.1→0.3.2, codex 0.3.6→0.3.7, gemini 1.6.4→1.6.5, llm 0.3.7→0.3.8, ollama 0.3.1→0.3.2) with `@ask-llm/plugin` correctly excluded via `ignore`. Root `package.json` gains four new scripts: `changeset`, `changeset:version`, `changeset:status`, `changeset:publish`. The `:publish` variant runs `yarn build && changeset publish` so the dist/ outputs are fresh before npm publishes. `release.yml` is rewritten: trigger changes from `push: tags: ["v*"]` to `push: branches: ["main"]` — the same workflow now handles BOTH phases (Version Packages PR creation AND publish) via `changesets/action@v1` with `version: yarn changeset:version` and `publish: yarn changeset:publish`. Concurrency-locked on `${{ github.workflow }}-${{ github.ref }}` to prevent simultaneous publishes racing on the same branch. `createGithubReleases: false` deliberately disables changesets/action's per-package GitHub Releases — the legacy `v<gemini-version>` unified-tag scheme is preserved by a custom step at the end of the workflow so URL-bookmarked v1.5.x..v1.6.x release pages still resolve. Per-package tags (`ask-gemini-mcp@1.6.5`, `ask-codex-mcp@0.3.7`, etc.) coexist with the unified `v1.6.5` tag — both useful for different audit purposes (per-package for npm timeline lookups, unified for release-page bookmarks). The mcp-publisher step and the version-sync from `package.json` to `server.json` / `marketplace.json` are preserved unchanged — both still run, gated on `steps.changesets.outputs.published == 'true'` so they fire only when changesets actually publishes (skipped on every "pending changesets, no version-packages merge yet" push). One security-hygiene fix landed alongside: the new unified-release step uses `env:` passthrough for the gemini version output instead of inline `${{ ... }}` interpolation in the shell `run:` block (avoids the GitHub Actions command-injection class even though the value is internally-controlled). CONTRIBUTING.md gets a new "Versioning your change" section explaining the `yarn changeset` prompt + the `chore: version packages` PR convention + the bundled-deps cascade so contributors don't have to read this ADR to understand the flow. **Out of scope**: (a) migrating historical CHANGELOG entries from `gh release` auto-generated notes — too much retrospective work, accept that CHANGELOG.md per-package starts fresh from v1.6.4+; (b) `@ask-llm/plugin` integration with changesets — plugin marketplace versioning is fundamentally different (no npm-publish step, just a marketplace.json edit), defer; (c) Slack/Discord notifications on Version Packages PR creation — nice-to-have, not blocking; (d) backporting changesets to release.yml's earlier tag-driven path as a fallback — explicitly rejected, the whole point of this ADR is to remove the tag-push dependency.
- **Consequences:** The maintainer's release workflow simplifies from "edit N package.jsons, commit, tag, push tag, hope SSH agent is unlocked" to "merge the Version Packages PR when ready." The bundled-dependencies cascade that bit the v1.6.3 release prep is now mechanical — adding a changeset on `@ask-llm/shared` automatically queues patch bumps for all bundlers via `updateInternalDependents: "always"`. CHANGELOGs land per-package under `packages/<name>/CHANGELOG.md` automatically — no more relying on `gh release --generate-notes`. The maintainer-bot's daily upstream-CLI audit cycle (issues #52, #54, etc.) now has a tighter feedback loop: a fix PR + changeset can land + ship the same day if the maintainer is ready to merge the Version Packages PR. **Backward compatibility**: existing v1.5.x..v1.6.x tags continue to resolve as GitHub Release pages because the unified-tag step is preserved. Old release URLs don't 404. The `prepack-bundle.mjs` / `postpack-restore.mjs` lifecycle from ADR-052 is preserved — those scripts fire inside each `npm publish` regardless of whether `changesets publish` or a manual `npm publish` invoked it. **First-release behavior**: the chore/adopt-changesets PR itself ships ZERO changesets — it's an infrastructure-only change. When it merges to main, the workflow runs, sees no pending changesets, and exits cleanly (no Version Packages PR opened, no publish). The next behavior change (any future PR with `yarn changeset` run) will be the first one to flow through the new pipeline end-to-end. **Test count**: unchanged — changesets is a release-mechanics tool, doesn't touch product code. **Future work**: (a) re-enable per-package GitHub Releases via `createGithubReleases: true` and drop the unified-tag step once enough time has passed that v1.5.x bookmarks are no longer load-bearing; (b) plugin-marketplace versioning automation — a separate problem space, possibly a custom GitHub Action that bumps marketplace.json on plugin-package changes; (c) Slack webhook on Version Packages PR creation if a teammate joins the project; (d) `bumpVersionsWithWorkspaceProtocolOnly: true` if the project later wants to publish `@ask-llm/shared` itself (not bundle it) — would require flipping `privatePackages.version` semantics. **Forensic record**: this ADR was added BEFORE the first real changeset-driven release. The first changeset-driven release will be visible in `git log` as a `chore: version packages` merge commit — not a manual `chore: bump versions for ...` commit. That difference is the marker for "this was the canonical changesets flow vs. the legacy manual flow." The last manual-flow release was v1.6.4 (commit `cd44eda`, 2026-05-13).

## ADR-075: Upstream CLI Compat — Codex `--sandbox workspace-write` (replaces `--full-auto`) + Gemini Trust Env Var Co-Emit
- **Date:** 2026-05-12
- **Status:** Accepted (closes #46; supersedes audit issues #41, #49, #50, #51 — same underlying breakages)
- **Context:** Two upstream-CLI changes landed in the days following the v1.6.3 release that silently regress code we ship. (1) **`codex --full-auto` deprecation**: codex `rust-v0.128.0` (current stable) emits `warning: --full-auto is deprecated; use --sandbox workspace-write instead.` on every invocation (verified empirically: `codex exec --full-auto --skip-git-repo-check --ephemeral -m gpt-5.5 "say ok"` produces the warning in stderr today). `rust-v0.129.0-alpha.x` graduates `--full-auto` to a hidden "removed_full_auto" trap with `conflicts_with = "dangerously_bypass_approvals_and_sandbox"` (per issue #46's source inspection of upstream `main`). Our `packages/codex-mcp/src/utils/codexExecutor.ts:buildArgs` hardcoded `CLI.FLAGS.FULL_AUTO = "--full-auto"` on every `ask-codex` call — users on 0.128 see a stderr warning leak through (parseCodexJsonlOutput drops non-JSONL lines, but the warning still pollutes), users on 0.129+ will hit a hard argv-parse failure once that branch stabilizes. (2) **`GEMINI_TRUST_WORKSPACE` env var rename**: gemini-cli `main` (heading toward v0.42) renamed the env var to `GEMINI_CLI_TRUST_WORKSPACE`. Upstream `git grep GEMINI_TRUST_WORKSPACE` returns zero hits — *no backward-compatible alias*. Our `ensureWorkspaceTrustEnv()` (ADR-069's hotfix) still wrote the old name only. Users who upgrade to nightly/preview gemini-cli would silently lose our auto-trust default and start hitting `FatalUntrustedWorkspaceError` in any directory not already in `~/.gemini/trustedFolders.json` — a hard regression of the ADR-069 fix. Compounding this, the user-facing `WORKSPACE_TRUST_REQUIRED` error message at `constants.ts:13-14` told users to `export GEMINI_TRUST_WORKSPACE=true`, which would do nothing on upgraded gemini-cli. Three implementation shapes were considered for the gemini env var: (A) **hard cutover** — write new name only, drop old — fixes ≥0.42 but regresses ≤0.41 majority who rely on the existing default-on auto-trust (issue #46's literal recommendation but with this side-effect); (B) **co-emit** — write both names with the same value, no version detection, both branches see the trust grant; (C) **co-emit with deprecation warn** — also Logger.warn when the user has set the OLD name in their MCP config. Option B was selected — it's the only one that keeps the working majority working *and* the upgrade path clean. Option C's warn would be noise for users who configured the old name legitimately when ADR-069 was the canonical shape; we'll add deprecation warning if/when we plan to actually drop the old name in a major release. Bundling #41 item 2 (default inversion from opt-out to opt-in via `ASK_GEMINI_TRUST_WORKSPACE=1`) was considered and explicitly deferred — that's a security-default change with backward-compat implications that deserves its own discussion, separate from the urgent compat patch.
- **Decision:** Three minimal surface changes, all backward compatible. (1) **Codex flag swap** in `packages/codex-mcp/src/constants.ts:CLI.FLAGS`: removed `FULL_AUTO: "--full-auto"`, added `SANDBOX: "--sandbox"` and `SANDBOX_WORKSPACE_WRITE: "workspace-write"`. In `codexExecutor.ts:buildArgs`, replaced `base.push(CLI.FLAGS.FULL_AUTO, CLI.FLAGS.JSON, ...)` with `base.push(CLI.FLAGS.SANDBOX, CLI.FLAGS.SANDBOX_WORKSPACE_WRITE, CLI.FLAGS.JSON, ...)`. The new two-token shape `--sandbox workspace-write` is the canonical replacement per codex's own deprecation message — verified by running `codex exec --sandbox workspace-write` against 0.128 (executes cleanly, no deprecation warning emitted). Critically, `codex exec` is non-interactive *by definition* per its own help text ("Run Codex non-interactively"), so we don't need the `--ask-for-approval never` part that `--full-auto` previously bundled — the subcommand carries that semantic. The `dangerously-bypass-approvals-and-sandbox` flag exists as a separate kill-switch for callers who want neither; we deliberately don't use it because `workspace-write` is the right tradeoff (codex can modify files in the workspace it was invoked from, can't escape). (2) **Gemini env var co-emit** in `geminiExecutor.ts:ensureWorkspaceTrustEnv()`: function now checks both env-var names as early-return signals (respecting either name as a user-supplied opt-out), and when neither is set, writes BOTH `GEMINI_TRUST_WORKSPACE = "true"` and `GEMINI_CLI_TRUST_WORKSPACE = "true"`. Zero runtime version detection — gemini ≤0.41 reads the old name, gemini ≥0.42 reads the new name, both branches get the trust grant from a single spawn-env mutation. The `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST=1` opt-out from ADR-069 remains the first early-return so strict-trust users are unaffected. (3) **Error message + README rewrite** in `gemini-mcp/src/constants.ts` (`WORKSPACE_TRUST_REQUIRED`) and `gemini-mcp/README.md`: error now lists both env-var names with version annotations, references the new `--skip-trust` CLI flag (available on gemini ≥0.41 — cleaner remediation than env vars for one-off use), and the README's prerequisites line spells out the co-emit pattern so users comparing their gemini-cli version to docs can self-diagnose. **Tests**: codex executor tests at `__tests__/codexExecutor.test.ts` updated in three places where buildArgs assertions checked the literal flag sequence — `CLI.FLAGS.FULL_AUTO` replaced with `CLI.FLAGS.SANDBOX, CLI.FLAGS.SANDBOX_WORKSPACE_WRITE` to keep the assertion shape pinning the exact argv order. ## ADR-074: Per-Provider Timeouts (`ASK_CODEX_TIMEOUT_MS` / `ASK_GEMINI_TIMEOUT_MS`) — Codex Default 800s
- **Date:** 2026-05-07
- **Status:** Accepted (closes #45)
- **Context:** Issue #45 reported that long codex prompts return `Command timed out after 210s` to the client even when the underlying codex CLI completes in under 210s, with the `get-usage-stats` endpoint showing 196.8s wall time and 7,062 output tokens after the timeout error — the user's hypothesis was a race between the timer and the close event with stats recorded post-hoc. A full data-flow trace showed that hypothesis is **structurally impossible given the current code**: the `setTimeout` timer in `commandExecutor.ts:99` and the `childProcess.on("close")` handler at `commandExecutor.ts:142` are gated by a single `isResolved` flag (`commandExecutor.ts:96`) — only one path can fire to completion. There is no streaming JSONL parser anywhere in the monorepo (verified by grepping for `task_completed`, `turn.completed`, streaming parse patterns in all of `packages/`), no `--output-last-message` side file flag, and no inline usage tracker that runs independent of the resolve path. `durationMs` for codex is computed at `codexExecutor.ts:177` as `Date.now() - startedAt` only after `executeCommand` resolves successfully — if the timer rejected first, `parseCodexJsonlOutput` never runs and `sessionUsage.record` is never called. The most likely actual sequence was two separate calls: a first that timed out (no stats recorded) and a second that succeeded at 196.8s (recorded in stats — "Total calls: 1" reflects the survivor); Claude Desktop may auto-retry on tool errors, or the user re-issued without realizing. The actionable bug, regardless of whether the user observed one or two calls, is that **210s is too aggressive a default for codex with reasoning models**. Codex `--full-auto` with `gpt-5.5` does multi-turn tool-use loops where each turn includes reasoning; substantive prompts (clinical analysis, deep code review, multi-section synthesis) routinely consume 300–600s of wall time. Gemini's stream-json mode is comparatively fast because gemini emits tokens incrementally as the model generates. Ollama runs locally over HTTP and its latency profile is different again. The user's global CLAUDE.md already encoded informal guidance "when using codex mcp the timeout should be at least 800 seconds" — formalizing that as a per-provider default surfaces the right knob in the right place. Three implementation shapes were considered: (A) raise the global `GMCPT_TIMEOUT_MS` default to 800s for everyone — punishes gemini and ollama users with 4× the timeout for no benefit since their CLIs don't need it; (B) add per-provider env vars with provider-specific defaults — gemini stays at 210s, codex moves to 800s, the global env var still works as an override-everything escape hatch; (C) implement streaming/heartbeat (issue's suggested fix #4) so partial output reaches the client before any timeout fires — much larger refactor, separate concern, and orthogonal to the timeout question because even with streaming the eventual SIGTERM still discards in-flight responses. Option B was selected as the minimal change that addresses the real root cause; option C remains future work for a separate ADR if streaming becomes a goal in its own right.
- **Decision:** Three layers of changes, all backward compatible. (1) **Shared layer** in `packages/shared/src/commandExecutor.ts`: added an optional 6th parameter `timeoutMs?: number` to `executeCommand` — if provided, it overrides the env-derived default; if omitted, behavior is identical to today (reads `GMCPT_TIMEOUT_MS` or falls back to 210s). Added and exported a new helper `resolveTimeoutMs(providerEnvVar: string, fallbackDefault: number): number` that implements the precedence ladder `provider env var > GMCPT_TIMEOUT_MS > fallback default`, with input validation rejecting non-numeric and non-positive values (mirrors the existing `getTimeoutMs` parser via shared `parseTimeoutEnv`). The helper is exported via `@ask-llm/shared` so each provider's executor owns its policy without putting provider-specific knowledge (codex vs gemini vs ollama) in the shared module — keeps the shared layer provider-agnostic, which is how it was designed (ADR-053). New constants in `packages/shared/src/constants.ts`: `EXECUTION.DEFAULT_CODEX_TIMEOUT_MS = 800_000`, `EXECUTION.CODEX_TIMEOUT_ENV_VAR = "ASK_CODEX_TIMEOUT_MS"`, `EXECUTION.GEMINI_TIMEOUT_ENV_VAR = "ASK_GEMINI_TIMEOUT_MS"` — env-var names live in one place so tests, docs, and executors all import from a single source of truth. The timeout error message in `commandExecutor.ts:111-114` was rewritten to reference both the per-provider knobs (`ASK_CODEX_TIMEOUT_MS / ASK_GEMINI_TIMEOUT_MS`) and the global `GMCPT_TIMEOUT_MS`, replacing the earlier message that only mentioned the global. (2) **Provider executors** wire the resolver: `packages/codex-mcp/src/utils/codexExecutor.ts` calls `resolveTimeoutMs(EXECUTION.CODEX_TIMEOUT_ENV_VAR, EXECUTION.DEFAULT_CODEX_TIMEOUT_MS)` once per `executeCodexCLI` invocation and threads the resolved `timeoutMs` to both the primary `executeCommand` call and the quota-fallback retry — pinning both paths in tests is critical because a regression where the fallback drops the param would only surface during a real quota event in production. `packages/gemini-mcp/src/utils/geminiExecutor.ts` follows the same shape with `EXECUTION.GEMINI_TIMEOUT_ENV_VAR` and `EXECUTION.DEFAULT_TIMEOUT_MS` (210s, unchanged from today) — symmetric coverage in both primary and `MODELS.FLASH` quota-fallback paths. **Ollama is deliberately not wired** because ollama uses `globalThis.fetch` over HTTP rather than `executeCommand` (`packages/ollama-mcp/src/utils/ollamaExecutor.ts:callOllama`); plumbing a timeout into an `AbortController` there is a separate code path with different semantics (HTTP request timeout vs subprocess SIGTERM), and the issue and its remediation are scoped to subprocess-spawned CLIs. (3) **Diagnose tool** in `packages/shared/src/doctor.ts` — surfaces the resolved per-provider timeouts in both the `DiagnosticReport.environment` structured field (added `codexTimeoutMs: number` and `geminiTimeoutMs: number`) and the human-readable formatter (replaced single `Timeout: 210000ms` line with `Timeouts: codex=800000ms, gemini=210000ms`). The diagnose output is the canonical thing users paste into bug reports, and the issue's reproduction explicitly cited the misleading single `Timeout: 210000ms` line — having the breakdown there makes "why did codex time out at 210s?" answerable at a glance. The `outputSchema` in `serverFactory.ts:38-45` was updated to validate the new fields. The original `timeoutMs` field is preserved for backward compatibility with existing consumers of the structured report. **Test coverage**: 14 new tests pinning the contract — unit tests for `resolveTimeoutMs` (provider env > global env > default precedence, invalid/zero/negative env handling), real-spawn tests verifying `executeCommand`'s `timeoutMs` parameter wins over env and the new error message references both env-var families, mock-based executor tests verifying both codex and gemini pass the resolved timeout to `executeCommand` in both primary and quota-fallback paths, doctor tests pinning the new "Timeouts:" line and updated `environment` schema, and SMOKE_TEST-gated integration tests in both `packages/codex-mcp/src/__tests__/integration.test.ts` and `packages/gemini-mcp/src/__tests__/integration.test.ts` that set `ASK_{CODEX,GEMINI}_TIMEOUT_MS=50` and assert the timeout fires before the real CLI can respond — proves end-to-end wiring through a live spawn. **Out of scope**: (a) raising the gemini default — gemini's `--output-format stream-json` emits tokens incrementally and the global 210s has not produced timeout reports against gemini calls; raising it preemptively is YAGNI. (b) ollama timeout knob — the HTTP path needs an `AbortController` integration that's a separate piece of work; the issue did not flag ollama. (c) streaming/heartbeat support (issue's #4) — large architectural effort, requires partial response delivery through MCP progress notifications rather than the all-or-nothing return value, deferred to its own ADR. (d) returning the buffered response on timeout (issue's #3) — the forensic finding showed the underlying mechanism the user hypothesized doesn't exist; "return the buffered response" implies parsing the partial JSONL, which is the streaming work in (c). (e) per-provider timeout overrides at the call site (e.g., a `timeoutMs` parameter on the `ask-codex` MCP tool input) — keeps the public tool surface lean; users who need ad-hoc overrides set the env var on the MCP server's process, which is one MCP config file edit.
- **Consequences:** Codex's effective default rises 4× from 210s to 800s, addressing the silent-data-loss class of bug reported in #45 — long reasoning prompts now have headroom to complete instead of hitting the timer mid-response. The user's existing `GMCPT_TIMEOUT_MS` workflow continues to work unchanged: setting it lifts both providers' timeouts simultaneously. Per-provider knobs are the new finer surface: a user who wants codex at 1200s but gemini at 60s sets `ASK_CODEX_TIMEOUT_MS=1200000 ASK_GEMINI_TIMEOUT_MS=60000` and the resolver picks the right one for each spawn. Diagnose output now visibly distinguishes per-provider timeouts so future bug reports cite the right number. **No breaking changes**: `executeCommand`'s `timeoutMs` parameter is optional and falls back to the previous env-only behavior when omitted; the `DiagnosticReport.environment.timeoutMs` field is preserved (new fields added alongside); MCP tool surface unchanged; CLI argv unchanged. **Test count**: shared 153 → 156 (+3: resolveTimeoutMs precedence, executeCommand timeoutMs wins, error-message env-var references); codex 39 → 43 (+4: default, GMCPT precedence, ASK_CODEX precedence, fallback path); gemini 95 → 99 (+4: default, GMCPT, ASK_GEMINI precedence, flash-fallback path); doctor changes covered by existing test rewrite. Plus +2 SMOKE-gated integration tests that don't run in normal CI. Total +13 unit + 2 smoke = 15 new test entries. **Forensic record**: the issue's hypothesis #3 ("response generated but discarded during transmission via a race between timer and close event") is documented in this ADR as **structurally impossible** given the current code so future contributors don't chase the same dead end. The actual mechanism — long reasoning prompts exceed the global default — is what the per-provider knob addresses. **Future work**: (a) ollama timeout knob (`ASK_OLLAMA_TIMEOUT_MS` via `AbortController` in `callOllama`) — natural next iteration, modest scope; (b) streaming support so partial response content is delivered to the client before any timeout fires — large effort requiring MCP progress notification reshaping, the right time is when there's a second use case beyond the timeout-recovery one; (c) per-call timeout override on the MCP tool input (e.g. `ask-codex { prompt, timeoutMs: 1200000 }`) — defer until at least one user requests it, server-side env var is the standard MCP configuration model; (d) revisit the codex 800s default once codex's reasoning models stabilize on faster inference paths — current value is conservative, may shrink in future once `gpt-5.5` reasoning latency improves.

## ADR-072: Plumb Codex `reasoning_output_tokens` → Shared `thinkingTokens`
- **Date:** 2026-05-04
- **Status:** Accepted (drains P1 §2 from issue #28; addresses #27 B1)
- **Context:** Codex CLI `rust-v0.125.0` (released 2026-04-24, stable since #27 was filed) added a `reasoning_output_tokens` field to the `turn.completed.usage` event for reasoning models (the `gpt-5.5` family). Our `CodexTurnCompleted.usage` interface in `packages/codex-mcp/src/utils/codexExecutor.ts:12-19` did not declare the new field, so it was silently dropped during JSONL parsing. Downstream impact: `buildUsageStats` hard-coded `thinkingTokens: undefined` for codex calls; `[Codex stats: ...]` footer never showed reasoning tokens; `get-usage-stats` MCP tool's cross-provider aggregation reported `thinkingTokens: 0` from codex regardless of how much reasoning the model actually did. For users running `/multi-review` against `gpt-5.5` with deep reasoning, this hid a meaningful slice of their token spend. Issue #28 §3 P1 #2 flagged this as a "small, isolated correctness win" — ~15 LoC, fully backward-compatible because the field is optional in our types and older codex versions simply don't emit it. The shared `UsageStats` type in `@ask-llm/shared` already has a `thinkingTokens` field (added for Gemini's `stats.models[*].tokens.thoughts`), so the right fix is to map codex's `reasoning_output_tokens` to the same shared field rather than introducing a codex-specific reasoning-tokens field that would drift the cross-provider aggregation surface.
- **Decision:** Three small edits in `packages/codex-mcp/src/utils/codexExecutor.ts`: (1) Added `reasoning_output_tokens?: number` to the `CodexTurnCompleted.usage` interface declaration with a comment explaining the codex 0.125+ provenance and the optional/backward-compatible nature. (2) Updated `buildUsageStats` to map `usage?.reasoning_output_tokens` → `thinkingTokens` (replacing the hard-coded `undefined`). The shared `UsageStats.thinkingTokens` field is the cross-provider aggregation surface — Gemini's `thoughts` and Codex's `reasoning_output_tokens` both flow into the same field, so `get-usage-stats` and `formatSessionUsage` aggregate correctly across mixed-provider sessions without per-provider unwrapping. (3) Updated `formatStats` to surface a `"7,500 thinking tokens"` line in the `[Codex stats: ...]` footer when `reasoning_output_tokens > 0`. Footer ordering: `input → output → thinking → cached`, mirroring `geminiExecutor.ts:formatStats` so cross-provider stats blocks line up visually in `/multi-review` reports. The `> 0` guard matches the cached-tokens precedent: zero reasoning tokens (non-reasoning models like older gpt-4 calls) shouldn't surface a noisy `"0 thinking tokens"` line. 3 new tests in `packages/codex-mcp/src/utils/__tests__/codexExecutor.test.ts`: present-and-positive (footer shows the value, `UsageStats.thinkingTokens` carries it for telemetry), present-and-zero (footer suppresses the line but `UsageStats.thinkingTokens === 0` is preserved — the explicit zero is meaningful telemetry, not the same as field-missing), absent (older codex versions where the field never appears — `UsageStats.thinkingTokens` is `undefined`, footer omits the line). The triple covers the upstream version-compatibility matrix without needing real codex versions in tests. **Out of scope:** (a) backfilling reasoning-token telemetry for codex calls *prior to v0.125* — not possible, the upstream JSONL didn't carry the field. (b) Surfacing reasoning tokens via a *separate* `codexReasoningTokens` field instead of reusing `thinkingTokens` — explicitly rejected because cross-provider aggregation would then need per-provider unwrapping logic in `formatSessionUsage`. (c) Distinguishing "model emitted 0 reasoning tokens" from "model didn't emit the field" in the footer — both correctly suppress the line; the distinction lives in `UsageStats.thinkingTokens` (`0` vs `undefined`) where telemetry consumers can read it.
- **Consequences:** `/multi-review` and `/brainstorm` reports against `gpt-5.5` now correctly show how many tokens the model spent on reasoning, alongside input + output + cached counts. Cross-provider sessions (e.g. `/compare gemini vs codex`) get apples-to-apples reasoning-token comparison via the shared `UsageStats.thinkingTokens` field. **No new tool surface, no schema changes, no new env vars** — purely a parser-side feature adoption. Backward compatibility is preserved by construction: older codex versions don't emit `reasoning_output_tokens`, so the optional-undefined path in TypeScript types and runtime checks handles them identically to today's behavior. Argv unchanged. Test count: codex-mcp 36 → 39 (+3 reasoning tests). **Drains P1 #2 from #28's punch list**, leaving P2 (#5 typed exit codes, #7 `-m` precedence smoke test, #8 `--add-dir` parity) and P3 items as remaining work. The shared `UsageStats.thinkingTokens` field is now used by Gemini, Codex, and any future provider with a reasoning-token concept — the precedent is "map provider-native reasoning-token field name → shared `thinkingTokens`" rather than adding per-provider fields.

## ADR-073: `/codex-verify` Skill + `codex-verifier` Agent — Claim Verification as a Distinct Contract from Issue Review
- **Date:** 2026-05-05
- **Status:** Accepted
- **Context:** A study of disler/the-verifier-agent (`~/GitHub/the-verifier-agent`, full read of `apps/verifier/verifier.ts`, `verifiable.ts`, `cross-agent.ts`, `.pi/verifier/agents/verifier.md`, `.pi/verifier/prompts/verify_on_stop.md`) surfaced a structural pattern this project's reviewer agents have been quietly approximating without naming it. The Pi verifier's central claim — that the *builder agent's final assistant message is a CLAIM, never proof* and that "without evidence, the verdict is `unsure`, not `verified`" (`.pi/verifier/agents/verifier.md:32`) — is a fundamentally different contract from "find new issues in the diff." Our existing `codex-reviewer` and `gemini-reviewer` agents conflate the two: they look for issues, then post-hoc validate by re-reading source (`codex-reviewer.md:94-106`, `multi-review/SKILL.md` Phase 3). That post-hoc validation catches false positives but only of the *finding* kind ("is this bug real?") — it never asks the orthogonal question ("did the assistant actually do what it claimed?"). The 2026-04-17 Gemini false-positive incident referenced in `multi-review/SKILL.md:13` was finding-verification failure; a separate failure mode — the assistant claiming "I bumped the threshold to 16384" but actually leaving it at 8192 — would slip past every existing skill in the plugin because none of them check the assistant's own claims against state. Three implementation shapes were considered: (A) extend `codex-reviewer.md` to additionally do claim verification (single agent, two contracts) — risks dilution because the same agent prompted to find issues drifts into "while I'm here let me also..." behavior, exactly what makes the Pi verifier *un-promptable* on purpose (Pi's input bar is locked); (B) add a new `codex-verifier` agent + `/codex-verify` skill mirroring Pi's read-only-tool-surface + atomic-claim-decomposition + five-grade CONFIDENCE-ladder pattern, leaving `codex-reviewer` and `/codex-review` untouched (two narrow contracts); (C) defer entirely and add a "verification mode" boolean to existing skills. Option B was selected — the Pi verifier's strongest design property ("structurally un-promptable") only holds when the verification contract has its own surface, separate from review.
- **Decision:** Two new files, zero changes to MCP servers. (1) **New agent: `packages/claude-plugin/agents/codex-verifier.md`** — frontmatter declares `tools: [Bash, Glob, Grep, Read, mcp__codex__ask-codex]` (no `Write`, no `Edit`, no `NotebookEdit`); description explicitly states "Distinct from `codex-reviewer` (issue hunt) by contract" so Claude Code's agent dispatcher routes "verify what was done" intents here and "find bugs in this diff" intents to the existing reviewer. The agent's body ports four Pi-verifier patterns verbatim: **read-only tool surface as a social contract** (enforcement is prompt-only, mirrors `verifier.md:30`), **atomic claim decomposition** ("I added the user with auth" → at least three claims, mirrors `verifier.md:31`), **evidence beats assertion** ("without evidence, the verdict is `unsure`, not `verified`" — verbatim from `verifier.md:32`), and the **five-grade CONFIDENCE ladder** (`PERFECT | VERIFIED | PARTIAL | FEEDBACK | FAILED` with the same semantics as `verifier.md:43-47` — completeness × outcome, not model self-rating). Crucially, the agent's prompt forbids fix proposals and forbids issue hunting: "Out-of-scope bugs you happen to notice during verification do not go in the Report. The operator has `codex-reviewer` for that." The Report block is the same five-section structure Pi uses (`What did you verify?` / `What could you not verify?` / `Corrective feedback` / `What do you need to verify this next time?` / `Verification metadata`) so a future programmatic parser could be shared. (2) **New skill: `packages/claude-plugin/skills/codex-verify/SKILL.md`** with `user_invocable: true` — five-phase workflow: gather diff + the assistant's last message verbatim, dispatch `codex-verifier`, **defensively parse** the Report block (if `CONFIDENCE` is missing, derive from `STATUS`: `verified→VERIFIED`, `failed→FEEDBACK`, `unsure→FAILED` — mirrors the defensive parser at `verifier.ts:1029-1049` because LLMs reliably forget the second of two adjacent metadata lines), present per-grade output, and **surface PARTIAL gaps loudly** ("PARTIAL is the most actionable grade for project-level investment: every unverifiable claim is a fixture or script you should consider adding"). (3) **Small edit to `multi-review/SKILL.md`** — added a "Two kinds of verification — pick the right skill" callout between the existing "Why verification matters" section and the Instructions, with a 2-row table mapping intent → skill so users running `/multi-review` when they actually wanted `/codex-verify` get redirected before they spend the wall-clock cost of a multi-provider dispatch. (4) **Same-day follow-on: synthesis-confidence on `brainstorm-coordinator`** — after the first three changes shipped, a follow-up question surfaced whether the same ladder should propagate to other skills/agents. `/codex-image` was deliberately **not** changed (its Phase 4 already runs `ls -la` against the filesystem rather than trusting the agent's claim — the file is binary present/absent, atomic-claim decomposition is meaningless, `PARTIAL` doesn't map; forcing the ladder there is cargo-cult). `brainstorm-coordinator` Phase 4 was a real fit because its existing per-finding Verified/Rejected/Unverifiable classification (from ADR-064) gave per-finding clarity but no per-synthesis grade — a user reading the brainstorm couldn't tell at a glance whether "most of this is inferred" vs "most of this is verified against real files." Added a four-grade synthesis-confidence ladder (`PERFECT | VERIFIED | PARTIAL | FAILED`) — **`FEEDBACK` is intentionally dropped** because brainstorming has no fix-loop semantic (no `verifier_prompt` equivalent, no corrective-prompt-back loop), so a five-grade port would have introduced a state with no possible meaning. The agent's prose explicitly documents the dropped grade and why, so a future contributor doesn't restore it by symmetry argument. The grade is surfaced as the first line of the synthesis output (`**Synthesis confidence:** [PERFECT | VERIFIED | PARTIAL | FAILED] — [reason]`) so users see the trust signal before they read the consensus points. Tests pin all four grades, the FEEDBACK-not-applicable note, the lineage to `/codex-verify`, the output-format placement, and Pi's "false PERFECT is worse than honest PARTIAL" honesty discipline. Explicitly out of scope: (a) a parallel `gemini-verifier` / `ollama-verifier` — Codex's `--full-auto --json` JSONL stream is what makes the per-claim narrow-scope prompt parseable; Gemini's free-text output and Ollama's HTTP-chat would each need their own framing pass and are deferred until Codex's surface proves the pattern; (b) a `multi-verify` skill — verification is high-precision-low-recall by design, parallelizing it across providers would just amplify any one model's blind spots; (c) automatic loopback (Pi's `verifier_prompt` → `pi.sendUserMessage(deliverAs:"followUp")`) — Claude Code has no equivalent lifecycle hook, so the corrective feedback is presented to the user as paste-back text rather than auto-injected; the loopback can be added later if Claude Code exposes the right hook; (d) MCP-side changes — no new tools, no executor edits, no constants; the value is entirely in the agent + skill prose contract.
- **Consequences:** Operators now have two complementary contracts: `/codex-review` (or `/multi-review`) finds new issues in the diff; `/codex-verify` proves the assistant's own claims against state. The Pi verifier's strongest property — that it's *structurally un-promptable for review work* — carries over because the verifier agent's tool surface lacks `Write`/`Edit` and its prompt explicitly forbids issue hunting. The five-grade CONFIDENCE ladder replaces the noisy 0–100 self-assessed score for this contract: `PARTIAL` is a first-class verdict that surfaces *gaps in the verification harness* (no oracle, no fixture, ambiguous claim), and those gaps become the next thing the operator templates into the project — every PARTIAL is a future fixture, every FAILED is a missing harness. This converts verification from a one-shot filter into a compounding signal: the verifier teaches you what your verifier is missing. Backward compatibility: every existing skill (`/codex-review`, `/gemini-review`, `/ollama-review`, `/multi-review`, `/brainstorm`) is unchanged in behavior; the multi-review edit is purely additive (one new section, no instruction changes). Plugin file count: agents 4→5 (add `codex-verifier.md`), skills 8→9 (add `codex-verify/`). Tool surface impact on published MCP servers: zero — neither `codex-mcp` nor `llm-mcp` is touched. Test count: unchanged at the package level — skills/agents are markdown prose; the existing `skills-and-agents.test.ts` fixture's structural assertions cover specific known files and won't fail from adding new ones (verified by inspection of the existing test patterns). Future work: (a) parallel `gemini-verifier` once a constraining-prompt experiment shows it produces parseable single-line verdicts despite Gemini's free-text bias; (b) optional cross-validator pass where the verifier dispatches to the *other* provider to verify its own verifier's verdict — adds robustness against single-provider blind spots, deferred until base contract proves itself in real use; (c) audit-log skill that records verifier verdicts to `.claude/verification-audits/<date>.json` so over time the operator builds a per-claim-type accuracy scorecard and can tune the prompt; (d) Claude Code's UserPromptSubmit / Stop hooks could in principle auto-trigger `/codex-verify` after every assistant turn that produced a diff, mirroring Pi's `before_agent_start` / `agent_end` lifecycle — deferred because auto-triggering high-token-cost verification on every turn is a defaults-debate worth having separately. **Note on numbering:** originally drafted as ADR-072 in the same week PR #44 also drafted an ADR-072 for codex `reasoning_output_tokens`; PR #44 merged first and kept 072, so this entry was renumbered to 073 at conflict-resolution time.

## ADR-071: Codex `--ignore-user-config` + `--ignore-rules` for Deterministic MCP Wrapping
- **Date:** 2026-04-28
- **Status:** Accepted (closes #31, addresses action #5 from #26 / #24 §2.2 / #25 §3.3)
- **Context:** Codex CLI `rust-v0.124.0` (released 2026-04-23) graduated hooks (`PreToolUse`, `PostToolUse`, `PermissionRequest`, `UserPromptSubmit`, `Stop`) to **stable** status. The release notes also flagged that *"project hooks and exec policies now require trusted workspaces"* — parallel to the Gemini workspace-trust change resolved in PR #29 / ADR-069. Issue #31 was filed as a follow-up to #26 §4.2 to **smoke-test whether user-installed `~/.codex/config.toml` hooks break our `--ephemeral` exec path**. The risk: if a user has `[hooks.PreToolUse]` configured, our spawned `codex exec` might try to run that hook against the ephemeral sandbox, fail, and bubble the failure as an error event in the JSONL stream rather than silently skipping. Two paths forward: (A) write a smoke test that creates a real `~/.codex/config.toml` with a no-op hook, runs `executeCodexCLI`, and asserts no failure — high-fidelity but binary-version-dependent and fragile (real codex required, may break differently per OS, may interfere with the user's actual config); (B) **preempt the entire risk class** by passing `--ignore-user-config` (skips `~/.codex/config.toml` entirely; auth credentials in `CODEX_HOME` are still loaded) and `--ignore-rules` (skips user/project `execpolicy .rules` files). Option B was independently flagged in #26 §4.6 as **action #5** — *"`--ignore-user-config` + `--ignore-rules` on the ephemeral path — still the highest-leverage change per #24 §2.2 and #25 §3.3."* The two flags together remove the entire surface of "user-side config affecting our deterministic CLI wrapper" — for an MCP server that programmatically calls codex, the user's interactive-mode preferences should never affect what the API caller asked for. Verified locally that Codex CLI 0.125.0 has both flags exposed via `codex exec --help`. **Caveat surfaced during PR review:** `~/.codex/config.toml` doesn't only hold hooks and preferences — it's also where users register `[[mcp_servers]]` entries that codex can invoke as agent tools during execution. `--ignore-user-config` skips those too. A user who has registered, say, a database-query MCP server in their codex config and expects it to be available inside `ask-codex-mcp` invocations would see degraded behavior (codex would not be able to call that DB tool while answering a prompt routed through this wrapper). To mitigate without giving up the deterministic default for the 99% case, an opt-out env var was added.
- **Decision:** (1) `packages/codex-mcp/src/constants.ts` `CLI.FLAGS` gains `IGNORE_USER_CONFIG: "--ignore-user-config"` and `IGNORE_RULES: "--ignore-rules"`. (2) `packages/codex-mcp/src/utils/codexExecutor.ts:buildArgs` emits both flags **by default**, between the existing process-level flags (`--skip-git-repo-check`, optional `--ephemeral`) and the model/output flags (`--full-auto`, `--json`, `-m`). The argv ordering follows the existing convention: lock down the environment first, then pick the model. A 6-line WHY-not-WHAT comment block above the conditional explains the rationale and points at the env-var opt-out. (3) **Opt-out via `ASK_CODEX_LOAD_USER_CONFIG=1`** — when set to literal `"1"`, both flags are omitted from argv and codex loads the user's `~/.codex/config.toml` (including any `[[mcp_servers]]` entries) and `.rules` files normally. The strict `=== "1"` equality check (matching the `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST === "1"` pattern from ADR-069) means any other value (`"true"`, `"yes"`, missing) keeps the deterministic default — explicit opt-in only, no truthy-coercion footgun. (4) Both flags apply on **every** default path — primary call, quota-fallback call, and session-resume call. The opt-out applies on every path too; there's no half-state. (5) Updated the existing `toEqual([...])` argv-shape test to include both new flags in the expected order, added a companion `toEqual([...])` for the resume-path argv shape, and added a new `executeCodexCLI ASK_CODEX_LOAD_USER_CONFIG opt-out` describe block with proper env save/restore in `beforeEach`/`afterEach` covering: default behavior emits both flags, `=1` omits both flags (with full `toEqual` to prove the rest of argv stays identical), opt-out applies on resume path too, and `"true"` (non-`"1"`) does NOT trigger opt-out (regression guard against future truthy-coercion refactors). (6) **Out of scope:** writing the original "smoke-test hooks" in #31 — preempted by this fix. Per-call (rather than per-process) user-config opt-in via an MCP tool argument is also out of scope: env-var-level granularity is sufficient for the use case (user either has custom MCP servers and wants them, or doesn't); per-call would require threading the flag through the MCP tool schema and isn't justified by current demand.
- **Consequences:** `executeCodexCLI` now produces deterministic behavior **by default** regardless of the host machine's `~/.codex/config.toml` or `.rules` files. A user with elaborate hooks in their interactive codex setup gets the same MCP behavior as a user with no config at all. Users with custom MCP servers registered in `~/.codex/config.toml` who want those available inside `ask-codex-mcp` invocations can opt back in with `ASK_CODEX_LOAD_USER_CONFIG=1` — same shape as the Gemini opt-out from ADR-069. The `auth still uses CODEX_HOME` clause in upstream's `--ignore-user-config` documentation means we don't break authenticated calls in either mode — only ignore preference/hook/MCP-server TOML when in the default path. Argv length per call grows by 2 strings (~36 bytes) in the default path; 0 strings in the opt-out path. **Pre-empts a class of future upstream churn**: when codex 0.126+ adds new hook lifecycle events or changes how `.rules` files are evaluated, this MCP wrapper's behavior is insulated by construction. **Closes #31** with the explanation "preempted by hardening" (smoke test was the means, deterministic flags are the end). Future maintenance: when a new codex release adds yet another flag for "ignore X user config," check whether MCP determinism wants the same treatment — the precedent set by this ADR is "yes, by default, with a comment explaining why, and an env-var opt-out for users who want the looser behavior."

## ADR-070: Large-Prompt Stdin Path — 16 KiB Threshold Across Gemini & Codex Executors
- **Date:** 2026-04-28
- **Status:** Accepted (resolves issue #30, deferred from #26 §4.3)
- **Context:** All three CLI-based executors (`packages/gemini-mcp/src/utils/geminiExecutor.ts`, `packages/codex-mcp/src/utils/codexExecutor.ts`, plus `ollama-mcp` which uses HTTP and is not affected) appended the full user prompt as a positional argv element. For typical short prompts that's fine, but `@largefile` expansion in Gemini, `/multi-review` and `/brainstorm` skill dispatches with sizable diffs, and any session-resumed conversation accumulating context could push argv close to the OS `ARG_MAX` ceiling — 256 KiB on macOS, 2 MiB on Linux. Codex `rust-v0.124.0` (released 2026-04-23) explicitly added prompt-plus-stdin support: *"If stdin is piped and a prompt is also provided, stdin is appended as a `<stdin>` block."* Gemini CLI's `-p` flag has *also* documented stdin coexistence for some time: *"Appended to input on stdin (if any)"* — meaning we can pipe the bulk via stdin and pass an empty `-p ""` placeholder (still required for headless mode). Three implementation shapes were considered: (A) extend `executeCommand` with a 5th optional positional `stdin?: string` arg — minimal non-breaking change; (B) refactor `executeCommand` to an options-object signature for cleanliness — affects all four call sites and breaks ABI; (C) introduce a new helper `executeCommandWithStdin()` — duplicates timeout/error/spawn logic. Option A was selected per the project's CLAUDE.md "no premature abstraction" rule.
- **Decision:** (1) `packages/shared/src/constants.ts` gains `EXECUTION.STDIN_THRESHOLD_BYTES = 16_384` (16 KiB) — well below macOS's 256 KiB `ARG_MAX` so non-piped calls never hit the limit either. The threshold is a build-time constant, not env-configurable; if a future use case justifies tuning it per-deployment, that becomes its own scoped change. (2) `packages/shared/src/commandExecutor.ts:executeCommand` now accepts an optional 5th positional `stdin?: string`. Implementation: existing `childProcess.stdin.on("error", ...)` + `.end()` lifecycle is preserved (issue #19's pipe-error fix); when `stdin !== undefined && stdin.length > 0`, `childProcess.stdin.write(stdin)` runs immediately before `.end()`. The empty-string case is treated as a no-op write but still uses the explicit signal — the `length > 0` guard avoids the unnecessary write call. (3) Per-executor wiring: each executor computes `useStdin = prompt.length > EXECUTION.STDIN_THRESHOLD_BYTES` once at the top of the executor function, then both primary and quota-fallback paths share the same `stdinPayload` and threshold decision — DRY by construction, no possibility of one path forgetting the flag. (4) Mechanism per-CLI: **Gemini** — `buildArgs` emits `-p ""` (empty placeholder) when `useStdin`, since headless mode still requires the `-p` flag; the actual content goes via stdin. Per Gemini's CLI help, stdin content concatenates with `-p` text. **Codex** — `buildArgs` skips appending the prompt as positional argv when `useStdin`; per Codex's CLI help, *"`[PROMPT]` is optional ... when absent, instructions are read from stdin."* (5) **Ollama deliberately untouched** — uses HTTP `POST /api/chat` not argv, so has no `ARG_MAX` ceiling. (6) 11 new tests across three packages: 4 in `packages/shared/src/__tests__/commandExecutor.test.ts` (real-spawn round-trip via `node -e "process.stdin.pipe(process.stdout)"` for stdin payload write/EOF, large-payload >16 KiB success, undefined-preserved-existing-behavior, empty-string-no-op); 4 in `packages/gemini-mcp/.../geminiExecutor.test.ts` (15 KiB → argv, 17 KiB → stdin with empty `-p`, fallback path preserves stdin, changeMode-wrapped small prompt stays argv); 3 in `packages/codex-mcp/.../codexExecutor.test.ts` (15 KiB → argv, 17 KiB → stdin no-positional, mini fallback preserves stdin). All existing argv-shape `toEqual([...])` tests stay untouched — the threshold flip is purely transport, never affects argv shape for sub-threshold prompts. (7) **One existing codex test had to be updated** — `passes onProgress callback to executeCommand` previously asserted `toHaveBeenCalledWith(cmd, args, onProgress)` (3 positional args). After this change, all `executeCommand` calls pass 5 positional args (with `undefined` for unused slots), so the test now uses `toHaveBeenCalledWith(cmd, args, onProgress, undefined, undefined)` — explicit, defensive against future signature drift.
- **Consequences:** `/multi-review` and `/brainstorm` with large diffs, `@largefile` expansion against multi-megabyte sources, and any future workflow shoving a big prompt at the executor will no longer collide with the OS `ARG_MAX` ceiling. Bonus privacy improvement: full prompts are no longer visible in `ps` / `ps aux` output for piped calls (only the short argv stub shows). Backward compatibility is preserved — every short-prompt call (<16 KiB) takes exactly the same code path it did before; no spawn semantics change for existing argv-only callers. The signature change to `executeCommand` is non-breaking because the new 5th param is optional. Test count: **+11 tests** (424 → 435 across the monorepo). Lint clean across all 6 packages, build clean. **Operational note caught during implementation:** the prepack-bundle.mjs script copies `packages/shared/dist` into `packages/<dep>/node_modules/@ask-llm/shared/` at publish time, and `postpack-restore.mjs` restores ONLY `package.json` — not the bundled `node_modules`. After a manual `npm publish` (or a publish that fails partway, like the EOTP loop on PR #29), the stale bundled copy lingers and *outranks* the workspace symlink in node module resolution, which silently broke this PR's TS build until I cleaned `packages/<pkg>/node_modules/`. The release pipeline (`release.yml`) runs in fresh CI runners so the cruft can't accumulate there — yet another argument for the tag-driven release flow over manual publish (memory note saved separately). Future maintenance: when the next executor ships (or the codex-mcp/llm-mcp wiring needs touching), this same `useStdin = prompt.length > THRESHOLD` shape applies; copy the existing pattern. Threshold tuning, if ever needed, is a one-line constant change. **Out of scope (intentionally):** (a) Making the 16 KiB threshold env-configurable — the constant is sufficient; tuning needs no caller-time control. (b) Wrapping Gemini-only via `@/tmp/prompt.txt` temp-file syntax instead of stdin — both Gemini and Codex now use stdin uniformly, which keeps the executor abstraction coherent. (c) Streaming the stdin payload in chunks for very-large (multi-MB) prompts — current `childProcess.stdin.write(string)` is fine up to Node's default highWaterMark; if someone hits a 100 MB prompt, that's a different conversation. (d) Updating `commandExecutor` to options-object form — explicit non-goal per CLAUDE.md.

## ADR-069: Gemini Workspace-Trust Compatibility — Default-On Env Var, Friendly Error, No Flash Retry
- **Date:** 2026-04-27
- **Status:** Accepted (resolves issue #26)
- **Context:** gemini-cli v0.39.1 (released 2026-04-24, upstream PR [#25814](https://github.com/google-gemini/gemini-cli/pull/25814)) shipped a `FatalUntrustedWorkspaceError` gate that fires before any model call when `gemini -p` is invoked against a directory that was never marked trusted in interactive mode. Fresh installs of `ask-gemini-mcp` on v0.39.1+ would fail with a raw stderr dump and no actionable guidance — the project's existing stderr handler in `geminiExecutor.ts:379` only specialized RESOURCE_EXHAUSTED, so trust errors fell through to a generic spawn failure surface. Worse, the catch block at `geminiExecutor.ts:491` would fire the quota-fallback path on any non-quota error mismatch, meaning a trust error on Pro would have triggered a Flash retry against the *same untrusted directory* — guaranteed to fail identically, just slower. Three fix approaches were considered: (A) add `--skip-trust` to argv; (B) set `GEMINI_TRUST_WORKSPACE=true` as a parent-process env var that propagates via `getSpawnEnv()`'s `{...process.env, ...}` spread; (C) extend `executeCommand` to accept per-call `extraEnv`. Option A is backward-incompatible — the local install at the time of writing was v0.38.2, which has no `--skip-trust` flag, so a fresh argv flag would error on older Geminis with "unknown option." Option B is the upstream-blessed compatibility surface (the gemini-cli's own CI uses `GEMINI_TRUST_WORKSPACE=true` in E2E workflows) and unknown env vars are silently ignored on older versions, making it forward-and-backward safe. Option C would be the cleanest abstraction but adds a parameter to a shared API for one caller's needs — explicit "no premature abstraction" violation per the project's CLAUDE.md. Option B was selected.
- **Decision:** (1) `packages/gemini-mcp/src/constants.ts` gains a `WORKSPACE_TRUST_PATTERNS` array containing both the typed error class name (`FatalUntrustedWorkspaceError`, stable identifier in stack traces) and the human-readable string (`not running in a trusted directory`, surfaces in stderr text) — matching either makes detection robust against upstream changing how the error surfaces in either channel. New `ERROR_MESSAGES.WORKSPACE_TRUST_REQUIRED` string lists three remediation paths in priority order: unset `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST` (let us re-apply the safe default), set `GEMINI_TRUST_WORKSPACE=true` yourself, or run `gemini` interactively in this directory once and mark it trusted. New `STATUS_MESSAGES.WORKSPACE_TRUST_DETECTED` for the stderr-handler log line. (2) `packages/gemini-mcp/src/utils/geminiExecutor.ts` gains a private `ensureWorkspaceTrustEnv()` helper called at the top of `executeGeminiCLI`. The helper has three idempotency guards: `process.env.ASK_GEMINI_REQUIRE_WORKSPACE_TRUST === "1"` (user opted out — do nothing), `process.env.GEMINI_TRUST_WORKSPACE !== undefined` (user pre-set their own value — don't override), otherwise sets `process.env.GEMINI_TRUST_WORKSPACE = "true"`. The mutation propagates to all subsequent spawned children for free because `shellPath.ts:getSpawnEnv()` already does `{...process.env, PATH: resolveShellPath()}` — no `commandExecutor.ts` changes required. (3) `createGeminiStderrHandler` now tracks both `quotaLogged` and `trustLogged` flags so the handler logs each error class once per invocation. (4) The catch block inserts a workspace-trust check **before** the quota check via a new `isWorkspaceTrustError(message)` helper — this is critical, because without short-circuiting, a trust error on Pro would either silently fall through to the generic re-throw OR (worse) hit the Flash retry path which would just fail again on the same untrusted directory. When detected, the catch throws a fresh `Error(ERROR_MESSAGES.WORKSPACE_TRUST_REQUIRED)` instead of bubbling the raw upstream error. (5) 6 new tests in `geminiExecutor.test.ts` under a new `executeGeminiCLI workspace trust handling` describe block, with proper `process.env` save-and-restore in `beforeEach`/`afterEach` to prevent cross-test env leakage: default behavior sets the env var, `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST=1` skips it, pre-existing user value isn't overridden, both error patterns (`FatalUntrustedWorkspaceError` and `not running in a trusted directory`) throw the friendly message and short-circuit Flash retry, and the orthogonality test that quota errors still trigger Flash fallback unchanged. **Explicitly out of scope:** Action items #2 (stdin piping for >16KiB prompts), #3 (Codex stabilized hooks smoke test), #4 (carried sessionId round-trip + untrusted-workspace integration test), #5–7 (carried-from-#24/#25 items) — each warrants its own focused PR rather than bundling. Issue #26 listed all 7 as candidates with #1 explicitly tagged "Hotfix required? Arguably yes for action #1" — this ADR ships only #1.
- **Consequences:** Fresh installs of `ask-gemini-mcp` on gemini-cli v0.39.1+ now work without manual workspace-trust setup, restoring parity with the ≤ v0.39.0 behavior the MCP server was written against. The env-var approach is forward-and-backward compatible: older Geminis (v0.38.x and earlier, which have no workspace-trust feature) silently ignore the unknown env var; newer Geminis (v0.39.1+) honor it. Power users who *want* workspace-trust enforcement (e.g., shared CI runners where directory provenance matters) can opt in by setting `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST=1` in their MCP config — when they then hit an untrusted directory, they get the friendly remediation message instead of a raw upstream error. Zero argv shape change, so all 25+ argv-based `toEqual([...])` tests in `geminiExecutor.test.ts` stay green untouched — the brittleness of those exact-array assertions now only fires on real argv changes, not env-side tweaks. The Flash retry cascade is preserved for genuine quota errors (orthogonality test confirms) but correctly skipped for trust errors. Test count: 88 in gemini-mcp (was 82, +6 trust tests); 422 across all packages. Lint clean across all 6 packages, full build clean. The friendly error message is markdown-readable so MCP clients render it nicely; the three numbered remediation steps mean a user can copy-paste the right command without reading the upstream PR. Future work: actions #2–7 from issue #26 each get their own PR; the integration-test scaffold for an actual untrusted-workspace round-trip (action #4) is the natural next step but requires either a git-isolated tempdir fixture or a vi-mocked `which` to skip the real `gemini` binary path. Deferred.

## ADR-068: `/codex-image` Plugin Skill — Skill-Only Architecture for Codex Image Generation
- **Date:** 2026-04-24
- **Status:** Accepted
- **Context:** codex-cli 0.125.0 stabilized the `image_generation` feature flag (verified via `codex features list` — stage `stable`, default `true`), making OpenAI's gpt-image-1 model accessible to any `codex exec` session as a model-invoked tool. There is no dedicated `codex image generate` subcommand; image generation is exposed as an internal tool the model decides to invoke when the prompt asks for an image, similar to how it decides to invoke `read_file` or `apply_patch`. Three architectures were considered for surfacing this in the project: (A) skill-only, where a new `/codex-image` plugin skill instructs Claude Code to call the existing `ask-codex` MCP tool with a prompt-engineered template; (B) dedicated MCP tool plus skill, adding `ask-codex-image.tool.ts` with explicit input schema (`prompt`, `outputPath`, `size`, `quality`) and structured output (`{filePath, model, usage}`); (C) hybrid, ship the skill now and promote to a dedicated tool later if non-Claude-Code MCP clients (Cursor, Warp, Copilot) ask for image generation as a first-class primitive. Option B costs ~150 LOC plus tests plus an output-format coupling that may break when Codex changes its image-tool reply format; Option A reuses every existing safety property of `ask-codex` (ADR-044 quota fallback, ADR-042 stdin handling, ADR-047 PATH resolution, ADR-067 model defaults) at zero new code in MCP packages.
- **Decision:** Ship Option A. New skill at `packages/claude-plugin/skills/codex-image/SKILL.md` with `user_invocable: true` and a 5-phase workflow: (1) refine the user's image prompt while preserving intent; only enrich if the prompt is sparse and the conversation provides design context; default-exclude "no humanoid figures, no glowing brains, no chatbot iconography" for AI-topic prompts because those are dominant diffusion-model failure modes. (2) Determine output path with the convention `/tmp/codex-images/$(date +%Y-%m-%d)/<slug>.png` (kebab-case slug derived from the prompt, max 40 chars) — this matches Codex's `--ephemeral` philosophy (ADR-028), avoids polluting user repos, and the date prefix instead of a session-id makes images findable later; user-supplied `outputPath` overrides the default. (3) Dispatch via the existing `ask-codex` MCP tool (NOT raw `codex exec` — explicitly forbidden in the skill instructions, mirroring `/multi-review`'s rule from ADR-064) with a prompt template that names the `image_generation` tool, embeds the absolute output path, and asks the agent to confirm the file size + path in its reply. (4) Verify with `ls -la <path>` after the MCP call returns — never trust the agent's textual claim alone, same verify-before-presenting principle as ADR-064 Phase 3. (5) Specifically catch four failure modes: feature flag disabled, codex-cli version too old, content policy rejection, disk-full/permission-denied — surface each verbatim rather than paraphrasing. **Explicitly out of scope:** (a) a dedicated MCP tool — promotion to Option B is deferred until non-CC client demand justifies the surface cost; (b) per-call model/size/quality flags as MCP arguments — instead, users include those in the prompt body and the model handles them; (c) image input (vision) wrapping — `codex exec -i <FILE>` is the orthogonal direction and gets its own future ADR if surfaced; (d) multi-image batches — single image per call keeps the skill's verify step simple; users can re-invoke for variations. No tests added — skills are markdown prose and don't ship into compiled packages, so the existing test suite stays at 414 tests; functional verification happens via live invocation in Claude Code.
- **Consequences:** Claude Code users get a one-command path to Codex's gpt-image-1 model that automatically inherits every operational safety net the project has built around `ask-codex` over the past 6 months. The skill's prompt-engineering layer (Phase 1) means casual asks like "generate a cat picture" produce reasonable images without the user having to write a full diffusion prompt — but power users can still pass detailed prompts verbatim and skip the refinement. Output path convention means `/tmp/codex-images/2026-04-24/` will accumulate generated images for one day before the OS rotates `/tmp`; users wanting persistence pass an explicit `outputPath`. Zero impact on the published MCP server packages — `ask-codex-mcp@0.3.1` stays at 2 tools (`ask-codex`, `ping`); `ask-llm-mcp@0.3.1` orchestrator unchanged. Plugin gains one skill (now 8 total: brainstorm, brainstorm-all, codex-review, codex-image, compare, gemini-review, multi-review, ollama-review). Future maintenance: if Codex changes the way `image_generation` reports its saved file path in `agent_message` events, the skill's Phase 4 `ls -la` verification still works correctly because it checks the filesystem directly rather than parsing the textual reply — Phase 4 is the format-agnostic safety net for Phase 3's prompting. Future work that's NOT in scope: image-input vision wrapper (separate skill), persistent image gallery / `~/.codex/images/` storage convention, batch generation for A/B comparison of prompt variants (would warrant promotion to Option B's dedicated MCP tool because batch operations benefit from structured outputs).

## ADR-066: `multi-llm` MCP Tool — Parallel Provider Dispatch as One Synchronous Primitive
- **Date:** 2026-04-17
- **Status:** Accepted
- **Context:** The brainstorm-coordinator agent and the `/multi-review`, `/brainstorm`, `/compare` skills all converge on the same atomic operation: "send the same prompt to N providers in parallel, collect all responses." Inside Claude Code, this is achievable via the Agent tool dispatching subagents, or via a Bash tool running parallel CLI invocations. Outside Claude Code (Cursor, Warp, Copilot, any other MCP client), there is no equivalent — those clients have MCP tool calls but no built-in parallel-dispatch primitive. The `ask-llm` orchestrator currently exposes only `ask-llm` (single-provider call), `ping`, `get-usage-stats`, and `diagnose` — none of these compose into "fan out a prompt to all available providers in one call." A non-Claude-Code user wanting multi-provider answers has to issue 2-3 sequential `ask-llm` tool calls and synthesize themselves. The user's same-day strategic pivot to "focus on MCP tools and their usage in Claude Code CLI" plus the candidate triage of #4 ("Subagent orchestration as MCP tools") raised the question of which flavor to ship: generic async `spawn-subagent`/`wait-for-subagent` primitives (Option A) vs a focused synchronous parallel-dispatch tool (Option B). Option A's hidden complexity (state management between tool calls, in-flight process lifecycle, cancellation semantics, persistence across server restarts) is significant and is partially duplicative of Claude Code's existing Agent + Bash tools. The user picked Option B — focused over generic.
- **Decision:** New `packages/llm-mcp/src/multiLlm.ts` module exposes three pure functions: `dispatchMultiLlm(opts)`, `formatMultiLlmReport(report)`, and `buildMultiLlmInputSchema(availableProviders)`. The dispatch function takes `{ prompt, providers, getExecutor, recordUsage? }` and returns `Promise<MultiLlmReport>` where each provider runs in parallel via `Promise.all` and per-provider failures are captured in the result rather than thrown. `Promise.all` is the right primitive here (NOT the ADR-050 Bash dispatch pattern) because the MCP tool handler runs in the persistent server process — there's no sub-agent lifetime issue that ADR-050 was designed to work around. The `MultiLlmReport` shape is `{ dispatchedAt: string, totalDurationMs: number, successCount: number, failureCount: number, results: MultiLlmResult[] }` where each `MultiLlmResult` is `{ provider, ok, response?, model?, sessionId?, usage?, durationMs, error? }`. Both shapes are validated by an exported Zod schema (`multiLlmReportSchema`) so the tool can declare it via `outputSchema` and clients can structurally extract per-provider results. The orchestrator's `index.ts` registers a new `multi-llm` tool inline that delegates to `dispatchMultiLlm`, plumbs `loadedExecutors.get` as the `getExecutor` callback, plumbs `sessionUsage.record` as the `recordUsage` callback (so multi-llm calls feed the same in-memory usage accumulator that `get-usage-stats` reports), and returns both the formatted markdown text (via `formatMultiLlmReport`) AND the structured `MultiLlmReport` (via `structuredContent`). **Explicitly not in this scope**: per-provider session continuity (every multi-llm call is a fresh dispatch — `sessionId` is not in the input schema), per-provider model override (you can't say "use gemini-flash for gemini and gpt-5.4-mini for codex" — each provider uses its default), streaming output (the structured report is built once when all providers complete), cancellation (`Promise.all` waits for the slowest provider; no early termination). All of those are real future enhancements but each adds enough complexity that bundling them risked the V1 scope. 15 new tests in `packages/llm-mcp/src/__tests__/multiLlm.test.ts` cover dispatch happy path (collects responses + sessionId + usage), per-call usage callback, executor-throws → ok=false, missing executor → ok=false, threadId fallback when sessionId is absent, ISO timestamp + numeric duration, parallelism (total time ≈ slowest, not sum), schema conformance round-trip, formatter rendering, input schema validation (provider whitelist enforcement, optional providers, prompt minimum length, fallback when no available providers).
- **Consequences:** The orchestrator now ships 5 MCP tools instead of 4: `ask-llm`, `ping`, `get-usage-stats`, `diagnose`, **`multi-llm`**. The `multi-llm` tool is the first one in the project that's purely orchestrator-only (per-provider servers can't compose multiple loaded executors, so they don't get this tool). Any MCP client — including non-Claude-Code clients like Cursor, Warp, Copilot — can now do multi-provider dispatch in one call: `tools/call multi-llm { prompt, providers: ["gemini","codex"] }` returns `{ structuredContent: { results: [...], successCount, failureCount, totalDurationMs, dispatchedAt } }`. Per-provider failures don't fail the whole call — clients see `results[i].ok = false, results[i].error = "..."` for failed providers and `results[i].ok = true` with response data for successful ones. This is the multi-provider dispatch primitive that `/compare`, `/multi-review`, and `/brainstorm` skills implement via Bash from inside Claude Code — now exposed at the MCP layer for any client. The orchestrator's tool surface grew by approximately 280 tokens (the new tool's schema including the nested `MultiLlmReport` outputSchema), bringing the orchestrator's published surface from ADR-065's ~900 tokens to ~1180 tokens — slightly above the pre-ADR-029 standalone Gemini surface of 973 tokens, but the orchestrator now does meaningfully more (single provider, multi-provider parallel, diagnostics, usage stats, all under one MCP server). Test count: 412 (up from 397, +15 multi-llm tests). **Coexistence with existing skills is intentional**: `/compare` (skill, Claude Code-only, includes Claude verification) and `multi-llm` (MCP tool, any client, raw structured) serve different audiences and shouldn't be merged. Future work: per-provider session continuity (would require `Record<string, string>` sessionId input shape), per-provider model override (similar shape complexity), early-termination support (e.g., "return as soon as 2 of 3 providers respond" for fault-tolerant patterns), streaming the merged report incrementally as providers complete (would require more sophisticated transport handling than the synchronous request-response we currently use).

## ADR-065: `outputSchema` on `ask-*` Tools — Unified `AskResponse` Shape
- **Date:** 2026-04-17
- **Status:** Accepted (extends ADR-055's outputSchema pattern)
- **Context:** ADR-055 added `outputSchema` to two diagnostic tools (`get-usage-stats`, `diagnose`) and proved out the pattern in the SDK. The four free-form provider tools (`ask-gemini`, `ask-codex`, `ask-ollama`, and the orchestrator's `ask-llm`) still returned plain text — programmatic consumers had to regex-parse the response footer to extract sessionId, threadId, or stats. Today the response text is structured by convention only: `[Session ID: abc-123]` for Gemini, `[Thread ID: thread-xyz]` for Codex, and per-provider stats footers (`[Gemini stats: ...]`, `[Codex stats: ...]`, `[Ollama stats: ...]`) — each with a different prefix label and format. A client wanting to feed the sessionId into a follow-up call has to (a) know which provider was called, (b) pick the right regex (`Session ID` vs `Thread ID`), and (c) extract the value. With outputSchema, that whole pipeline collapses to `result.structuredContent.sessionId`. The user's strategic pivot to MCP-tool-surface focus (2026-04-17) makes this the right time — programmatic MCP clients are exactly the audience that benefits.
- **Decision:** New `packages/shared/src/askResponse.ts` defines a canonical `AskResponse` shape via Zod: `{ provider: "gemini" | "codex" | "ollama"; response: string; model: string; sessionId?: string; usage?: UsageStats }`. Two design decisions baked into the shape: (1) **`response` is the raw model text only** — no `[Session ID: ...]` footer, no `Gemini response:` prefix, no `[Gemini stats: ...]` block. The formatted text content in `content[0].text` keeps the human-readable footers for backward compatibility, but `structuredContent.response` is clean for programmatic consumers. (2) **Unified `sessionId` field across providers** — Codex's `thread_id` maps to `structuredContent.sessionId`, Gemini's `session_id` also maps to `structuredContent.sessionId`. Clients can resume any provider's conversation by passing this single field as the `sessionId` arg on the next call, regardless of which provider is chosen — semantically they're both "session-resumption identifier" and the type-level naming difference is a leakage of provider-specific vocabulary. Updated four tools to use the shape: `ask-gemini` (Gemini-MCP), `ask-codex` (Codex-MCP), `ask-ollama` (Ollama-MCP), and the orchestrator's inline `ask-llm` registration. Each tool now adds `outputSchema: askResponseSchema` to its `UnifiedTool` definition (or to the inline `server.registerTool` config for the orchestrator) and changes its `execute` to return `{text: string, structuredContent: AskResponse}` instead of a bare string. The existing `registerTools` factory in `@ask-llm/shared/serverFactory.ts` already branches on the result variant (added in ADR-055), so no changes needed there. **Explicitly out of scope for this round:** `ask-gemini-edit` is excluded because its output is *structured differently* (changeMode OLD/NEW edit blocks parsed into an `EditChunk[]`) — it deserves its own dedicated `editResponseSchema` in a future ADR rather than being shoehorned into the free-form `AskResponse` shape. 7 new tests in `packages/shared/src/__tests__/askResponse.test.ts` validate the schema (minimum required shape, all 3 providers accepted, unknown providers rejected, optional sessionId, full UsageStats, malformed usage rejected, response field type-checking).
- **Consequences:** Programmatic MCP clients can now structurally extract the model response text, the session-resumption identifier, the actual model that answered (which may differ from the requested model when fallback fired), and the full UsageStats — without regex-parsing the response footer. Multi-turn conversation continuation collapses from "look for Session ID or Thread ID label, regex-extract the value, store it, pass it back next call" to "store `result.structuredContent.sessionId`, pass it back next call." Token-cost dashboards can read `result.structuredContent.usage` directly without parsing per-provider stats footer formats. The orchestrator's `ask-llm` benefits most because it already abstracts over providers — clients no longer need provider-specific extraction logic. **Backward compatibility is preserved**: every existing tool call still returns the same `content[0].text` (with the same footers and prefixes); `structuredContent` is purely additive. Clients that don't request or read `structuredContent` see no change. Tool surface impact: each of the 4 tools' published schema grew by approximately 110 tokens (the JSON schema for `AskResponse`), so total surface across the 4 tools grows by ~440 tokens. Per-server impact: ask-gemini-mcp ~110 token surface increase, ask-codex-mcp ~110, ask-ollama-mcp ~110, ask-llm-mcp ~110 (orchestrator). The orchestrator's total surface goes from ADR-058's ~790 tokens to ~900 tokens — still within ADR-029's "much smaller than pre-ADR-029 standalone Gemini's 973 tokens" framing. Test count: 397 (up from 390, +7 schema tests). The shape is intentionally *minimal* — no `requestedModel` vs `actualModel` split, no `formattedText` field duplicating `content[0].text`, no provider-specific extension fields. Future fields can be added as optional without breaking existing clients. Future work: `outputSchema` for `ask-gemini-edit` returning structured edit blocks (deferred); `outputSchema` for `fetch-chunk` (also deferred — its output is similarly structured-different).

## ADR-064: `/multi-review` and `/brainstorm` Polish — Verification, Diff Preprocessing, Failure Resilience
- **Date:** 2026-04-17
- **Status:** Accepted (extends ADR-049, ADR-050, ADR-060)
- **Context:** A real `/multi-review` session against this codebase on 2026-04-17 surfaced four concrete pain points in the existing skill instructions: (1) **No diff preprocessing** — the user had to manually `git add -N` untracked files to include them in the review, and the diff was bloated by ADR/doc additions that providers don't need to "review"; without filtering, a 1206-line diff ballooned to 152KB and providers stalled or hit capacity. (2) **No size guidance** — sending arbitrarily large diffs caused Gemini to time out at 25min and Codex to take 28min on 152KB; users had no documented threshold for when to truncate vs. when to send as-is. (3) **No verification step** — Gemini returned two findings at 95/100 confidence that were *factually wrong* (a `z.enum([])` claim that ignored an existing fallback at line 98 of the same file, and an "MCP SDK doesn't support outputSchema/structuredContent" claim that was directly contradicted by the actual `mcp.d.ts`). Both would have been "fixed" if the skill instructions said "trust the confidence score" — neither was caught until I read the source manually after the multi-review returned. (4) **No fallback dispatch** — when the plugin's reviewer agents (`gemini-reviewer`, `codex-reviewer`) aren't available in the current Claude Code session (e.g., plugin not installed yet), the skill provided no fallback path. The actual session worked around this by dispatching directly via the project's `dist/run.js` and `dist/codex-run.js` runner binaries with the ADR-050 dispatch pattern — but that workaround was discovered ad-hoc rather than documented. The strategic context (the user's same-day pivot to "focus on MCP/plugin") raised the priority of polishing these skills since they ARE the surface where MCP value gets exposed inside Claude Code.
- **Decision:** Substantially expanded `packages/claude-plugin/skills/multi-review/SKILL.md` from 6 brief instruction steps to 5 phases with explicit guidance on each pain point. (1) **Phase 1 (gather and prepare)**: documents `git status --short` first, `git add -N <files>` for untracked, pathspec exclusion of docs/binaries (`:!docs/` `:!apps/docs/` `:!*.md` `:!yarn.lock` `:!*.lock` `:!*.png` `:!*.jpg` `:!*.svg`), and a three-tier size policy (`<50KB` send as-is, `50–150KB` warn the user about expected wall time, `>150KB` ask before sending and offer truncate-or-split). (2) **Phase 2 (dispatch with fallback)**: explicit fallback path — if reviewer agents unavailable, use direct Bash dispatch via `${CLAUDE_PLUGIN_ROOT}/dist/run.js` and `dist/codex-run.js` with the ADR-050 pattern (single foreground blocking call, direct backgrounding, per-PID `wait`, `GMCPT_TIMEOUT_MS=1500000` for 25-min provider timeout, Bash tool `timeout: 600000` for the 10-min outer cap). Explicitly forbids raw `gemini -p` / `codex exec` because they bypass the project's quota fallback (ADR-044), Codex stdin handling (ADR-042), and PATH resolution (ADR-047). (3) **Phase 3 (verify each finding)**: the highest-leverage addition — for every finding above 80/100 confidence, Read the file at the cited line and check if the claim is actually true; classify as `VERIFIED` (claim holds), `REJECTED` (false positive), or `UNVERIFIABLE` (cannot confirm without runtime). This step exists specifically because of the 2026-04-17 false-positives — the example is documented in the skill's preamble. (4) **Phase 4 (resilient failure handling)**: when a provider fails (timeout, exit ≠ 0, 0-byte output), surface the failure inline with stderr instead of silently dropping; explicitly says "a single-provider review is still useful — present what you have." (5) **Phase 5 (synthesis)**: new output structure with `Verified by both`, `Verified by Gemini only`, `Verified by Codex only`, `Rejected`, `Unverifiable`, and per-provider stats including verification counts. The `/brainstorm` skill received a smaller polish since most of its behavior lives in the brainstorm-coordinator agent: added the same diff-preprocessing guidance (filter docs/binaries, warn on `>150KB`), a "confidence scores are not an oracle" reminder citing the same 2026-04-17 false-positive case, and a pointer to `/multi-review` for code-specific source-verified review. The `brainstorm-coordinator.md` agent's Phase 4 synthesis now opens with a "Cross-check high-confidence external claims first" paragraph and adds `VERIFIED / REJECTED / UNVERIFIABLE` classification, plus a new `Rejected (false positives)` section in the synthesis output template. 16 new tests in `packages/claude-plugin/src/__tests__/skills-and-agents.test.ts` pin every load-bearing structural element across the three files: diff preprocessing instructions, size-tier thresholds, verification language, finding classification labels, fallback-dispatch documentation, the ADR-050 pattern, the "no raw CLI" rule, failure-resilience instructions, and the cross-check + rejected-section additions on the agent.
- **Consequences:** `/multi-review` now produces verified findings, not raw provider output. The 2026-04-17 false-positive scenario (where I spent significant time chasing claims that turned out to be wrong) becomes structurally hard to repeat — Phase 3 of the skill explicitly tells Claude to verify before presenting. Diff preprocessing eliminates the entire class of "silent fail because we sent a 4MB diff including yarn.lock and PNGs" — the pathspec filters and size tiers make the cost visible to the user before dispatch. Fallback dispatch means the skill works whether or not the plugin's reviewer agents are loaded in the current session, which matters during plugin development and for users trying `/multi-review` before installing the full plugin. Failure resilience means the user never sees a confident-looking synthesis that's actually based on one provider after the other failed — partial results are explicit. Test coverage: 16 new tests pin every behavior; if a future edit drops the verification step or removes the size tiers, tests fail before the regression ships. Plugin test count: 82 (up from 66; +16 polish tests). Total project tests: 390 (up from 374). No code change in any executor or MCP server — this is purely skill/agent prose polish. The `/multi-review` skill grew from 42 lines to 105 lines; the `/brainstorm` skill grew from 33 to 50 lines; the brainstorm-coordinator agent's Phase 4 grew by 6 lines for cross-checking + rejected section. No tool surface change, no new ADR fields, no new types. Future work: apply the same verification-first pattern to `/gemini-review`, `/codex-review`, `/ollama-review` single-provider skills (deferred — they're simpler and don't have the same multi-provider complexity).

## ADR-063: Session Continuity Hardening — Three Codex/Ollama/Sessions Fixes Bundled
- **Date:** 2026-04-17
- **Status:** Accepted (corrects ADR-058 implementation in three places)
- **Context:** ADR-058 promised end-to-end session continuity across all three providers (Gemini native `--resume`, Codex native `exec resume`, Ollama server-side replay). A `/multi-review` pass on 2026-04-17 caught three concrete bugs in the implementation that all undermine that promise. The findings were Codex-flagged at 90–95/100 confidence and were verified by reading the source. Together they invalidated three of the four claims ADR-058 made: (a) "Codex native session continuity works" — broken because `--ephemeral` was being passed on every Codex call including resume, but `--ephemeral` is documented to skip session persistence, so the Thread ID returned to the user could not actually be resumed; (b) "Server-side replay is safe" — broken because session JSON files were written under `os.tmpdir()` with default `mkdirSync`/`writeFileSync` permissions (typically 0o755 dir / 0o644 file with standard umask), making prompts and Ollama responses readable by any other user on the machine; (c) "Pass empty string to start a new Ollama session" (a documented entry path) — broken because the cache check used `sessionId ? null : ...` truthiness, so the empty string was falsy and the cache was hit instead of creating a fresh session. Strategic context (from the user's "focus on MCP/plugin not CLI" pivot the same day) raised the urgency: under the new focus, ADR-058's session capability is exactly the kind of MCP-tool-surface promise that has to actually work.
- **Decision:** Bundled three small fixes under one ADR rather than three separate ones, since they are conceptually related (all session continuity correctness) and were identified together. (1) **Codex `--ephemeral` only when no session is wanted**: `packages/codex-mcp/src/utils/codexExecutor.ts:buildArgs` now conditionally appends `CLI.FLAGS.EPHEMERAL` only when `sessionId` is undefined. When `sessionId` IS set (initial call where the user wants a resumable thread, or a `resume <id>` call), `--ephemeral` is dropped so Codex persists the session. Also changed the cache-key gate from `sessionId ? null : ...` to `wantsSession = sessionId !== undefined` so that an empty-string sessionId disables caching there too (the same fix applied for Ollama in #3). (2) **Session file permissions hardened**: `packages/shared/src/sessions.ts` now defines `SESSION_DIR_MODE = 0o700` and `SESSION_FILE_MODE = 0o600` constants. `ensureSessionDir()` creates the directory with `{ recursive: true, mode: 0o700 }` AND validates an existing directory via `lstatSync` — rejecting symlinks (defense in depth against tmp races where an attacker pre-creates the path as a symlink to somewhere they want to read), rejecting non-directories, and `chmodSync`-tightening the mode if it's currently world-readable. `saveSession()` now uses an atomic temp+rename pattern: writes to `${finalPath}.tmp.${pid}.${timestamp}` with `mode: 0o600`, then `renameSync` to the final path. Atomic rename means readers never see partial writes; the unique tmp suffix means concurrent saves don't collide. On failure the tmp file is unlinked. The exported `__test__` map gained `SESSION_DIR_MODE` and `SESSION_FILE_MODE` so tests can assert against them. (3) **Ollama empty-string sessionId disables cache**: `packages/ollama-mcp/src/utils/ollamaExecutor.ts` mirrors the Codex fix — `wantsSession = sessionId !== undefined` keys the cache decision on presence, not truthiness. The downstream `appendAndSaveSession(sessionId, ...)` call already correctly handled the empty string (it generated a new UUID via `createSessionId()` when `id` was falsy), so once the cache stops short-circuiting, the documented "pass empty string for new session" UX path works end-to-end. 10 new tests across the three files: 4 in `codexExecutor.test.ts` (ephemeral present without session, omitted with session, exec-resume sequence with session, cache disabled by empty-string sessionId); 4 in `sessions.test.ts` (dir mode 0o700, file mode 0o600, existing world-readable dir gets tightened, no leftover .tmp files); 2 in `ollamaExecutor.test.ts` (undefined sessionId hits cache, empty-string sessionId bypasses cache).
- **Consequences:** ADR-058's three broken sub-features now actually work end-to-end. Codex users can resume conversations via `sessionId: <thread-id>` parameter on the `ask-codex` MCP tool — the second turn correctly references the persisted session instead of dying with "session not found." Ollama session files on macOS/Linux are now owner-only (0o700/0o600), eliminating the cross-user prompt/response disclosure on shared `/tmp/`. Tests skip the permission assertions on Windows (`it.skipIf(process.platform === "win32")`) since POSIX permission semantics don't translate cleanly. The "empty-string sessionId starts a new session" UX path documented in `ask-ollama.tool.ts` and `ask-codex.tool.ts` schemas now actually does what it claims. No tool surface change — these are corrections to existing behavior, not new capabilities. Test count: 374 (up from 364, +10). Lint clean across all packages, full build clean. The lstat-based symlink rejection in `ensureSessionDir` is a defense-in-depth measure against a tmp-race attack (attacker creates `/tmp/ask-llm-sessions` as a symlink to `~/.ssh/` before the first server invocation) — even if `mkdirSync` doesn't follow the symlink, validating before write protects existing-directory cases.

## ADR-062: Interactive CLI REPL — `npx ask-llm-mcp repl`
- **Date:** 2026-04-17
- **Status:** Accepted
- **Context:** After the GitHub Action work was withdrawn (ADR-061), the user redirected effort toward CLI/agentic development. The first piece of that — and the natural composition of everything Wave 3 had already built — is an interactive REPL: a terminal-resident multi-turn shell where the user picks a provider, asks a question, gets a streamed response, and continues the conversation with full prior context. This was sitting in the design space waiting for the supporting pieces to land: session continuity (ADR-058) provides multi-turn state per provider, Gemini stream-json (ADR-057) provides live progressive content via the existing `onProgress` callback, the doctor's CLI subcommand (ADR-056) established the `cli.ts` dispatcher pattern that a `repl` subcommand naturally extends, and session usage tracking (ADR-054/055) gives the user real-time visibility into what their conversation is costing in tokens. None of those required new infrastructure to assemble into a REPL — just thoughtful wiring.
- **Decision:** Added `repl` as a third dispatch path in `packages/llm-mcp/src/cli.ts` (alongside `doctor` and the default `startServer()`). The REPL itself lives in `packages/llm-mcp/src/repl.ts` and exposes four pure-function-testable surfaces (`formatHelp`, `formatBanner`, `handleSlash`, `dispatchPrompt`) plus the `startRepl()` driver. State is held in a `ReplState` interface: `{ currentProvider, sessions: Map<string, string>, available, sessionUsage }` — one session id per provider (NOT one shared session across all providers, because each provider's session format is incompatible with the others). Slash commands implemented: `/help`, `/provider <name>` (switch + auto-resume that provider's session if any), `/providers` (list with active marker), `/new` (clear current provider's session), `/session <id>` (manual resume), `/sessions` (show all per-provider session ids), `/usage` (formatSessionUsage from ADR-054), `/clear`, `/quit` / `/exit`. Anything not starting with `/` is sent as a prompt to the current provider's executor. Streaming: the `onProgress` callback wired through `dispatchPrompt` writes deltas directly to stdout as they arrive — Gemini stream-json (ADR-057) emits assistant message deltas, which surface in the REPL as live progressive output rather than a frozen wait followed by a wall of text. Codex and Ollama don't yet stream progressively, so they print full response when complete (deferred per ADR-057). The REPL captures `result.sessionId ?? result.threadId` from each response and stores it in `state.sessions[currentProvider]` for the next turn — Gemini's `--resume`, Codex's native resume (ADR-058), and Ollama's server-side replay (ADR-058) all flow through the same code path because the executor result types were normalized in ADR-054. Two new exports added to `packages/llm-mcp/src/index.ts`: `ExecutorFn` (was a private type alias) and `getLoadedExecutor(name)` (a getter so the REPL can reach the same loaded executors map that `startServer()` uses without reimplementing provider detection). 29 new tests in `packages/llm-mcp/src/__tests__/repl.test.ts` cover every slash command path, every state mutation in `handleSlash`, and the `dispatchPrompt` happy path / error path / streaming path / executor-missing path using a vi-mocked executor and a Writable stream capture. The full readline loop in `startRepl()` is intentionally NOT unit-tested — its behavior depends on terminal vs pipe input mode, which is brittle to mock and adds little signal. End-to-end smoke-tested live: `printf "/help\n/quit\n" | node dist/cli.js repl` correctly detects all three providers, renders the banner, displays the help, and exits cleanly.
- **Consequences:** Multi-turn conversations across all three providers are now accessible from the terminal without requiring an MCP client. Users who want to debate a design with Gemini can do so iteratively (`gemini> ...`, `gemini> follow up...`, etc.) with `--resume` keeping context. Users who want to compare provider styles can `/provider codex` mid-session and pick up Codex's separate session thread. The 24-hour Ollama session TTL from ADR-058 is the implicit upper bound on REPL conversation persistence. The REPL adds zero MCP tool surface (it's a CLI subcommand, not a tool) — orchestrator's published surface stays at 4 tools. New CLI subcommand surface: `repl` joins `doctor` and the default server-start path. Tests: 37 in `ask-llm-mcp` (up from 8; +29 REPL tests), 364 total across all packages. The REPL is the foundation for further CLI/agentic work — workflow pipelines, provider routing intelligence, etc., all naturally compose on top of `dispatchPrompt`'s primitive. Future enhancements (deferred): multi-line input (heredoc or `:edit` to open `$EDITOR`), persistent state across REPL invocations (currently the `state.sessions` map is process-lifetime; a config file at `~/.ask-llm/last-sessions.json` would let users resume yesterday's conversations on launch), file pasting (`@path/to/file` syntax to inject file contents into the prompt), `/provider gemini --model gemini-3-flash-preview` for explicit model override.

The composite-action pattern means startup cost is dominated by `npm install -g`, typically 15–25 seconds — acceptable for PR review latency. The `ask-llm-version: latest` default means users get quota-fallback fixes (e.g., ADR-044's pattern updates) automatically as new versions ship; users who want pinning can set a specific version. Tool surface impact: zero (this is an external integration, not an MCP tool). Plugin test count: 66 → 97 (+31). Total project tests: 335 → 366. - **Date:** 2026-04-17
- **Status:** Accepted
- **Context:** The `@ask-llm/plugin` package was the only workspace member shipping with `"test": "echo 'No tests yet'"`. It is also paradoxically the most user-visible component (skills, agents, hooks, brainstorm-coordinator) and the one with the most production lifecycle-bug history (ADR-040 hook temp-file leaks, ADR-048 Stop-hook removal, ADR-049 Claude-as-research-participant, ADR-050 sub-agent background-job SIGKILL). The right test discipline for this package is *configuration consistency* — check that the things that matter for plugin loading, skill discovery, agent capability, and hook safety are present and well-formed.

(2) `skills-and-agents.test.ts` (24 tests) — uses `it.each` to validate every skill's `SKILL.md` has required frontmatter (name matches directory, description >20 chars, body present), every agent's frontmatter has name/description/model/color, each reviewer agent declares the right MCP tool, reviewer agents are restricted from `Edit`/`Write`/`NotebookEdit` tools (over-privilege check), and the brainstorm-coordinator specifically has the load-bearing structural elements: runs on `opus`, has `WebFetch`+`WebSearch` (ADR-049's research surface), declares all three external provider MCP tools, and the body contains the "sub-agent background job" warning, "Phase 3A"/"Phase 3B" sequential structure (ADR-049), and "blocking foreground" / "wait" language from the ADR-050 fix. (3) `pre-commit-script.test.ts` (9 tests) — verifies the bash script has `set -euo pipefail`, only fires on `git commit`, excludes secret-bearing pathspecs (`!*.env`, `!*.key`, etc.), uses `mktemp /tmp/ask-llm-`, installs the EXIT/HUP/INT/TERM cleanup trap (ADR-040), caps diff size, respects `ASK_LLM_HOOK_TIMEOUT`, and includes the macOS GUI PATH resolution from ADR-047. (4) `providers.test.ts` (4 tests) — verifies the `ProviderExecutor` array exports exactly three providers with matching name/command and a function `execute`. Each test maps to a documented historical failure mode — accidentally regressing any of the changes in ADRs 040, 048, 049, or 050 will fail a test before reaching review. The "no test coverage of TypeScript code" choice is deliberate — `src/index.ts`'s 35 lines of executor wiring would only catch test-induced bugs if tested with mocks, not real bugs in production. The 4 `providers.test.ts` tests are the minimal sanity check that exists at the boundary. ## ADR-058: Session Continuity Across All Three Providers — Native Resume + Server-Side Replay
- **Date:** 2026-04-16
- **Status:** Accepted (partially supersedes ADR-034's `sessionId` schema removal)
- **Context:** The project's primary use case (per `CLAUDE.md`) is "low-friction back-and-forth between Claude and Gemini for validation and debate." That requires multi-turn conversations: ask provider X to review a plan, get pushback, refine, ask again with prior context intact. Today only Gemini's executor accepts a `sessionId` (ADR-021's `--resume`), and ADR-034 deliberately *removed* `sessionId` from the `ask-gemini` MCP tool schema for simplicity, leaving session IDs surfaced in responses but unable to be passed back in. Effectively no provider supports continuation through the MCP surface. The Codex CLI 0.118 has native `codex exec resume <SESSION_ID> [PROMPT]` support — the same `thread_id` already returned in `thread.started` events is the resumable id. Ollama has no native session support but its `POST /api/chat` accepts a full `messages` array; conversation state can be maintained server-side and replayed on each call. Decision C from the user's earlier triage locked the design: server-side store at `/tmp/ask-llm-sessions/`, mirroring the existing `chunkCache.ts` pattern, with Codex and Ollama as the providers needing it. Investigation revealed Codex actually has native resume too — server-side replay is only required for Ollama. This narrows the blast radius further than the original triage assumed.
- **Decision:** (1) New `packages/shared/src/sessions.ts` mirrors `chunkCache.ts`'s patterns: file-based store at `os.tmpdir()/ask-llm-sessions/<id>.json`, 24-hour TTL (longer than chunkCache's 10 minutes since conversations span longer), 200-file cap (vs chunkCache's 50 since concurrent conversations are common). The `SessionRecord` shape is `{ id, provider, model, messages: SessionMessage[], createdAt, updatedAt }` where each message is `{ role: "user" | "assistant", content: string }`. A `MAX_MESSAGES_PER_SESSION = 40` cap prevents unbounded context growth — when exceeded, the OLDEST messages are dropped, keeping the most recent 40 (20 user/assistant pairs). The store rejects unsafe session IDs at both read and write boundaries via `isSafeSessionId()` regex (`/^[A-Za-z0-9_-]{8,128}$/`), preventing path traversal even before the file system gets involved. Public API: `createSessionId()` (UUID via `node:crypto.randomUUID`), `loadSession(id)`, `saveSession(record)`, `appendAndSaveSession(id, provider, model, userContent, assistantContent)` (the high-level helper that loads-or-creates, appends, caps, saves, and returns the assembled state), and `buildPriorMessages(id)` (load helper returning the message array for replay). 18 unit tests cover round-trip, TTL expiry, message-cap trimming, unsafe-id rejection, and the create-or-resume semantics. (2) Codex executor gains `sessionId?: string` option. When provided, `buildArgs` emits `codex exec resume <SESSION_ID> [flags] <prompt>` instead of `codex exec [flags] <prompt>` — same flags (`--full-auto`, `--json`, `--ephemeral`, `--skip-git-repo-check`, `-m model`) work with both. Verified via `codex exec resume --help`. The existing `thread.started` event capture means the returned `threadId` is already the resumable id — users paste it back as `sessionId` to continue. Response cache is bypassed when sessionId is set (same prompt in different conversations should yield different answers). (3) Ollama executor gains `sessionId?: string`. When provided, `buildPriorMessages(sessionId)` loads the prior history and the executor sends `[...prior, { role: "user", content: prompt }]` to `POST /api/chat` instead of just the new message. After a successful response, `appendAndSaveSession` records the new turn. The response footer includes `[Session ID: <id>]` (and `(new)` if the executor created the session — i.e., the user passed an unknown id or none). Cache also bypassed when sessionId is set. (4) Tool schemas: re-added `sessionId: z.string().optional()` to `ask-gemini`, `ask-codex`, `ask-ollama`, AND the orchestrator's `ask-llm` tool. The orchestrator's `ExecutorFn` type signature gained `sessionId?: string` input + `sessionId?: string` / `threadId?: string` output, so it routes the value to the right executor and surfaces the appropriate id in the response footer. This is a partial reversal of ADR-034's "removed sessionId from tool schemas — sandbox is a niche feature, sessionId is returned in responses for manual use." The reversal acknowledges that "manual use" means nothing in MCP context where the schema *is* the only interface. The cost is one optional parameter per tool; the value is end-to-end multi-turn capability across all three providers.
- **Consequences:** All three providers now support multi-turn conversations through the MCP surface. Gemini and Codex use native CLI resume (zero replay cost — the providers retain state). Ollama uses server-side replay (the prior `messages` array is sent on each turn, costing input tokens proportional to conversation depth — but Ollama runs locally with no token bill, and the 40-message cap bounds growth). Tool surface impact: each tool gains one optional parameter (~50 tokens added to each tool's published schema). Aggregate: ask-gemini + ask-codex + ask-ollama + ask-llm orchestrator = ~200 tokens added across the published surface. The orchestrator's surface goes from ADR-056's ~740 tokens to ~790 tokens — still 19% smaller than pre-ADR-029 standalone Gemini (973 tokens). ADR-034's "exactly four tools, simplified to 2 params" claim is technically violated for `ask-gemini` (now 3 params), but ADR-034's intent (don't over-expose niche options) is preserved by keeping `sessionId` optional and clearly described. Security: `isSafeSessionId` blocks `../`, slashes, and shell metacharacters at both read and write boundaries — the file path can never escape `/tmp/ask-llm-sessions/`. The Ollama session store opts users in to local persistence of conversation history; the description on the tool schema makes this explicit. 269 tests pass (18 new sessions tests in `@ask-llm/shared`), lint clean across all 6 packages, full build clean, no smoke-test failures. Future work: an MCP `resources/list` entry could expose `sessions://list` for users to discover their active sessions — deferred to a follow-up ADR.

## ADR-057: Stream-JSON Output for Gemini Executor — Live Progressive Content via Progress Notifications
- **Date:** 2026-04-16
- **Status:** Accepted
- **Context:** Gemini calls through the MCP server are inherently slow — large-context analyses on `gemini-3.1-pro-preview` routinely take 60–180 seconds, and the brainstorm-coordinator's external dispatches can stretch to 10 minutes (ADR-050). The existing executor uses `--output-format json` which buffers the full response and emits it as a single JSON object at the end. During execution, the project's `createProgressTracker` (ADR-039/041) emits keep-alive notifications every 25 seconds with the last 150 chars of raw stdout — but raw stdout is unparsed CLI noise, not the model's text, so the notifications display gibberish like `"text":"...","delta":true},{"type":"messa` mid-token. Users perceive long calls as unresponsive even when the model is actively producing tokens. Gemini CLI 0.37+ supports `--output-format stream-json` which emits a JSONL event stream: `{"type":"init",...}` / `{"type":"message","role":"assistant","content":"...","delta":true}` per chunk / `{"type":"result","status":"...","stats":{...}}` at end. Switching to this format unlocks two things: (1) the `--include-directories` analysis can stream prose to the user as the model produces it; (2) the existing `onProgress` plumbing (already wired through `executeCommand` → `createProgressTracker.updateOutput`) already accepts text — we just need a parser that extracts assistant content from JSONL events and forwards it. The MCP client (Claude Code, Claude Desktop, etc.) renders progress notifications inline, so the user sees real model output unfolding instead of a frozen spinner. Codex and Ollama have their own streaming protocols (Codex JSONL is already collected at end, Ollama HTTP `stream: true` would be an HTTP body refactor) — they're explicitly out of scope for this ADR; one provider at a time keeps blast radius small and lets us verify the pattern before generalizing.
- **Decision:** (1) Added `STREAM_JSON: "stream-json"` to `OUTPUT_FORMATS` in `packages/gemini-mcp/src/constants.ts` and switched `buildArgs` to emit it in place of `JSON`. (2) New `parseGeminiStreamJsonl(raw, model, durationMs, fellBack): GeminiExecutorResult` function in `geminiExecutor.ts` that splits raw stdout by newlines, parses each JSON line, and aggregates: `init.session_id` → sessionId; `message` events with `role: "assistant"` → assembled content (delta:true appends, non-delta replaces); `result.stats` → token usage; `result.status === "error"` or standalone `error` events → throw. The stream stats shape (`models[name]: {input_tokens, output_tokens, cached, ...}`) is converted to the canonical `GeminiCliStats` shape (`models[name]: {tokens: {input, candidates, cached, thoughts}}`) via a small `streamStatsToCliStats` helper, so the existing `formatStats` and `buildUsageStats` code paths work unchanged. (3) New `makeStreamingProgressForwarder(onProgress)` factory wraps the original `onProgress` callback. It maintains a buffer across chunks (since chunks can split mid-line), splits on newlines as they complete, parses each line as JSONL, and forwards just the `content` of `message` events with `role: "assistant"` to the original `onProgress`. User and non-message events are filtered out. Malformed lines are silently skipped. (4) `executeGeminiCLI` now wraps `onProgress` with the forwarder before passing to `executeCommand`, and uses `parseGeminiStreamJsonl` instead of `parseGeminiJsonOutput` for the final parse. Both the primary path and the Flash quota-fallback path use the streaming pair. (5) Backward compatibility: `parseGeminiStreamJsonl` detects the case where no parsed line had a `type` field (i.e., the output isn't actually stream-json — older Gemini CLI versions, or a CLI version that ignored the flag) and falls back to the existing `parseGeminiJsonOutput`. Existing tests using `JSON.stringify({response:"..."})` mocks continue to work via this fallback path. (6) 18 new tests in `geminiStreamJsonl.test.ts` cover the parser (delta accumulation, non-delta replace, sessionId capture, stats conversion, error events, malformed lines, legacy fallback) and the forwarder (buffering across chunks, multi-event chunks, role filtering, no-op when callback absent). The pre-existing `geminiExecutor.test.ts` got 2 mechanical updates (3 args-shape assertions changed `JSON` → `STREAM_JSON`, and 3 string-literal `"json"` checks changed to `"stream-json"`).
- **Consequences:** Long Gemini calls now display the model's prose progressively in MCP progress notifications. The primary UX pain point for `/brainstorm`, `/multi-review`, `/gemini-review`, and `ask-gemini` direct calls is addressed. The streaming forwarder is provider-agnostic in spirit (it's just a JSONL chunk filter) — when Codex and Ollama get their own streaming refactors, the same per-event filter pattern applies; only the event shapes differ. The legacy fallback in `parseGeminiStreamJsonl` means we can ship without breaking users on Gemini CLI < 0.37 — they get the old buffer-and-parse behavior (just without live progress). No tool surface change: the MCP-facing schema is identical. No ADR supersedence — this is purely an executor-internal change. 251 tests pass (18 new in `gemini-mcp`, total now 251), lint clean, full build clean. Smoke-tested live with a real prompt: stream events arrive in order, deltas accumulate to the same text the buffered parser would have produced, sessionId and stats land correctly. Future per-provider streaming (Codex `--output-format jsonl` with progressive parse, Ollama HTTP `stream: true` body iteration) is a natural follow-up but not required.

## ADR-056: `doctor` Diagnostic Surface — CLI Subcommand on Orchestrator + `diagnose` MCP Tool
- **Date:** 2026-04-16
- **Status:** Accepted
- **Context:** `docs/BUGS.md` is dominated by environment and install-time failures: Node v18 incompatibility with gemini-cli's ES2024 regex `v` flag (ANT-242), macOS GUI apps not inheriting shell PATH (ADR-047), the npm 9 `EUNSUPPORTEDPROTOCOL workspace:*` install failure (ADR-052), Codex CLI hanging on approval prompts without `--full-auto` (ADR-046), Gemini quota error pattern changes (ADR-044), Codex stdin pipe errors in agent sub-processes (ADR-042), the rate-limit-self-defeating loop (ADR-051). Each new user environment surfaces a slightly different combination of these. Users currently have to either read the BUGS.md tail, infer from cryptic errors, or open a GitHub issue. The project has a `Logger.checkNodeVersion()` startup probe (ADR-046) and `shellPath.ts` PATH resolution (ADR-047), but these are runtime behaviors, not user-runnable diagnostics — and crucially, when the MCP server fails to start, neither runs at a point the user can observe. A user-runnable diagnostic that works *outside* the MCP transport would convert this entire failure mode class from "open a GitHub issue" into "paste the doctor output." Two surfaces are needed: a CLI form (`npx ask-llm-mcp doctor`) for the case where the MCP server cannot start at all, and an MCP tool form (`diagnose`) for the case where the server is up but a provider is misbehaving. Decision A from the user's earlier triage locked both surfaces.
- **Decision:** (1) New `packages/shared/src/doctor.ts` defines the canonical diagnostic shape: `DiagnosticReport` (status: `ok`/`warning`/`error`, generatedAt, environment, providers, checks), `DiagnosticCheck` (per-check pass/warn/fail with optional `fix` hint), `ProviderProbe` (per-provider availability + cliPath + cliVersion + error), and a `ProviderSpec` input shape that callers populate with provider-specific information. The `runDiagnostics(specs)` function performs: Node version check (>= v20), PATH resolution check (uses the existing `resolveShellPath()` from ADR-047), per-provider probe (CLI providers via `which`/`where` + `<command> --version`, HTTP providers like Ollama via the supplied `probeAvailability()` callback that does an HTTP HEAD against the local endpoint), and ASK_LLM_PATH override warning. `formatDiagnosticReport(report)` produces a human-readable text rendering with check glyphs (✓/!/✗). The function is provider-agnostic — the orchestrator passes its provider list, but per-provider servers can pass their own subset if they ever add a doctor surface later. 16 unit tests cover formatter rendering, status calculation, env probe, and provider probe (with a deliberate `command: "this-command-definitely-does-not-exist-..."` to verify graceful unavailable handling without mocking). (2) `packages/llm-mcp/src/cli.ts` is upgraded from "`startServer().catch(...)`" to a tiny dispatcher that detects `process.argv[2] === "doctor"`, runs the diagnostic, and exits with code 0 if status is `ok`/`warning` or 1 if `error`. Supports `--json` flag for machine-readable output. Provider specs are built by `packages/llm-mcp/src/utils/providerSpecs.ts` from the existing `PROVIDERS` registry — the `availabilityModule`/`availabilityFn` indirection from ADR-029 is reused so HTTP providers like Ollama get the right reachability check. (3) New `createDiagnoseTool(providers): UnifiedTool` factory in `serverFactory.ts` mirrors `createUsageStatsTool`: `outputSchema` defined as a Zod shape matching `DiagnosticReport`, execute returns `{text, structuredContent}` from a single `runDiagnostics()` call. The orchestrator's `index.ts` registers it inline via `server.registerTool()` alongside `ask-llm`, `ping`, and `get-usage-stats`. Per-provider servers (gemini-mcp, codex-mcp, ollama-mcp) do NOT get the `diagnose` tool — they would only see their own provider, so the surface adds little value while increasing the tool-count footprint that ADR-054 already widened. The CLI doctor remains the universal access point for per-provider needs. (4) Out of scope for this ADR (deferred): per-provider CLI binaries also gaining a `doctor` subcommand (could be added trivially later); auth-status probing (would risk triggering interactive flows); live `ping`-style provider invocations (would consume tokens and miss the diagnostic-only purpose).
- **Consequences:** Tool count: orchestrator goes from ADR-055's 3 tools (ask-llm, ping, get-usage-stats) to 4 (adds diagnose). Per-provider counts (Gemini 5, Codex/Ollama 3) are unchanged. The new tool's outputSchema adds approximately 220 tokens to the orchestrator's published surface (the nested check/provider/environment shapes), bringing the orchestrator's total surface to roughly 740 tokens — about 24% smaller than pre-ADR-029 standalone Gemini (973 tokens), and the increase buys a structured surface for diagnostics that was previously absent entirely. The CLI doctor adds zero MCP tool surface and is the higher-value surface for the typical failure mode (server won't start). Future BUGS.md entries that describe environment misconfigurations should be matched by a corresponding check in `runDiagnostics` — the doctor's value compounds as more checks are added. The `ProviderSpec` shape is intentionally caller-supplied so the doctor module stays decoupled from the orchestrator's provider registry and could be reused by per-provider binaries later if needed. 233 tests pass (16 new doctor tests in shared), lint clean, full build clean. Smoke-tested end-to-end on the maintainer's environment: detects Node 24.13.0, 52 PATH entries, gemini 0.37.0, codex-cli 0.118.0, and Ollama endpoint reachable; exits 0 on overall `ok`.

## ADR-055: Add MCP `outputSchema` to `get-usage-stats` and Expose Live Snapshot as a Resource
- **Date:** 2026-04-16
- **Status:** Accepted (extends ADR-054)
- **Context:** ADR-054 promoted `get-usage-stats` to a first-class tool returning a markdown-formatted summary as `content[0].text`. That works for human consumers but is hostile to programmatic ones — any client that wants to read individual fields (e.g., to drive a dashboard, enforce a per-session token budget, or feed totals into a parent agent's context) has to parse markdown. The MCP spec since 2025-06-18 supports `outputSchema` on tools, where a Zod schema declares the shape of `structuredContent` returned alongside the text content. Clients that introspect `structuredContent` get the data structurally; clients that don't ignore the field and read the text. SDK 1.27.1 supports both `outputSchema` in `registerTool` config and `registerResource` for static URI-keyed resources. Separately, MCP Resources (distinct from Tools) are the spec-blessed surface for "pollable state" — a tool call implies side effects, a resource read implies inspection. The session usage snapshot is a textbook fit: it is read-only, idempotent, and a resource read costs zero against the tool-token budget that ADR-029 established as a load-bearing invariant.
- **Decision:** (1) Extended `UnifiedTool` with an optional `outputSchema?: ZodTypeAny` field and a `ToolResult = string | { text: string; structuredContent: Record<string, unknown> }` return type for `execute`. Tools that don't opt in keep returning a plain string and behave identically to before this ADR. Tools that opt in return both representations from a single `execute` call — a single source of truth, no risk of text/structured drift. `registerTools` in `@ask-llm/shared/serverFactory.ts` detects the union variant at runtime and emits the appropriate `CallToolResult` (`{content, structuredContent}` when structured, `{content}` when plain string). The orchestrator (`llm-mcp/src/index.ts`), which has its own bespoke registration path, applies the same branch inline. (2) `createUsageStatsTool` factory now returns a tool with `outputSchema: sessionUsageSnapshotSchema` (a Zod definition mirroring `SessionUsageSnapshot`: totals, per-provider buckets, per-model buckets) and an `execute` that returns both the markdown text and the raw `SessionUsageSnapshot` cast to `Record<string, unknown>`. Six new unit tests verify schema/structured-content round-trips and the per-instance closure isolation. (3) New `registerSessionUsageResource(server, sessionUsage)` helper in `serverFactory.ts` registers an MCP Resource at the URI `usage://current-session` with `mimeType: application/json`. The read callback returns the live `JSON.stringify(sessionUsage.snapshot(), null, 2)` — re-reading at any time yields the current totals, no caching. All four servers wire it in alongside the tool registration. (4) Explicitly out of scope: `outputSchema` for `ask-*` tools (their responses are free-form LLM text — structuring is artificial and would require a parsing layer that doesn't exist today); `models://<provider>` resources for model discovery (would require new provider model-list APIs and per-provider availability handling — separate scope, deferred).
- **Consequences:** Programmatic consumers can now read structured `SessionUsageSnapshot` data via either path: a `tools/call` of `get-usage-stats` returns `structuredContent` (clients that don't understand the field gracefully ignore it and use the markdown), or a `resources/read` of `usage://current-session` returns canonical JSON. Both surfaces share the same `sessionUsage.snapshot()` source — no duplication or drift risk. Tool-token-budget impact is bounded: `outputSchema` adds approximately 95 tokens to the `get-usage-stats` registration (the JSON schema for the snapshot) — present once per server. The orchestrator surface goes from ADR-054's ~422 tokens to ~517 tokens (still 47% smaller than the pre-ADR-029 standalone Gemini surface of 973 tokens). MCP Resources sit in their own `resources/list` namespace and contribute zero to the `tools/list` budget. The tool-count claim from ADR-054 is unchanged (Gemini 5, others 3) — adding `outputSchema` doesn't add a tool. Resources are an entirely new published surface for this project, but they're the spec-blessed home for inspection-only state and ADR-029's token-efficiency principle is preserved by construction. The pattern is now in place for future `outputSchema` opt-ins on other tools when the data shape genuinely warrants it. 217 tests pass, lint clean.

## ADR-054: Add `get-usage-stats` as First-Class Tool, Remove Smithery Sandbox Infrastructure
- **Date:** 2026-04-16
- **Status:** Accepted (supersedes ADR-017; revises tool-count claims in ADR-004, ADR-029, ADR-034)
- **Context:** Two related changes landed together. (1) Phase 1 of session usage tracking (introduced earlier the same day) added a `get-usage-stats` MCP tool to all four servers via a `registerUsageStatsTool(server, sessionUsage)` helper that registered the tool inline, bypassing the shared `toolRegistry`. A multi-provider review (Codex completed; Gemini timed out due to a global `gemini-3.1-pro-preview` capacity outage) flagged two consensus-class issues at confidence 91/100 and 88/100: the new tool silently expanded the published tool surface in violation of three accepted ADRs (ADR-004's "exclusively `ask-gemini`/`fetch-chunk`/`ping`" minimality principle, ADR-029's explicit "ask-llm + ping = 312 tokens" orchestrator surface, and ADR-034's "Gemini MCP now exposes 4 tools" enumeration), and `createSandboxServer()` (used by Smithery for capability scanning per ADR-017) iterated `toolRegistry` only — so the sandbox-scanned schema diverged from the runtime schema for the new tool. (2) ADR-017's entire sandbox infrastructure existed solely for Smithery integration, and the Roadmap recorded Smithery as "skipped (requires paid plan for stdio servers)" — meaning `createSandboxServer()` had no live consumer in any of the four published packages. Combined: the only code path that would have surfaced finding (1) was itself dead code from finding (2).

(2) Remove all Smithery sandbox infrastructure: `createSandboxServer()` is deleted from `@ask-llm/shared/serverFactory.ts` and from `@ask-llm/shared/src/index.ts`; the matching `export function createSandboxServer()` wrappers are removed from `gemini-mcp/src/index.ts`, `codex-mcp/src/index.ts`, and `ollama-mcp/src/index.ts`. The `createRequire(import.meta.url)` try/catch fallback (also from ADR-017) is retained — it is harmless and provides defensive protection if the package is ever loaded in an unusual context. (3) Tool-surface counts are now: `ask-gemini-mcp` 4 → 5 tools (ask-gemini, ask-gemini-edit, fetch-chunk, ping, **get-usage-stats**); `ask-codex-mcp` 2 → 3 tools (ask-codex, ping, **get-usage-stats**); `ask-ollama-mcp` 2 → 3 tools (ask-ollama, ping, **get-usage-stats**); `ask-llm-mcp` orchestrator 2 → 3 tools (ask-llm, ping, **get-usage-stats**). The new tool's schema footprint is approximately 110 tokens (description + empty input schema + annotations), so the orchestrator's previous 312-token surface (ADR-029) becomes ~422 tokens — a 35% increase in absolute terms but still ~57% smaller than standalone Gemini before ADR-029 (973 tokens). The diagnostic value of giving every MCP client immediate, structured visibility into per-session token spend, fallback frequency, and provider/model breakdowns justifies the cost; users sensitive to tool-count overhead retain the option to filter at the client layer if their client supports it. ADR-017's `createSandboxServer` design is superseded — Smithery is not a target distribution channel and there are no other consumers.
- **Consequences:** Tool registration is now consistent across all four servers — every tool, including `get-usage-stats`, flows through the same path with the same guarantees (concurrency-safe progress, sanitized errors, annotations honored). The factory pattern keeps the closure over per-instance `SessionUsage` while remaining DRY; ADR-039's "no module-level mutable state" principle is preserved because the closure is created at server-instance startup rather than at module load. Tool-count claims in ADR-004, ADR-029, and ADR-034 are now historical — they describe the surface as of their respective dates but no longer reflect the current invariant. Future ADRs that change tool counts should reference this ADR as the canonical baseline. Smithery support is permanently removed; the project's distribution channels are npm + the official MCP Registry + the Claude Code marketplace, none of which require sandbox scanning. ~50 lines of dead infrastructure removed. All 211 tests pass, lint clean. The two findings from the multi-provider review are now both resolved by construction: there is no surface divergence (no sandbox), and the tool surface change is documented and accepted via this ADR.

## ADR-052: Rewrite workspace:* → * in Published MCP Packages to Fix npm 9 EUNSUPPORTEDPROTOCOL
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** User report: `npx -y ask-llm-mcp` from Claude Desktop (which ships with Node 18 / npm 9.7.1) fails immediately during install with `npm ERR! code EUNSUPPORTEDPROTOCOL — Unsupported URL Type "workspace:": workspace:*`. The server never boots. Investigation revealed that all four published MCP packages (`ask-gemini-mcp@1.5.5`, `ask-codex-mcp@0.2.5`, `ask-ollama-mcp@0.2.5`, `ask-llm-mcp@0.2.5`) have `"@ask-llm/shared": "workspace:*"` (and for `ask-llm-mcp`, also the three other sibling packages) literally in their published `dependencies` field. The bug had been latent in the published packages for a long time — it only surfaced because of a subtle distinction between two install paths: (1) `npm install tarball.tgz` reads the local tarball directly, respects `bundledDependencies`, and uses the bundled `node_modules/` inside the tarball — succeeds on npm 9; (2) `npm exec`/`npx -y package` fetches the package MANIFEST from the npm registry FIRST (literally the published `package.json`), computes the dependency tree from that manifest via the arborist, and only THEN downloads the tarball. npm 9's arborist doesn't recognize the `workspace:` URL type in a registry manifest and fails before it ever sees the bundled deps. `bundledDependencies` does not help on the manifest-fetch path. Confirmed empirically: installing the `ask-llm-mcp@0.2.5` tarball with `npm install /path/to/tarball.tgz` on Node 18.15.0 / npm 9.7.1 succeeds, but `npx -y ask-llm-mcp` on the same machine fails with `EUNSUPPORTEDPROTOCOL`. The "works for some users, fails for others" pattern was explained by npx cache: once an extraction existed at `~/.npm/_npx/<hash>/`, successive runs reused it and never re-ran the manifest-fetch path. A new version bump invalidated the cache and cold-installs failed.
- **Decision:** Added `scripts/prepack-bundle.mjs` — a small Node helper that each MCP package calls from its `prepack` script. The helper does three things: (1) backs up the package's `package.json` → `package.json.bak` so postpack can restore it; (2) copies the requested workspace sibling packages (`packages/shared/dist` + `package.json`, etc.) into `./node_modules/` under the npm-package names (`@ask-llm/shared`, `ask-gemini-mcp`, etc.) — this is the existing `bundledDependencies` bundling mechanism; (3) rewrites `workspace:*` → `*` in the top-level `package.json` AND in every bundled nested `package.json`. The `*` semver range is universally parseable by every npm version, and at install time npm honors `bundledDependencies` by using the bundled folder instead of going back to the registry. Added `scripts/postpack-restore.mjs` that runs after `npm pack` (including after `npm publish`) to restore `package.json` from the backup, so the developer's working tree still has `workspace:*` and yarn's workspace resolution continues to work at dev time. Each MCP package's `prepack` was replaced with `node ../../scripts/prepack-bundle.mjs <deps>` — gemini/codex/ollama just pass `shared`, llm-mcp passes all four (`shared gemini-mcp codex-mcp ollama-mcp`). Each MCP package also got a `postpack: node ../../scripts/postpack-restore.mjs` entry. Added `packages/*/package.json.bak` to `.gitignore` as defensive cleanup for aborted packs. Rejected alternatives: (a) rewriting `workspace:*` → a concrete version like `^0.2.6` — `@ask-llm/shared` is private (not published), so the registry would 404 on resolution; (b) using `file:./node_modules/...` paths — more complex and no better than `*` with bundledDependencies; (c) switching to `yarn npm publish` — Yarn Berry's publish rewrites workspace protocol automatically, but `yarn pack` ignores `bundledDependencies` entirely (per the explicit comment in `.github/workflows/release.yml`), so switching would regress the bundled-dep shipping mechanism; (d) fixing this at install time with a `preinstall` hook — runs too late, the arborist parses the manifest before any install hooks fire.
- **Consequences:** Published `ask-llm-mcp@0.2.7`, `ask-codex-mcp@0.2.7`, `ask-ollama-mcp@0.2.7`, `ask-gemini-mcp@1.5.7` install cleanly under npm 9 (Claude Desktop's Node 18) as well as npm 10/11. End-to-end verified: built tarball, inspected extracted contents, installed under Node 18.15.0 / npm 9.7.1, executed the bundled CLI. The developer working tree still has `workspace:*` after `npm publish` completes (postpublish restores from `.bak`), so yarn workspace resolution continues to work unchanged for dev. Bumped `@ask-llm/shared` to 0.2.7 for consistency even though it isn't published.
- **2026-04-10 correction — `postpack` vs `postpublish`:** The first release attempt (1.5.6 / 0.2.6) registered the restore as a `postpack` hook. This produced a subtle split-brain: the published *tarball* had the rewritten `"*"` (correct), but the published *registry manifest* still had `"workspace:*"` (broken). Root cause: npm's publish pipeline reads `package.json` metadata for the manifest **AFTER** `postpack` runs (visible via the `npm warn publish auto-corrected` log line appearing after postpack output). So `postpack` restoring the working tree undid the rewrite before npm read it for the manifest. Moving the restore from `postpack` to `postpublish` fixes it — `postpublish` runs after the manifest has been uploaded to the registry, so the manifest reflects the rewritten `"*"`. Verified by running `npm publish --dry-run` and observing the ordering: `prepack → pack → npm warn publish (manifest read, still rewritten) → Publishing → postpublish (restore)`. The 1.5.6 / 0.2.6 versions are now *tombstones* on npm — their tarballs are usable via `npm install tarball.tgz` but `npx -y` still fails on them because the registry manifest is broken. **Users must install 1.5.7 / 0.2.7 or later to get the working fix.** Trade-off: `postpublish` only runs on `npm publish`, not on `npm pack`, so `npm pack` alone leaves the working tree rewritten — a developer running `npm pack` manually must `git checkout packages/X/package.json` to restore. This is acceptable because `npm pack` is rarely run outside CI, and the failure mode is visible (working tree shows `"*"` instead of `"workspace:*"`) rather than silent.

## ADR-051: Smoke Tests Skip-With-Warning on Quota/Rate-Limit Errors
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** ADR-043 introduced a Husky pre-push hook that runs real smoke tests against locally-installed CLIs (Gemini, Codex, Ollama) on every push. The intent was a "developer hygiene" gate that catches broken provider integrations before they reach `main`. In practice, the design has a self-defeating feedback loop: the smoke test itself makes a real Gemini API call, which counts against the user's quota window. When a developer pushes multiple commits in a single working session (as happened during the #22/#23/Stop-hook release sequence on 2026-04-10), the smoke test on push N can fail with HTTP 429 simply because pushes 1..N-1 burned the quota. Empirical evidence from the 2026-04-10 session: smoke tests passed at 12:19, 12:27, 14:22; failed with `rateLimitExceeded` at 13:01 and 14:31. The 14:22 → 14:31 gap was only 9 minutes — well within the same rate-limit window — so the success-then-failure pattern is direct evidence that the smoke test is sabotaging itself. The failing test is in `packages/gemini-mcp` integration tests; it has nothing to do with most pushes (e.g., docs-only pushes that don't touch gemini-mcp at all), yet it blocks them all the same.
- **Decision:** Modified `scripts/smoke-test.sh` to wrap each provider's smoke test in a `run_smoke` function that captures both the exit code and the test output, and applies a quota-detection escape hatch: if the test exits non-zero AND the captured output contains any known quota/rate-limit signal (`rateLimitExceeded`, `RESOURCE_EXHAUSTED`, `TerminalQuotaError`, `exhausted your capacity`, `code=429` — the same patterns the gemini-mcp executor already uses for its own quota fallback in ADR-044), the smoke test is treated as skip-with-warning rather than a hard failure. The pre-push hook continues for the remaining providers and the push proceeds. Real failures (non-quota errors) still abort the push as before. Added a `FORCE_SMOKE=1` env var that disables the escape entirely, so a developer who genuinely needs to verify quota-bearing tests can opt back into hard failure mode. Used POSIX-clean shell idioms (`set -o pipefail` + `|| rc=$?`) so the script works under bash 3.2 / `sh` POSIX mode on macOS without bash 4 features.
- **Consequences:** The rate-limit-self-defeating loop is broken — multi-push sessions no longer fail on the second or third push of the day. Pushes that don't touch `gemini-mcp` source code are no longer blocked by upstream Gemini quota issues. The trade-off is that a real Gemini regression occurring at the same moment as a quota outage would be silently downgraded to a warning instead of a hard failure — but this is an extremely narrow window, and `FORCE_SMOKE=1` provides the explicit escape when it matters. Other providers (Ollama, Codex) get the same treatment for symmetry; Ollama runs locally so quota errors there are essentially impossible, and Codex has its own quota fallback chain (gpt-5.4 → gpt-5.4-mini per ADR-028) that the smoke test exercises. Future maintainers will see the warning prominently in pre-push output when the escape fires, so the fact that "tests were skipped" is never silent. This is a pragmatic narrowing of ADR-043's hard-gate semantics — the gate is still there, just resilient to one specific class of upstream failure.

## ADR-050: Fix Sub-Agent Background Job Lifecycle Bug in brainstorm-coordinator
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** Issue #23. The `brainstorm-coordinator` agent dispatched external providers (Gemini, Codex, Ollama) via background bash jobs and then waited for Claude Code notifications to wake it up. This was broken: sub-agents cannot own background processes that outlive their turn. When the coordinator stopped issuing foreground tool calls, its turn ended, Claude Code tore down the sub-agent's shell context, and SIGKILLed every in-flight provider process. Codex at `reasoning=high` was especially vulnerable because it can take several minutes to produce a response — during that time the coordinator has no foreground work left, the turn ends, and Codex dies silently. The failure mode was invisible: 0-byte output files, no error, no exit code. User evidence trail from a real session: coordinator agent `ab5ea281ebabe67e0`, Codex output `/private/tmp/full-review-codex-response.md` (0 bytes), Codex stderr (86 lines, session header only, no model response). The user's sketched fix in the issue used `(cmd &) && wait` which contains a subtle subshell bug — parentheses spawn a subshell that detaches the child, so `wait` returns immediately and the "fix" produces the same orphan-and-SIGKILL behavior as the original. Discovered alongside issue #22 (Claude as first-class participant) — the two issues interact: #22's "Phase 3B runs in parallel with Phase 3A" framing is impossible to honor inside a sub-agent when #23 forbids background jobs.
- **Decision:** Restructured Phase 3 of `packages/claude-plugin/agents/brainstorm-coordinator.md` to run sequentially: Phase 3B (Claude Opus research, foreground) runs first, then Phase 3A (external provider dispatch) runs second as a SINGLE blocking foreground Bash tool call. Phase 3A uses direct backgrounding in the outer shell (`cmd > out 2>&1 & pid=$!`) — NOT subshells — and per-PID `wait` calls so every provider is a direct child of the outer bash and exit codes are captured independently. Required `timeout: 600000` (10 minutes, the Bash tool maximum) on the dispatch call — the default 2-minute timeout kills Codex at high reasoning effort mid-response, recreating the silent-failure class. Added per-provider stdout AND stderr capture so Phase 4 can distinguish "provider ran, returned nothing meaningful" from "provider was killed". Added a new "Critical: Sub-Agent Background Job Lifecycle" section in the agent prompt that documents the failure mode, the three anti-patterns to avoid (`run_in_background: true`, `(cmd &) && wait`, multi-call dispatch), and the two required patterns, with explicit reasoning. The sequential ordering also reinforces ADR-049's blindness constraint: Phase 3B completes before Phase 3A dispatches anything, so Claude cannot anchor on external responses because they do not exist yet. Rejected alternatives from the issue: (B) moving dispatch to the main conversation context — requires restructuring the skill-to-agent handoff and loses encapsulation; (C) using MCP tools directly instead of shelling out — the MCP tools are in the agent's frontmatter tools list but the user reported they did not surface in their session; shelling out is the reliably available path.
- **Consequences:** `/brainstorm` and `/brainstorm-all` no longer silently drop provider responses when the sub-agent turn ends. Codex at high reasoning effort now completes normally within the 10-minute cap. Per-provider exit codes and stderr are captured and surfaced in Phase 4, so failures are loud instead of silent. Total wall clock for a brainstorm run is now `time(Phase 3B) + time(slowest external provider)` — a modest increase over the previous (broken) "parallel" behavior, and bounded by the 10-minute Bash timeout cap. Providers exceeding the cap will still be killed, but cleanly with a foreground timeout error instead of a silent SIGKILL. Future maintainers have an explicit, in-prompt warning that documents the sub-agent background-job lifecycle gotcha — a Claude Code platform behavior that is not otherwise documented in the plugin codebase. Closes issue #23.

## ADR-049: Claude Opus as First-Class Research Participant in brainstorm-coordinator
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** Issue #22. The `brainstorm-coordinator` agent runs on Claude Opus but used Claude purely as a synthesizer — it dispatched the topic to external providers (Gemini, Codex, Ollama), waited for their responses, and aggregated them. Claude never did its own independent research even though it has filesystem access and can read the actual repository state. This was a capability leak: when external providers hallucinated or pattern-matched without verifying against real files, nothing caught them — Claude just relayed their output in the synthesis. Meanwhile, the one participant with the strongest grounding (Claude with repo access) contributed nothing independent. Documented in a real session where `/brainstorm-all` returned a single external response (Gemini) while Claude had direct access to the actual file being discussed and a real PR failure log; the user had to manually do the Claude research in the main conversation afterward and re-synthesize. Three subsidiary questions were resolved in the issue: (a) Phase 3B is always on, not opt-out — the whole point is that Opus capability is wasted if it's pure orchestration; (b) Claude is listed as a peer in the Participants Consulted table, not separated as an arbiter; (c) token cost is accepted as modest since Claude was already running as the coordinator — Phase 3B just asks it to think harder before synthesizing.
- **Decision:** Restructured `packages/claude-plugin/agents/brainstorm-coordinator.md` so Phase 3 has two parallel sub-phases: (3A) external provider dispatch — existing behavior; (3B) Claude Opus independent research — Claude reads the actual artifacts referenced by the topic (Read/Glob/Grep), traces real code paths, uses WebFetch/WebSearch on referenced external docs, forms independent findings, and records per-finding confidence as **Verified** (backed by actual file reads or fetched docs) or **Inferred** (reasoned from the topic description). Phase 4 synthesis now treats Claude as a peer alongside external providers, with verified findings weighted higher than inferred ones in consensus scoring. Blindness to external responses is enforced structurally: Phase 3B must complete before Phase 4 reads any external response, otherwise Claude anchors on what the externals said. Added `WebFetch` and `WebSearch` to the agent's tools frontmatter — the research phase needs web access for topics referencing external docs. Updated the output format to list Claude Opus in the Participants Consulted section with a `(verified against real files: path/to/x)` annotation. Applied matching updates to `skills/brainstorm/SKILL.md` and `skills/brainstorm-all/SKILL.md` descriptions and instructions so both the partial-provider and all-provider variants frame Claude as a participant. Bumped `@ask-llm/plugin` to 0.2.10.
- **Consequences:** `/brainstorm` and `/brainstorm-all` now return findings from up to 4 participants (Gemini + Codex + Ollama + Claude Opus) instead of up to 3. Verified findings from Claude catch regressions that external providers miss when the relevant state is in local files they can't see (framework-specific semantics, recently-changed code, repo conventions). Blindness ordering prevents Claude from collapsing into an "I confirm what Gemini said" echo. Token cost increases modestly — Opus does real work in Phase 3B instead of only synthesis. If a user later wants a pure-external-consensus mode, an `--external-only` flag could be added, but this is explicitly deferred until someone actually asks for it. Closes issue #22.

## ADR-048: Remove Stop Hook from Claude Code Plugin
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** The Claude Code plugin shipped a `Stop` hook (`scripts/stop-review.sh`) that ran `gemini -p` against `git diff HEAD` on every Claude Code `Stop` event. Two problems surfaced in practice. (1) **Wrong trigger semantic:** the `Stop` event fires every time Claude finishes responding to a user turn, not once at the end of a session. In an iterative conversation this meant 5–20 Gemini invocations reviewing overlapping diffs of the same work-in-progress, burning quota and blocking the UI for up to 60s per turn. (2) **Structural coverage gap:** `git diff HEAD` excludes untracked files, so any session that created new files (the majority of feature work) received no review at all — the hook silently claimed coverage it did not provide. Both issues were documented in `docs/BUGS.md` but could not be fixed independently: widening the diff capture made the latency bug worse, and there is no alternative trigger semantically equivalent to "end of session" exposed to plugin hooks today. The plugin already provides an explicit on-demand alternative via the `/gemini-review` slash command, and the `PreToolUse` pre-commit hook still catches the meaningful integration boundary (code about to be committed).
- **Decision:** Removed the Stop hook entirely. (1) Deleted `packages/claude-plugin/scripts/stop-review.sh`. (2) Removed the `Stop` block from `packages/claude-plugin/hooks/hooks.json`, keeping `PreToolUse` for the pre-commit review. (3) Updated `packages/claude-plugin/README.md`, root `README.md`, and `apps/docs/plugin/overview.md` to remove Stop hook references from feature tables and prose. (4) Marked the two Stop-hook entries in `docs/BUGS.md` as resolved via removal. (5) Annotated `docs/ROADMAP.md` to record the removal. (6) Bumped `@ask-llm/plugin` to 0.2.9 and updated the marketplace manifest. Rejected alternatives: (a) switching to a `SessionEnd` event — no such plugin-level event exists in the current Claude Code schema; (b) adding debounce/state-file gating — band-aid over the wrong-trigger problem; (c) widening `git diff HEAD` to include untracked files — compounds the per-turn latency bug.
- **Consequences:** Plugin users no longer get automatic per-turn advisory reviews. Users who want a session review now invoke `/gemini-review` explicitly — this restores user agency over quota spend and privacy (diffs no longer silently leave the machine on every turn). The pre-commit hook remains as the automated safety net at the real integration boundary. Existing installs pick up the removal on next plugin update; the change is backwards-compatible in the sense that nothing breaks — users simply stop seeing unsolicited reviews. If a future Claude Code version exposes a true `SessionEnd` event, the feature can be reintroduced cleanly on a correct trigger.

## ADR-047: Shell PATH Resolution for macOS GUI Apps
- **Date:** 2026-04-08
- **Status:** Accepted
- **Context:** macOS GUI applications (Claude Desktop, Cursor, etc.) do not inherit the user's shell environment — `.zshrc`/`.zprofile` are never sourced. This means nvm, fnm, Volta, and Homebrew paths are missing from PATH. When our MCP server spawns CLI tools like `gemini` or `codex`, those CLIs resolve to the wrong Node version (e.g., system v18 instead of user's v24), causing ES2024 incompatibility crashes. This affects all macOS users who install Node via version managers.
- **Decision:** Added `shellPath.ts` to `@ask-llm/shared` with a 3-tier PATH resolution strategy: (1) `ASK_LLM_PATH` env var override if set, (2) Extract real PATH from user's login shell via `execFileSync(shell, ["-ilc", "echo $PATH"])`, (3) Heuristic fallback scanning `~/.nvm/versions/node/`, `~/.volta/bin`, `/opt/homebrew/bin`, `/usr/local/bin` for Node v20+. The resolved PATH is cached once at startup and passed to all `spawn()` calls via `getSpawnEnv()`. Also updated `llm-mcp`'s `isCommandAvailable()` to use the resolved PATH.
- **Consequences:** CLI tools spawned by our MCP servers now use the user's real PATH regardless of how the parent process was launched. The login shell extraction adds ~100ms to first command execution (cached thereafter). Windows is unaffected (GUI apps inherit PATH correctly). Users with non-standard setups can override via `ASK_LLM_PATH` env var in MCP config.

## ADR-044: Fix Gemini Quota Fallback Detection for Newer CLI Versions
- **Date:** 2026-04-06
- **Status:** Accepted
- **Context:** Gemini CLI changed its quota error format in newer versions. The original executor only checked for `RESOURCE_EXHAUSTED` in error messages, but newer CLI versions return `TerminalQuotaError: You have exhausted your capacity on this model` — a completely different string. ## ADR-045: Lower Default Timeout to 210s for Claude Desktop Compatibility
- **Date:** 2026-04-06
- **Status:** Accepted
- **Context:** Claude Desktop has a hard 4-minute (240s) client-side timeout for MCP tool calls. The previous default server-side timeout was 5 minutes (300s). When a Codex CLI call took >4 minutes, Claude Desktop showed a generic "MCP server unresponsive" error before the server could return a meaningful timeout. The server also had a race condition: on timeout, it killed the process but didn't immediately reject the promise — waiting for the `close` event which could fire with confusing output. Reported in GitHub issue #20.
- **Decision:** (1) Lowered `DEFAULT_TIMEOUT_MS` from 300,000 to 210,000 (3.5 minutes) — 30s below Claude Desktop's limit. (2) Changed the timeout handler to immediately reject with a descriptive error message including the current timeout value and `GMCPT_TIMEOUT_MS` override hint. (3) Process cleanup (SIGTERM → SIGKILL) continues in the background after rejection.
- **Consequences:** Server-side timeout errors now reach the user before Claude Desktop's client timeout. Error messages are actionable ("try shorter prompt or increase GMCPT_TIMEOUT_MS"). Users needing longer can set the env var. All MCP clients benefit from faster, clearer timeout feedback.

## ADR-042: Fix Codex stdin Pipe Error in Agent Sub-Processes
- **Date:** 2026-04-03
- **Status:** Accepted
- **Context:** Issue #19 — Codex CLI fails with "stdin pipe error" when called from the brainstorm-coordinator agent, while working fine in direct tool calls. Root cause: `commandExecutor.ts` used `stdio: ["ignore", "pipe", "pipe"]`, setting stdin to `/dev/null`. Codex CLI probes stdin during initialization, and in agent sub-process context `/dev/null` causes a broken pipe. This is the same class of bug as the Gemini "exit code 42" issue (ADR-015), but Gemini's fix (use `-p` flag) doesn't apply to Codex.
- **Decision:** Changed stdin from `"ignore"` to `"pipe"` with immediate `.end()` call. Added a no-op error handler on stdin to absorb EPIPE if the child process exits before the close completes. This gives all child processes a proper EOF-terminated pipe instead of `/dev/null`.
- **Consequences:** Fixes Codex in brainstorm-coordinator. Applies to all providers (Gemini, Codex) but is safe — Gemini already uses `-p` for non-interactive mode, so a proper EOF pipe is equivalent. No behavior change for working providers.

## ADR-030: Multi-Approach Benchmark (MCP vs Skill vs Subagent vs Orchestrator)
- **Date:** 2026-03-20
- **Status:** In Progress (static analysis complete, manual runs pending)
- **Context:** The project offers four approaches to external LLM consultation: Standalone MCP, Orchestrator MCP, Skill, and Subagent. ADR-024 was a one-off experiment on a single file. - **Consequences:** Produces `docs/benchmarks/overhead.md` (static token costs), `docs/benchmarks/results.md` (latency + quality), and `docs/benchmarks/RECOMMENDATION.md` (user-facing decision tree). ## ADR-025: GitHub Actions Workflow Hardening for Fork PRs

(2) `claude.yml` detects fork PRs via GitHub API and gracefully skips with an explanatory comment instead of crashing. ## ADR-024: MCP vs Skill vs Subagent Context Comparison Experiment
- **Date:** 2026-03-01
- **Status:** Informational
- **Context:** To understand how different Claude Code invocation methods affect Gemini's code review quality, we ran the same code review task through three approaches: MCP tool call (structured template with explicit instructions), Skill (markdown skill file with checklist), and Subagent (general-purpose agent with natural language prompt). Each approach produced a review of the same `geminiExecutor.ts` file.
- **Finding:** Prompt structure inversely correlates with Gemini thinking tokens. The structured MCP template (explicit fields, numbered instructions) produced 6,666 thinking tokens in 74s. The Skill approach (markdown checklist) produced 8,942 thinking tokens in 118s. The raw Subagent prompt (unstructured natural language) produced 16,617 thinking tokens in 212s — 2.5x more thinking than MCP for comparable review quality.
- **Decision:** Document findings for future prompt engineering decisions. All three approaches found the same three actionable bugs (extractJson greedy first-match, extractJson escape-outside-string, missing thinking tokens in formatStats), suggesting review quality is consistent regardless of thinking token count.
- **Consequences:** When optimizing for speed/cost, prefer structured prompts with explicit output format instructions. When optimizing for thoroughness on novel problems, less-structured prompts may explore more reasoning paths. No code changes from this ADR.

## ADR-007: Utils Audit — Add Process Timeout and Windows Compatibility
- **Date:** 2026-02-23
- **Status:** Accepted
- **Context:** `executeCommand` had no timeout mechanism — a hung Gemini CLI process would leak indefinitely. On Windows, `spawn("gemini", ...)` fails with ENOENT because it resolves to `gemini.cmd`.
- **Decision:** Add configurable timeout: 5-minute default, overridable via `GMCPT_TIMEOUT_MS` env var. On timeout, send SIGTERM, then SIGKILL after 5s grace period. Add `shell: process.platform === "win32"` to spawn options. Replace O(n^2) `stdout += data` with `Buffer[]` array + `Buffer.concat()` at process close.
- **Consequences:** Prevents resource leaks from hung processes. Windows users can now use the tool. Large Gemini responses no longer cause quadratic memory allocation.

## ADR-015: Revert to `-p` Flag for Gemini CLI v0.29+
- **Date:** 2026-02-24
- **Status:** Accepted (supersedes ADR-006)
- **Context:** Gemini CLI v0.29.5 changed behavior again. The `--` separator (positional argument) now launches **interactive mode**, which expects stdin input. Since our MCP server spawns Gemini with `stdio: ["ignore", ...]`, stdin is closed, causing exit code 42: "No input provided via stdin." The `-p`/`--prompt` flag deprecation from v0.23 has been reversed — `-p` is now the correct way to trigger **non-interactive (headless) mode**.
- **Decision:** Revert `CLI.FLAGS.SEPARATOR = "--"` back to `CLI.FLAGS.PROMPT = "-p"` in `src/constants.ts`. Update both main and fallback code paths in `geminiExecutor.ts` to use `CLI.FLAGS.PROMPT`. This undoes ADR-006's change while keeping all other improvements from that audit (removed broken `@` quoting, etc.).
- **Consequences:** Fixes exit code 42 errors for users on Gemini CLI v0.29+. Users on older CLI versions (v0.23–v0.28) that still reject `-p` may need to upgrade their Gemini CLI. The `-p` flag is the officially supported non-interactive invocation method going forward.

## ADR-014: Add Vitest Test Suite
- **Date:** 2026-02-23
- **Status:** Accepted
- **Context:** The project had zero test infrastructure — no test runner, no test files, and a placeholder `npm test` script. The codebase has several pure utility modules (changeModeParser, changeModeChunker, changeModeTranslator) ideal for unit testing without mocking. The project is native ESM (`"type": "module"`, `"module": "Node16"`), which makes Jest painful (requires `--experimental-vm-modules` workarounds).
- **Decision:** Use Vitest as the test runner — it handles ESM natively and `tsx` is already a devDependency. Start with pure-function unit tests (no mocking) for the three changeMode utilities, registry tests using lightweight stub tools, and a smoke test verifying tool registration. Defer mocking-heavy tests (commandExecutor, chunkCache, geminiExecutor) and E2E tests to future iterations.
- **Consequences:** 42 tests across 5 test files provide baseline coverage for the most logic-dense modules. `npm test` now runs `vitest run` instead of a no-op. Test infrastructure is in place for incremental coverage expansion.

## ADR-018: Claude Code Plugin — Hybrid Multi-Entrypoint Architecture
- **Date:** 2026-02-25
- **Status:** Accepted (implemented in ADR-027)
- **Context:** We want a Claude Code plugin for automated Gemini code reviews (pre-commit hook, Stop hook, on-demand skill, isolated subagent). The core question was whether to reuse code from the existing MCP server. Hooks run as shell commands and have no MCP client, so they can't call MCP tools directly.
- **Decision:** Hybrid multi-entrypoint approach. Add a second CLI binary `ask-gemini-run` (`src/run.ts`) that calls `executeGeminiCLI()` directly and prints to stdout. The plugin uses MCP tools for AI-context interactions (subagent, skill) and the direct CLI for shell-context interactions (hooks). Both entry points share the same core logic (`geminiExecutor.ts`). No monorepo needed — just two `bin` entries in `package.json`. Rejected: MCP-only (hooks can't use MCP), Bash-only (duplicates hardened logic), monorepo (overkill).
- **Consequences:** Zero logic duplication. Hooks get the same model fallback, quota handling, and chunking as the MCP server. Plugin is mostly markdown/config (~3 files). The `ask-gemini-run` binary also serves as a standalone CLI for users who want Gemini access without MCP.
- **Design doc:** [docs/plans/2026-02-25-claude-code-plugin-design.md](plans/2026-02-25-claude-code-plugin-design.md)

## ADR-019: Structured JSON Output via --output-format json
- **Date:** 2026-02-25
- **Status:** Accepted
- **Context:** The Gemini CLI supports `--output-format json` which returns `{ response, stats, error }` instead of raw text. Our MCP server was parsing raw text output, losing token usage stats and getting unstructured error messages. Since we always invoke the CLI programmatically, structured JSON is strictly better.
- **Decision:** Always pass `--output-format json` to the Gemini CLI. Parse JSON in `geminiExecutor.ts`, extract the `response` text, and append a one-line stats summary (input/output tokens, model). Fall back to raw text if JSON parsing fails (backward compat with older CLI versions). No schema changes — the improvement is transparent to MCP clients.
- **Consequences:** Users get token usage visibility. Errors are structured and more informative. Older Gemini CLI versions that don't support `--output-format` degrade gracefully to the current behavior.
- **Follow-up (2026-02-25):** Added `extractJson()` to handle CLI warning/debug lines before JSON object, and improved error handling for error responses that have only a code (no message).
- **Design doc:** [docs/plans/2026-02-25-structured-json-output-design.md](plans/2026-02-25-structured-json-output-design.md)

## ADR-017: Smithery CJS Compatibility — createSandboxServer Export
- **Date:** 2026-02-24
- **Status:** Accepted
- **Context:** Smithery bundles MCP servers with esbuild in CJS mode for capability scanning. Our ESM code uses `createRequire(import.meta.url)` to read `package.json` at runtime, but in CJS bundles `import.meta` is empty (undefined), causing the module to crash on load. This prevented Smithery from scanning tool/prompt schemas.
- **Decision:** (1) Wrap `createRequire(import.meta.url)` in a try/catch with hardcoded fallback values (`name: "ask-gemini-mcp"`, `version: "0.0.0"`). (2) Export a `createSandboxServer()` function that returns a configured `McpServer` with tools/prompts registered via stub handlers — Smithery's recommended pattern for servers that need special handling during scanning. (3) Separate CLI entry point (`src/cli.ts`) from library module (`src/index.ts`) — `index.ts` no longer auto-executes `main()`, preventing stdio conflicts when Smithery imports the module. `package.json` `bin` now points to `dist/cli.js`.
- **Consequences:** The module loads cleanly in both ESM (normal operation) and CJS (Smithery bundling) contexts. Smithery can scan tool schemas without invoking real Gemini CLI or triggering stdio transport. The fallback version "0.0.0" is only used in CJS bundles, not in production stdio operation.
