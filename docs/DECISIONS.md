## ADR-072: Plumb Codex `reasoning_output_tokens` â†’ Shared `thinkingTokens`
- **Date:** 2026-05-04
- **Status:** Accepted (drains P1 Â§2 from issue #28; addresses #27 B1)
- **Context:** Codex CLI `rust-v0.125.0` (released 2026-04-24, stable since #27 was filed) added a `reasoning_output_tokens` field to the `turn.completed.usage` event for reasoning models (the `gpt-5.5` family). Our `CodexTurnCompleted.usage` interface in `packages/codex-mcp/src/utils/codexExecutor.ts:12-19` did not declare the new field, so it was silently dropped during JSONL parsing. Downstream impact: `buildUsageStats` hard-coded `thinkingTokens: undefined` for codex calls; `[Codex stats: ...]` footer never showed reasoning tokens; `get-usage-stats` MCP tool's cross-provider aggregation reported `thinkingTokens: 0` from codex regardless of how much reasoning the model actually did. For users running `/multi-review` against `gpt-5.5` with deep reasoning, this hid a meaningful slice of their token spend. Issue #28 Â§3 P1 #2 flagged this as a "small, isolated correctness win" â€” ~15 LoC, fully backward-compatible because the field is optional in our types and older codex versions simply don't emit it. The shared `UsageStats` type in `@ask-llm/shared` already has a `thinkingTokens` field (added for Gemini's `stats.models[*].tokens.thoughts`), so the right fix is to map codex's `reasoning_output_tokens` to the same shared field rather than introducing a codex-specific reasoning-tokens field that would drift the cross-provider aggregation surface.
- **Decision:** Three small edits in `packages/codex-mcp/src/utils/codexExecutor.ts`: (1) Added `reasoning_output_tokens?: number` to the `CodexTurnCompleted.usage` interface declaration with a comment explaining the codex 0.125+ provenance and the optional/backward-compatible nature. (2) Updated `buildUsageStats` to map `usage?.reasoning_output_tokens` â†’ `thinkingTokens` (replacing the hard-coded `undefined`). The shared `UsageStats.thinkingTokens` field is the cross-provider aggregation surface â€” Gemini's `thoughts` and Codex's `reasoning_output_tokens` both flow into the same field, so `get-usage-stats` and `formatSessionUsage` aggregate correctly across mixed-provider sessions without per-provider unwrapping. (3) Updated `formatStats` to surface a `"7,500 thinking tokens"` line in the `[Codex stats: ...]` footer when `reasoning_output_tokens > 0`. Footer ordering: `input â†’ output â†’ thinking â†’ cached`, mirroring `geminiExecutor.ts:formatStats` so cross-provider stats blocks line up visually in `/multi-review` reports. The `> 0` guard matches the cached-tokens precedent: zero reasoning tokens (non-reasoning models like older gpt-4 calls) shouldn't surface a noisy `"0 thinking tokens"` line. 3 new tests in `packages/codex-mcp/src/utils/__tests__/codexExecutor.test.ts`: present-and-positive (footer shows the value, `UsageStats.thinkingTokens` carries it for telemetry), present-and-zero (footer suppresses the line but `UsageStats.thinkingTokens === 0` is preserved â€” the explicit zero is meaningful telemetry, not the same as field-missing), absent (older codex versions where the field never appears â€” `UsageStats.thinkingTokens` is `undefined`, footer omits the line). The triple covers the upstream version-compatibility matrix without needing real codex versions in tests. **Out of scope:** (a) backfilling reasoning-token telemetry for codex calls *prior to v0.125* â€” not possible, the upstream JSONL didn't carry the field. (b) Surfacing reasoning tokens via a *separate* `codexReasoningTokens` field instead of reusing `thinkingTokens` â€” explicitly rejected because cross-provider aggregation would then need per-provider unwrapping logic in `formatSessionUsage`. (c) Distinguishing "model emitted 0 reasoning tokens" from "model didn't emit the field" in the footer â€” both correctly suppress the line; the distinction lives in `UsageStats.thinkingTokens` (`0` vs `undefined`) where telemetry consumers can read it.
- **Consequences:** `/multi-review` and `/brainstorm` reports against `gpt-5.5` now correctly show how many tokens the model spent on reasoning, alongside input + output + cached counts. Cross-provider sessions (e.g. `/compare gemini vs codex`) get apples-to-apples reasoning-token comparison via the shared `UsageStats.thinkingTokens` field. **No new tool surface, no schema changes, no new env vars** â€” purely a parser-side feature adoption. Backward compatibility is preserved by construction: older codex versions don't emit `reasoning_output_tokens`, so the optional-undefined path in TypeScript types and runtime checks handles them identically to today's behavior. Argv unchanged. Test count: codex-mcp 36 â†’ 39 (+3 reasoning tests). **Drains P1 #2 from #28's punch list**, leaving P2 (#5 typed exit codes, #7 `-m` precedence smoke test, #8 `--add-dir` parity) and P3 items as remaining work. The shared `UsageStats.thinkingTokens` field is now used by Gemini, Codex, and any future provider with a reasoning-token concept â€” the precedent is "map provider-native reasoning-token field name â†’ shared `thinkingTokens`" rather than adding per-provider fields.

## ADR-071: Codex `--ignore-user-config` + `--ignore-rules` for Deterministic MCP Wrapping
- **Date:** 2026-04-28
- **Status:** Accepted (closes #31, addresses action #5 from #26 / #24 Â§2.2 / #25 Â§3.3)
- **Context:** Codex CLI `rust-v0.124.0` (released 2026-04-23) graduated hooks (`PreToolUse`, `PostToolUse`, `PermissionRequest`, `UserPromptSubmit`, `Stop`) to **stable** status. The release notes also flagged that *"project hooks and exec policies now require trusted workspaces"* â€” parallel to the Gemini workspace-trust change resolved in PR #29 / ADR-069. Issue #31 was filed as a follow-up to #26 Â§4.2 to **smoke-test whether user-installed `~/.codex/config.toml` hooks break our `--ephemeral` exec path**. The risk: if a user has `[hooks.PreToolUse]` configured, our spawned `codex exec` might try to run that hook against the ephemeral sandbox, fail, and bubble the failure as an error event in the JSONL stream rather than silently skipping. Two paths forward: (A) write a smoke test that creates a real `~/.codex/config.toml` with a no-op hook, runs `executeCodexCLI`, and asserts no failure â€” high-fidelity but binary-version-dependent and fragile (real codex required, may break differently per OS, may interfere with the user's actual config); (B) **preempt the entire risk class** by passing `--ignore-user-config` (skips `~/.codex/config.toml` entirely; auth credentials in `CODEX_HOME` are still loaded) and `--ignore-rules` (skips user/project `execpolicy .rules` files). Option B was independently flagged in #26 Â§4.6 as **action #5** â€” *"`--ignore-user-config` + `--ignore-rules` on the ephemeral path â€” still the highest-leverage change per #24 Â§2.2 and #25 Â§3.3."* The two flags together remove the entire surface of "user-side config affecting our deterministic CLI wrapper" â€” for an MCP server that programmatically calls codex, the user's interactive-mode preferences should never affect what the API caller asked for. Verified locally that Codex CLI 0.125.0 has both flags exposed via `codex exec --help`. **Caveat surfaced during PR review:** `~/.codex/config.toml` doesn't only hold hooks and preferences â€” it's also where users register `[[mcp_servers]]` entries that codex can invoke as agent tools during execution. `--ignore-user-config` skips those too. A user who has registered, say, a database-query MCP server in their codex config and expects it to be available inside `ask-codex-mcp` invocations would see degraded behavior (codex would not be able to call that DB tool while answering a prompt routed through this wrapper). To mitigate without giving up the deterministic default for the 99% case, an opt-out env var was added.
- **Decision:** (1) `packages/codex-mcp/src/constants.ts` `CLI.FLAGS` gains `IGNORE_USER_CONFIG: "--ignore-user-config"` and `IGNORE_RULES: "--ignore-rules"`. (2) `packages/codex-mcp/src/utils/codexExecutor.ts:buildArgs` emits both flags **by default**, between the existing process-level flags (`--skip-git-repo-check`, optional `--ephemeral`) and the model/output flags (`--full-auto`, `--json`, `-m`). The argv ordering follows the existing convention: lock down the environment first, then pick the model. A 6-line WHY-not-WHAT comment block above the conditional explains the rationale and points at the env-var opt-out. (3) **Opt-out via `ASK_CODEX_LOAD_USER_CONFIG=1`** â€” when set to literal `"1"`, both flags are omitted from argv and codex loads the user's `~/.codex/config.toml` (including any `[[mcp_servers]]` entries) and `.rules` files normally. The strict `=== "1"` equality check (matching the `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST === "1"` pattern from ADR-069) means any other value (`"true"`, `"yes"`, missing) keeps the deterministic default â€” explicit opt-in only, no truthy-coercion footgun. (4) Both flags apply on **every** default path â€” primary call, quota-fallback call, and session-resume call. The opt-out applies on every path too; there's no half-state. (5) Updated the existing `toEqual([...])` argv-shape test to include both new flags in the expected order, added a companion `toEqual([...])` for the resume-path argv shape, and added a new `executeCodexCLI ASK_CODEX_LOAD_USER_CONFIG opt-out` describe block with proper env save/restore in `beforeEach`/`afterEach` covering: default behavior emits both flags, `=1` omits both flags (with full `toEqual` to prove the rest of argv stays identical), opt-out applies on resume path too, and `"true"` (non-`"1"`) does NOT trigger opt-out (regression guard against future truthy-coercion refactors). (6) **Out of scope:** writing the original "smoke-test hooks" in #31 â€” preempted by this fix. Per-call (rather than per-process) user-config opt-in via an MCP tool argument is also out of scope: env-var-level granularity is sufficient for the use case (user either has custom MCP servers and wants them, or doesn't); per-call would require threading the flag through the MCP tool schema and isn't justified by current demand.
- **Consequences:** `executeCodexCLI` now produces deterministic behavior **by default** regardless of the host machine's `~/.codex/config.toml` or `.rules` files. A user with elaborate hooks in their interactive codex setup gets the same MCP behavior as a user with no config at all. Users with custom MCP servers registered in `~/.codex/config.toml` who want those available inside `ask-codex-mcp` invocations can opt back in with `ASK_CODEX_LOAD_USER_CONFIG=1` â€” same shape as the Gemini opt-out from ADR-069. The `auth still uses CODEX_HOME` clause in upstream's `--ignore-user-config` documentation means we don't break authenticated calls in either mode â€” only ignore preference/hook/MCP-server TOML when in the default path. Argv length per call grows by 2 strings (~36 bytes) in the default path; 0 strings in the opt-out path. **Pre-empts a class of future upstream churn**: when codex 0.126+ adds new hook lifecycle events or changes how `.rules` files are evaluated, this MCP wrapper's behavior is insulated by construction. **Closes #31** with the explanation "preempted by hardening" (smoke test was the means, deterministic flags are the end). Future maintenance: when a new codex release adds yet another flag for "ignore X user config," check whether MCP determinism wants the same treatment â€” the precedent set by this ADR is "yes, by default, with a comment explaining why, and an env-var opt-out for users who want the looser behavior."

## ADR-070: Large-Prompt Stdin Path â€” 16 KiB Threshold Across Gemini & Codex Executors
- **Date:** 2026-04-28
- **Status:** Accepted (resolves issue #30, deferred from #26 Â§4.3)
- **Context:** All three CLI-based executors (`packages/gemini-mcp/src/utils/geminiExecutor.ts`, `packages/codex-mcp/src/utils/codexExecutor.ts`, plus `ollama-mcp` which uses HTTP and is not affected) appended the full user prompt as a positional argv element. For typical short prompts that's fine, but `@largefile` expansion in Gemini, `/multi-review` and `/brainstorm` skill dispatches with sizable diffs, and any session-resumed conversation accumulating context could push argv close to the OS `ARG_MAX` ceiling â€” 256 KiB on macOS, 2 MiB on Linux. Codex `rust-v0.124.0` (released 2026-04-23) explicitly added prompt-plus-stdin support: *"If stdin is piped and a prompt is also provided, stdin is appended as a `<stdin>` block."* Gemini CLI's `-p` flag has *also* documented stdin coexistence for some time: *"Appended to input on stdin (if any)"* â€” meaning we can pipe the bulk via stdin and pass an empty `-p ""` placeholder (still required for headless mode). Three implementation shapes were considered: (A) extend `executeCommand` with a 5th optional positional `stdin?: string` arg â€” minimal non-breaking change; (B) refactor `executeCommand` to an options-object signature for cleanliness â€” affects all four call sites and breaks ABI; (C) introduce a new helper `executeCommandWithStdin()` â€” duplicates timeout/error/spawn logic. Option A was selected per the project's CLAUDE.md "no premature abstraction" rule.
- **Decision:** (1) `packages/shared/src/constants.ts` gains `EXECUTION.STDIN_THRESHOLD_BYTES = 16_384` (16 KiB) â€” well below macOS's 256 KiB `ARG_MAX` so non-piped calls never hit the limit either. The threshold is a build-time constant, not env-configurable; if a future use case justifies tuning it per-deployment, that becomes its own scoped change. (2) `packages/shared/src/commandExecutor.ts:executeCommand` now accepts an optional 5th positional `stdin?: string`. Implementation: existing `childProcess.stdin.on("error", ...)` + `.end()` lifecycle is preserved (issue #19's pipe-error fix); when `stdin !== undefined && stdin.length > 0`, `childProcess.stdin.write(stdin)` runs immediately before `.end()`. The empty-string case is treated as a no-op write but still uses the explicit signal â€” the `length > 0` guard avoids the unnecessary write call. (3) Per-executor wiring: each executor computes `useStdin = prompt.length > EXECUTION.STDIN_THRESHOLD_BYTES` once at the top of the executor function, then both primary and quota-fallback paths share the same `stdinPayload` and threshold decision â€” DRY by construction, no possibility of one path forgetting the flag. (4) Mechanism per-CLI: **Gemini** â€” `buildArgs` emits `-p ""` (empty placeholder) when `useStdin`, since headless mode still requires the `-p` flag; the actual content goes via stdin. Per Gemini's CLI help, stdin content concatenates with `-p` text. **Codex** â€” `buildArgs` skips appending the prompt as positional argv when `useStdin`; per Codex's CLI help, *"`[PROMPT]` is optional ... when absent, instructions are read from stdin."* (5) **Ollama deliberately untouched** â€” uses HTTP `POST /api/chat` not argv, so has no `ARG_MAX` ceiling. (6) 11 new tests across three packages: 4 in `packages/shared/src/__tests__/commandExecutor.test.ts` (real-spawn round-trip via `node -e "process.stdin.pipe(process.stdout)"` for stdin payload write/EOF, large-payload >16 KiB success, undefined-preserved-existing-behavior, empty-string-no-op); 4 in `packages/gemini-mcp/.../geminiExecutor.test.ts` (15 KiB â†’ argv, 17 KiB â†’ stdin with empty `-p`, fallback path preserves stdin, changeMode-wrapped small prompt stays argv); 3 in `packages/codex-mcp/.../codexExecutor.test.ts` (15 KiB â†’ argv, 17 KiB â†’ stdin no-positional, mini fallback preserves stdin). All existing argv-shape `toEqual([...])` tests stay untouched â€” the threshold flip is purely transport, never affects argv shape for sub-threshold prompts. (7) **One existing codex test had to be updated** â€” `passes onProgress callback to executeCommand` previously asserted `toHaveBeenCalledWith(cmd, args, onProgress)` (3 positional args). After this change, all `executeCommand` calls pass 5 positional args (with `undefined` for unused slots), so the test now uses `toHaveBeenCalledWith(cmd, args, onProgress, undefined, undefined)` â€” explicit, defensive against future signature drift.
- **Consequences:** `/multi-review` and `/brainstorm` with large diffs, `@largefile` expansion against multi-megabyte sources, and any future workflow shoving a big prompt at the executor will no longer collide with the OS `ARG_MAX` ceiling. Bonus privacy improvement: full prompts are no longer visible in `ps` / `ps aux` output for piped calls (only the short argv stub shows). Backward compatibility is preserved â€” every short-prompt call (<16 KiB) takes exactly the same code path it did before; no spawn semantics change for existing argv-only callers. The signature change to `executeCommand` is non-breaking because the new 5th param is optional. Test count: **+11 tests** (424 â†’ 435 across the monorepo). Lint clean across all 6 packages, build clean. **Operational note caught during implementation:** the prepack-bundle.mjs script copies `packages/shared/dist` into `packages/<dep>/node_modules/@ask-llm/shared/` at publish time, and `postpack-restore.mjs` restores ONLY `package.json` â€” not the bundled `node_modules`. After a manual `npm publish` (or a publish that fails partway, like the EOTP loop on PR #29), the stale bundled copy lingers and *outranks* the workspace symlink in node module resolution, which silently broke this PR's TS build until I cleaned `packages/<pkg>/node_modules/`. The release pipeline (`release.yml`) runs in fresh CI runners so the cruft can't accumulate there â€” yet another argument for the tag-driven release flow over manual publish (memory note saved separately). Future maintenance: when the next executor ships (or the codex-mcp/llm-mcp wiring needs touching), this same `useStdin = prompt.length > THRESHOLD` shape applies; copy the existing pattern. Threshold tuning, if ever needed, is a one-line constant change. **Out of scope (intentionally):** (a) Making the 16 KiB threshold env-configurable â€” the constant is sufficient; tuning needs no caller-time control. (b) Wrapping Gemini-only via `@/tmp/prompt.txt` temp-file syntax instead of stdin â€” both Gemini and Codex now use stdin uniformly, which keeps the executor abstraction coherent. (c) Streaming the stdin payload in chunks for very-large (multi-MB) prompts â€” current `childProcess.stdin.write(string)` is fine up to Node's default highWaterMark; if someone hits a 100 MB prompt, that's a different conversation. (d) Updating `commandExecutor` to options-object form â€” explicit non-goal per CLAUDE.md.

## ADR-069: Gemini Workspace-Trust Compatibility â€” Default-On Env Var, Friendly Error, No Flash Retry
- **Date:** 2026-04-27
- **Status:** Accepted (resolves issue #26)
- **Context:** gemini-cli v0.39.1 (released 2026-04-24, upstream PR [#25814](https://github.com/google-gemini/gemini-cli/pull/25814)) shipped a `FatalUntrustedWorkspaceError` gate that fires before any model call when `gemini -p` is invoked against a directory that was never marked trusted in interactive mode. Fresh installs of `ask-gemini-mcp` on v0.39.1+ would fail with a raw stderr dump and no actionable guidance â€” the project's existing stderr handler in `geminiExecutor.ts:379` only specialized RESOURCE_EXHAUSTED, so trust errors fell through to a generic spawn failure surface. Worse, the catch block at `geminiExecutor.ts:491` would fire the quota-fallback path on any non-quota error mismatch, meaning a trust error on Pro would have triggered a Flash retry against the *same untrusted directory* â€” guaranteed to fail identically, just slower. Three fix approaches were considered: (A) add `--skip-trust` to argv; (B) set `GEMINI_TRUST_WORKSPACE=true` as a parent-process env var that propagates via `getSpawnEnv()`'s `{...process.env, ...}` spread; (C) extend `executeCommand` to accept per-call `extraEnv`. Option A is backward-incompatible â€” the local install at the time of writing was v0.38.2, which has no `--skip-trust` flag, so a fresh argv flag would error on older Geminis with "unknown option." Option B is the upstream-blessed compatibility surface (the gemini-cli's own CI uses `GEMINI_TRUST_WORKSPACE=true` in E2E workflows) and unknown env vars are silently ignored on older versions, making it forward-and-backward safe. Option C would be the cleanest abstraction but adds a parameter to a shared API for one caller's needs â€” explicit "no premature abstraction" violation per the project's CLAUDE.md. Option B was selected.
- **Decision:** (1) `packages/gemini-mcp/src/constants.ts` gains a `WORKSPACE_TRUST_PATTERNS` array containing both the typed error class name (`FatalUntrustedWorkspaceError`, stable identifier in stack traces) and the human-readable string (`not running in a trusted directory`, surfaces in stderr text) â€” matching either makes detection robust against upstream changing how the error surfaces in either channel. New `ERROR_MESSAGES.WORKSPACE_TRUST_REQUIRED` string lists three remediation paths in priority order: unset `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST` (let us re-apply the safe default), set `GEMINI_TRUST_WORKSPACE=true` yourself, or run `gemini` interactively in this directory once and mark it trusted. New `STATUS_MESSAGES.WORKSPACE_TRUST_DETECTED` for the stderr-handler log line. (2) `packages/gemini-mcp/src/utils/geminiExecutor.ts` gains a private `ensureWorkspaceTrustEnv()` helper called at the top of `executeGeminiCLI`. The helper has three idempotency guards: `process.env.ASK_GEMINI_REQUIRE_WORKSPACE_TRUST === "1"` (user opted out â€” do nothing), `process.env.GEMINI_TRUST_WORKSPACE !== undefined` (user pre-set their own value â€” don't override), otherwise sets `process.env.GEMINI_TRUST_WORKSPACE = "true"`. The mutation propagates to all subsequent spawned children for free because `shellPath.ts:getSpawnEnv()` already does `{...process.env, PATH: resolveShellPath()}` â€” no `commandExecutor.ts` changes required. (3) `createGeminiStderrHandler` now tracks both `quotaLogged` and `trustLogged` flags so the handler logs each error class once per invocation. (4) The catch block inserts a workspace-trust check **before** the quota check via a new `isWorkspaceTrustError(message)` helper â€” this is critical, because without short-circuiting, a trust error on Pro would either silently fall through to the generic re-throw OR (worse) hit the Flash retry path which would just fail again on the same untrusted directory. When detected, the catch throws a fresh `Error(ERROR_MESSAGES.WORKSPACE_TRUST_REQUIRED)` instead of bubbling the raw upstream error. (5) 6 new tests in `geminiExecutor.test.ts` under a new `executeGeminiCLI workspace trust handling` describe block, with proper `process.env` save-and-restore in `beforeEach`/`afterEach` to prevent cross-test env leakage: default behavior sets the env var, `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST=1` skips it, pre-existing user value isn't overridden, both error patterns (`FatalUntrustedWorkspaceError` and `not running in a trusted directory`) throw the friendly message and short-circuit Flash retry, and the orthogonality test that quota errors still trigger Flash fallback unchanged. **Explicitly out of scope:** Action items #2 (stdin piping for >16KiB prompts), #3 (Codex stabilized hooks smoke test), #4 (carried sessionId round-trip + untrusted-workspace integration test), #5â€“7 (carried-from-#24/#25 items) â€” each warrants its own focused PR rather than bundling. Issue #26 listed all 7 as candidates with #1 explicitly tagged "Hotfix required? Arguably yes for action #1" â€” this ADR ships only #1.
- **Consequences:** Fresh installs of `ask-gemini-mcp` on gemini-cli v0.39.1+ now work without manual workspace-trust setup, restoring parity with the â‰¤ v0.39.0 behavior the MCP server was written against. The env-var approach is forward-and-backward compatible: older Geminis (v0.38.x and earlier, which have no workspace-trust feature) silently ignore the unknown env var; newer Geminis (v0.39.1+) honor it. Power users who *want* workspace-trust enforcement (e.g., shared CI runners where directory provenance matters) can opt in by setting `ASK_GEMINI_REQUIRE_WORKSPACE_TRUST=1` in their MCP config â€” when they then hit an untrusted directory, they get the friendly remediation message instead of a raw upstream error. Zero argv shape change, so all 25+ argv-based `toEqual([...])` tests in `geminiExecutor.test.ts` stay green untouched â€” the brittleness of those exact-array assertions now only fires on real argv changes, not env-side tweaks. The Flash retry cascade is preserved for genuine quota errors (orthogonality test confirms) but correctly skipped for trust errors. Test count: 88 in gemini-mcp (was 82, +6 trust tests); 422 across all packages. Lint clean across all 6 packages, full build clean. The friendly error message is markdown-readable so MCP clients render it nicely; the three numbered remediation steps mean a user can copy-paste the right command without reading the upstream PR. Future work: actions #2â€“7 from issue #26 each get their own PR; the integration-test scaffold for an actual untrusted-workspace round-trip (action #4) is the natural next step but requires either a git-isolated tempdir fixture or a vi-mocked `which` to skip the real `gemini` binary path. Deferred.

## ADR-068: `/codex-image` Plugin Skill â€” Skill-Only Architecture for Codex Image Generation
- **Date:** 2026-04-24
- **Status:** Accepted
- **Context:** codex-cli 0.125.0 stabilized the `image_generation` feature flag (verified via `codex features list` â€” stage `stable`, default `true`), making OpenAI's gpt-image-1 model accessible to any `codex exec` session as a model-invoked tool. There is no dedicated `codex image generate` subcommand; image generation is exposed as an internal tool the model decides to invoke when the prompt asks for an image, similar to how it decides to invoke `read_file` or `apply_patch`. Three architectures were considered for surfacing this in the project: (A) skill-only, where a new `/codex-image` plugin skill instructs Claude Code to call the existing `ask-codex` MCP tool with a prompt-engineered template; (B) dedicated MCP tool plus skill, adding `ask-codex-image.tool.ts` with explicit input schema (`prompt`, `outputPath`, `size`, `quality`) and structured output (`{filePath, model, usage}`); (C) hybrid, ship the skill now and promote to a dedicated tool later if non-Claude-Code MCP clients (Cursor, Warp, Copilot) ask for image generation as a first-class primitive. Option B costs ~150 LOC plus tests plus an output-format coupling that may break when Codex changes its image-tool reply format; Option A reuses every existing safety property of `ask-codex` (ADR-044 quota fallback, ADR-042 stdin handling, ADR-047 PATH resolution, ADR-067 model defaults) at zero new code in MCP packages.
- **Decision:** Ship Option A. New skill at `packages/claude-plugin/skills/codex-image/SKILL.md` with `user_invocable: true` and a 5-phase workflow: (1) refine the user's image prompt while preserving intent; only enrich if the prompt is sparse and the conversation provides design context; default-exclude "no humanoid figures, no glowing brains, no chatbot iconography" for AI-topic prompts because those are dominant diffusion-model failure modes. (2) Determine output path with the convention `/tmp/codex-images/$(date +%Y-%m-%d)/<slug>.png` (kebab-case slug derived from the prompt, max 40 chars) â€” this matches Codex's `--ephemeral` philosophy (ADR-028), avoids polluting user repos, and the date prefix instead of a session-id makes images findable later; user-supplied `outputPath` overrides the default. (3) Dispatch via the existing `ask-codex` MCP tool (NOT raw `codex exec` â€” explicitly forbidden in the skill instructions, mirroring `/multi-review`'s rule from ADR-064) with a prompt template that names the `image_generation` tool, embeds the absolute output path, and asks the agent to confirm the file size + path in its reply. (4) Verify with `ls -la <path>` after the MCP call returns â€” never trust the agent's textual claim alone, same verify-before-presenting principle as ADR-064 Phase 3. (5) Specifically catch four failure modes: feature flag disabled, codex-cli version too old, content policy rejection, disk-full/permission-denied â€” surface each verbatim rather than paraphrasing. **Explicitly out of scope:** (a) a dedicated MCP tool â€” promotion to Option B is deferred until non-CC client demand justifies the surface cost; (b) per-call model/size/quality flags as MCP arguments â€” instead, users include those in the prompt body and the model handles them; (c) image input (vision) wrapping â€” `codex exec -i <FILE>` is the orthogonal direction and gets its own future ADR if surfaced; (d) multi-image batches â€” single image per call keeps the skill's verify step simple; users can re-invoke for variations. No tests added â€” skills are markdown prose and don't ship into compiled packages, so the existing test suite stays at 414 tests; functional verification happens via live invocation in Claude Code.

## ADR-066: `multi-llm` MCP Tool â€” Parallel Provider Dispatch as One Synchronous Primitive
- **Date:** 2026-04-17
- **Status:** Accepted
- **Context:** The brainstorm-coordinator agent and the `/multi-review`, `/brainstorm`, `/compare` skills all converge on the same atomic operation: "send the same prompt to N providers in parallel, collect all responses." Inside Claude Code, this is achievable via the Agent tool dispatching subagents, or via a Bash tool running parallel CLI invocations. Outside Claude Code (Cursor, Warp, Copilot, any other MCP client), there is no equivalent â€” those clients have MCP tool calls but no built-in parallel-dispatch primitive. The `ask-llm` orchestrator currently exposes only `ask-llm` (single-provider call), `ping`, `get-usage-stats`, and `diagnose` â€” none of these compose into "fan out a prompt to all available providers in one call." A non-Claude-Code user wanting multi-provider answers has to issue 2-3 sequential `ask-llm` tool calls and synthesize themselves. The user's same-day strategic pivot to "focus on MCP tools and their usage in Claude Code CLI" plus the candidate triage of #4 ("Subagent orchestration as MCP tools") raised the question of which flavor to ship: generic async `spawn-subagent`/`wait-for-subagent` primitives (Option A) vs a focused synchronous parallel-dispatch tool (Option B). Option A's hidden complexity (state management between tool calls, in-flight process lifecycle, cancellation semantics, persistence across server restarts) is significant and is partially duplicative of Claude Code's existing Agent + Bash tools. The user picked Option B â€” focused over generic.
- **Decision:** New `packages/llm-mcp/src/multiLlm.ts` module exposes three pure functions: `dispatchMultiLlm(opts)`, `formatMultiLlmReport(report)`, and `buildMultiLlmInputSchema(availableProviders)`. The dispatch function takes `{ prompt, providers, getExecutor, recordUsage? }` and returns `Promise<MultiLlmReport>` where each provider runs in parallel via `Promise.all` and per-provider failures are captured in the result rather than thrown. `Promise.all` is the right primitive here (NOT the ADR-050 Bash dispatch pattern) because the MCP tool handler runs in the persistent server process â€” there's no sub-agent lifetime issue that ADR-050 was designed to work around. The `MultiLlmReport` shape is `{ dispatchedAt: string, totalDurationMs: number, successCount: number, failureCount: number, results: MultiLlmResult[] }` where each `MultiLlmResult` is `{ provider, ok, response?, model?, sessionId?, usage?, durationMs, error? }`. Both shapes are validated by an exported Zod schema (`multiLlmReportSchema`) so the tool can declare it via `outputSchema` and clients can structurally extract per-provider results. The orchestrator's `index.ts` registers a new `multi-llm` tool inline that delegates to `dispatchMultiLlm`, plumbs `loadedExecutors.get` as the `getExecutor` callback, plumbs `sessionUsage.record` as the `recordUsage` callback (so multi-llm calls feed the same in-memory usage accumulator that `get-usage-stats` reports), and returns both the formatted markdown text (via `formatMultiLlmReport`) AND the structured `MultiLlmReport` (via `structuredContent`). **Explicitly not in this scope**: per-provider session continuity (every multi-llm call is a fresh dispatch â€” `sessionId` is not in the input schema), per-provider model override (you can't say "use gemini-flash for gemini and gpt-5.4-mini for codex" â€” each provider uses its default), streaming output (the structured report is built once when all providers complete), cancellation (`Promise.all` waits for the slowest provider; no early termination). All of those are real future enhancements but each adds enough complexity that bundling them risked the V1 scope. 15 new tests in `packages/llm-mcp/src/__tests__/multiLlm.test.ts` cover dispatch happy path (collects responses + sessionId + usage), per-call usage callback, executor-throws â†’ ok=false, missing executor â†’ ok=false, threadId fallback when sessionId is absent, ISO timestamp + numeric duration, parallelism (total time â‰ˆ slowest, not sum), schema conformance round-trip, formatter rendering, input schema validation (provider whitelist enforcement, optional providers, prompt minimum length, fallback when no available providers).
- **Consequences:** The orchestrator now ships 5 MCP tools instead of 4: `ask-llm`, `ping`, `get-usage-stats`, `diagnose`, **`multi-llm`**. The `multi-llm` tool is the first one in the project that's purely orchestrator-only (per-provider servers can't compose multiple loaded executors, so they don't get this tool). Any MCP client â€” including non-Claude-Code clients like Cursor, Warp, Copilot â€” can now do multi-provider dispatch in one call: `tools/call multi-llm { prompt, providers: ["gemini","codex"] }` returns `{ structuredContent: { results: [...], successCount, failureCount, totalDurationMs, dispatchedAt } }`. Per-provider failures don't fail the whole call â€” clients see `results[i].ok = false, results[i].error = "..."` for failed providers and `results[i].ok = true` with response data for successful ones. This is the multi-provider dispatch primitive that `/compare`, `/multi-review`, and `/brainstorm` skills implement via Bash from inside Claude Code â€” now exposed at the MCP layer for any client. The orchestrator's tool surface grew by approximately 280 tokens (the new tool's schema including the nested `MultiLlmReport` outputSchema), bringing the orchestrator's published surface from ADR-065's ~900 tokens to ~1180 tokens â€” slightly above the pre-ADR-029 standalone Gemini surface of 973 tokens, but the orchestrator now does meaningfully more (single provider, multi-provider parallel, diagnostics, usage stats, all under one MCP server). Test count: 412 (up from 397, +15 multi-llm tests). **Coexistence with existing skills is intentional**: `/compare` (skill, Claude Code-only, includes Claude verification) and `multi-llm` (MCP tool, any client, raw structured) serve different audiences and shouldn't be merged. Future work: per-provider session continuity (would require `Record<string, string>` sessionId input shape), per-provider model override (similar shape complexity), early-termination support (e.g., "return as soon as 2 of 3 providers respond" for fault-tolerant patterns), streaming the merged report incrementally as providers complete (would require more sophisticated transport handling than the synchronous request-response we currently use).

## ADR-065: `outputSchema` on `ask-*` Tools â€” Unified `AskResponse` Shape
- **Date:** 2026-04-17
- **Status:** Accepted (extends ADR-055's outputSchema pattern)
- **Context:** ADR-055 added `outputSchema` to two diagnostic tools (`get-usage-stats`, `diagnose`) and proved out the pattern in the SDK. The four free-form provider tools (`ask-gemini`, `ask-codex`, `ask-ollama`, and the orchestrator's `ask-llm`) still returned plain text â€” programmatic consumers had to regex-parse the response footer to extract sessionId, threadId, or stats. Today the response text is structured by convention only: `[Session ID: abc-123]` for Gemini, `[Thread ID: thread-xyz]` for Codex, and per-provider stats footers (`[Gemini stats: ...]`, `[Codex stats: ...]`, `[Ollama stats: ...]`) â€” each with a different prefix label and format. A client wanting to feed the sessionId into a follow-up call has to (a) know which provider was called, (b) pick the right regex (`Session ID` vs `Thread ID`), and (c) extract the value. With outputSchema, that whole pipeline collapses to `result.structuredContent.sessionId`. The user's strategic pivot to MCP-tool-surface focus (2026-04-17) makes this the right time â€” programmatic MCP clients are exactly the audience that benefits.
- **Decision:** New `packages/shared/src/askResponse.ts` defines a canonical `AskResponse` shape via Zod: `{ provider: "gemini" | "codex" | "ollama"; response: string; model: string; sessionId?: string; usage?: UsageStats }`. Two design decisions baked into the shape: (1) **`response` is the raw model text only** â€” no `[Session ID: ...]` footer, no `Gemini response:` prefix, no `[Gemini stats: ...]` block. The formatted text content in `content[0].text` keeps the human-readable footers for backward compatibility, but `structuredContent.response` is clean for programmatic consumers. (2) **Unified `sessionId` field across providers** â€” Codex's `thread_id` maps to `structuredContent.sessionId`, Gemini's `session_id` also maps to `structuredContent.sessionId`. Clients can resume any provider's conversation by passing this single field as the `sessionId` arg on the next call, regardless of which provider is chosen â€” semantically they're both "session-resumption identifier" and the type-level naming difference is a leakage of provider-specific vocabulary. Updated four tools to use the shape: `ask-gemini` (Gemini-MCP), `ask-codex` (Codex-MCP), `ask-ollama` (Ollama-MCP), and the orchestrator's inline `ask-llm` registration. Each tool now adds `outputSchema: askResponseSchema` to its `UnifiedTool` definition (or to the inline `server.registerTool` config for the orchestrator) and changes its `execute` to return `{text: string, structuredContent: AskResponse}` instead of a bare string. The existing `registerTools` factory in `@ask-llm/shared/serverFactory.ts` already branches on the result variant (added in ADR-055), so no changes needed there. **Explicitly out of scope for this round:** `ask-gemini-edit` is excluded because its output is *structured differently* (changeMode OLD/NEW edit blocks parsed into an `EditChunk[]`) â€” it deserves its own dedicated `editResponseSchema` in a future ADR rather than being shoehorned into the free-form `AskResponse` shape. 7 new tests in `packages/shared/src/__tests__/askResponse.test.ts` validate the schema (minimum required shape, all 3 providers accepted, unknown providers rejected, optional sessionId, full UsageStats, malformed usage rejected, response field type-checking).
- **Consequences:** Programmatic MCP clients can now structurally extract the model response text, the session-resumption identifier, the actual model that answered (which may differ from the requested model when fallback fired), and the full UsageStats â€” without regex-parsing the response footer. Multi-turn conversation continuation collapses from "look for Session ID or Thread ID label, regex-extract the value, store it, pass it back next call" to "store `result.structuredContent.sessionId`, pass it back next call." Token-cost dashboards can read `result.structuredContent.usage` directly without parsing per-provider stats footer formats. The orchestrator's `ask-llm` benefits most because it already abstracts over providers â€” clients no longer need provider-specific extraction logic. **Backward compatibility is preserved**: every existing tool call still returns the same `content[0].text` (with the same footers and prefixes); `structuredContent` is purely additive. Clients that don't request or read `structuredContent` see no change. Tool surface impact: each of the 4 tools' published schema grew by approximately 110 tokens (the JSON schema for `AskResponse`), so total surface across the 4 tools grows by ~440 tokens. Per-server impact: ask-gemini-mcp ~110 token surface increase, ask-codex-mcp ~110, ask-ollama-mcp ~110, ask-llm-mcp ~110 (orchestrator). The orchestrator's total surface goes from ADR-058's ~790 tokens to ~900 tokens â€” still within ADR-029's "much smaller than pre-ADR-029 standalone Gemini's 973 tokens" framing. Test count: 397 (up from 390, +7 schema tests). The shape is intentionally *minimal* â€” no `requestedModel` vs `actualModel` split, no `formattedText` field duplicating `content[0].text`, no provider-specific extension fields. Future fields can be added as optional without breaking existing clients. Future work: `outputSchema` for `ask-gemini-edit` returning structured edit blocks (deferred); `outputSchema` for `fetch-chunk` (also deferred â€” its output is similarly structured-different).

## ADR-064: `/multi-review` and `/brainstorm` Polish â€” Verification, Diff Preprocessing, Failure Resilience
- **Date:** 2026-04-17
- **Status:** Accepted (extends ADR-049, ADR-050, ADR-060)
- **Context:** A real `/multi-review` session against this codebase on 2026-04-17 surfaced four concrete pain points in the existing skill instructions: (1) **No diff preprocessing** â€” the user had to manually `git add -N` untracked files to include them in the review, and the diff was bloated by ADR/doc additions that providers don't need to "review"; without filtering, a 1206-line diff ballooned to 152KB and providers stalled or hit capacity. Both would have been "fixed" if the skill instructions said "trust the confidence score" â€” neither was caught until I read the source manually after the multi-review returned. (4) **No fallback dispatch** â€” when the plugin's reviewer agents (`gemini-reviewer`, `codex-reviewer`) aren't available in the current Claude Code session (e.g., plugin not installed yet), the skill provided no fallback path. The strategic context (the user's same-day pivot to "focus on MCP/plugin") raised the priority of polishing these skills since they ARE the surface where MCP value gets exposed inside Claude Code.

(1) **Phase 1 (gather and prepare)**: documents `git status --short` first, `git add -N <files>` for untracked, pathspec exclusion of docs/binaries (`:!docs/` `:!apps/docs/` `:!*.md` `:!yarn.lock` `:!*.lock` `:!*.png` `:!*.jpg` `:!*.svg`), and a three-tier size policy (`<50KB` send as-is, `50â€“150KB` warn the user about expected wall time, `>150KB` ask before sending and offer truncate-or-split). (2) **Phase 2 (dispatch with fallback)**: explicit fallback path â€” if reviewer agents unavailable, use direct Bash dispatch via `${CLAUDE_PLUGIN_ROOT}/dist/run.js` and `dist/codex-run.js` with the ADR-050 pattern (single foreground blocking call, direct backgrounding, per-PID `wait`, `GMCPT_TIMEOUT_MS=1500000` for 25-min provider timeout, Bash tool `timeout: 600000` for the 10-min outer cap). Explicitly forbids raw `gemini -p` / `codex exec` because they bypass the project's quota fallback (ADR-044), Codex stdin handling (ADR-042), and PATH resolution (ADR-047). (3) **Phase 3 (verify each finding)**: the highest-leverage addition â€” for every finding above 80/100 confidence, Read the file at the cited line and check if the claim is actually true; classify as `VERIFIED` (claim holds), `REJECTED` (false positive), or `UNVERIFIABLE` (cannot confirm without runtime). This step exists specifically because of the 2026-04-17 false-positives â€” the example is documented in the skill's preamble. (4) **Phase 4 (resilient failure handling)**: when a provider fails (timeout, exit â‰  0, 0-byte output), surface the failure inline with stderr instead of silently dropping; explicitly says "a single-provider review is still useful â€” present what you have." (5) **Phase 5 (synthesis)**: new output structure with `Verified by both`, `Verified by Gemini only`, `Verified by Codex only`, `Rejected`, `Unverifiable`, and per-provider stats including verification counts. The `/brainstorm` skill received a smaller polish since most of its behavior lives in the brainstorm-coordinator agent: added the same diff-preprocessing guidance (filter docs/binaries, warn on `>150KB`), a "confidence scores are not an oracle" reminder citing the same 2026-04-17 false-positive case, and a pointer to `/multi-review` for code-specific source-verified review. The `brainstorm-coordinator.md` agent's Phase 4 synthesis now opens with a "Cross-check high-confidence external claims first" paragraph and adds `VERIFIED / REJECTED / UNVERIFIABLE` classification, plus a new `Rejected (false positives)` section in the synthesis output template. 16 new tests in `packages/claude-plugin/src/__tests__/skills-and-agents.test.ts` pin every load-bearing structural element across the three files: diff preprocessing instructions, size-tier thresholds, verification language, finding classification labels, fallback-dispatch documentation, the ADR-050 pattern, the "no raw CLI" rule, failure-resilience instructions, and the cross-check + rejected-section additions on the agent.
- **Consequences:** `/multi-review` now produces verified findings, not raw provider output. The 2026-04-17 false-positive scenario (where I spent significant time chasing claims that turned out to be wrong) becomes structurally hard to repeat â€” Phase 3 of the skill explicitly tells Claude to verify before presenting. Diff preprocessing eliminates the entire class of "silent fail because we sent a 4MB diff including yarn.lock and PNGs" â€” the pathspec filters and size tiers make the cost visible to the user before dispatch. Fallback dispatch means the skill works whether or not the plugin's reviewer agents are loaded in the current session, which matters during plugin development and for users trying `/multi-review` before installing the full plugin. Failure resilience means the user never sees a confident-looking synthesis that's actually based on one provider after the other failed â€” partial results are explicit. Test coverage: 16 new tests pin every behavior; if a future edit drops the verification step or removes the size tiers, tests fail before the regression ships. Plugin test count: 82 (up from 66; +16 polish tests). Total project tests: 390 (up from 374). No code change in any executor or MCP server â€” this is purely skill/agent prose polish. The `/multi-review` skill grew from 42 lines to 105 lines; the `/brainstorm` skill grew from 33 to 50 lines; the brainstorm-coordinator agent's Phase 4 grew by 6 lines for cross-checking + rejected section. No tool surface change, no new ADR fields, no new types. Future work: apply the same verification-first pattern to `/gemini-review`, `/codex-review`, `/ollama-review` single-provider skills (deferred â€” they're simpler and don't have the same multi-provider complexity).

## ADR-063: Session Continuity Hardening â€” Three Codex/Ollama/Sessions Fixes Bundled
- **Date:** 2026-04-17
- **Status:** Accepted (corrects ADR-058 implementation in three places)
- **Context:** ADR-058 promised end-to-end session continuity across all three providers (Gemini native `--resume`, Codex native `exec resume`, Ollama server-side replay). A `/multi-review` pass on 2026-04-17 caught three concrete bugs in the implementation that all undermine that promise. The findings were Codex-flagged at 90â€“95/100 confidence and were verified by reading the source. Together they invalidated three of the four claims ADR-058 made: (a) "Codex native session continuity works" â€” broken because `--ephemeral` was being passed on every Codex call including resume, but `--ephemeral` is documented to skip session persistence, so the Thread ID returned to the user could not actually be resumed; (b) "Server-side replay is safe" â€” broken because session JSON files were written under `os.tmpdir()` with default `mkdirSync`/`writeFileSync` permissions (typically 0o755 dir / 0o644 file with standard umask), making prompts and Ollama responses readable by any other user on the machine; (c) "Pass empty string to start a new Ollama session" (a documented entry path) â€” broken because the cache check used `sessionId ? null : ...` truthiness, so the empty string was falsy and the cache was hit instead of creating a fresh session. Strategic context (from the user's "focus on MCP/plugin not CLI" pivot the same day) raised the urgency: under the new focus, ADR-058's session capability is exactly the kind of MCP-tool-surface promise that has to actually work.
- **Decision:** Bundled three small fixes under one ADR rather than three separate ones, since they are conceptually related (all session continuity correctness) and were identified together. (1) **Codex `--ephemeral` only when no session is wanted**: `packages/codex-mcp/src/utils/codexExecutor.ts:buildArgs` now conditionally appends `CLI.FLAGS.EPHEMERAL` only when `sessionId` is undefined. When `sessionId` IS set (initial call where the user wants a resumable thread, or a `resume <id>` call), `--ephemeral` is dropped so Codex persists the session. Also changed the cache-key gate from `sessionId ? null : ...` to `wantsSession = sessionId !== undefined` so that an empty-string sessionId disables caching there too (the same fix applied for Ollama in #3). (2) **Session file permissions hardened**: `packages/shared/src/sessions.ts` now defines `SESSION_DIR_MODE = 0o700` and `SESSION_FILE_MODE = 0o600` constants. `ensureSessionDir()` creates the directory with `{ recursive: true, mode: 0o700 }` AND validates an existing directory via `lstatSync` â€” rejecting symlinks (defense in depth against tmp races where an attacker pre-creates the path as a symlink to somewhere they want to read), rejecting non-directories, and `chmodSync`-tightening the mode if it's currently world-readable. `saveSession()` now uses an atomic temp+rename pattern: writes to `${finalPath}.tmp.${pid}.${timestamp}` with `mode: 0o600`, then `renameSync` to the final path. Atomic rename means readers never see partial writes; the unique tmp suffix means concurrent saves don't collide. On failure the tmp file is unlinked. The exported `__test__` map gained `SESSION_DIR_MODE` and `SESSION_FILE_MODE` so tests can assert against them. (3) **Ollama empty-string sessionId disables cache**: `packages/ollama-mcp/src/utils/ollamaExecutor.ts` mirrors the Codex fix â€” `wantsSession = sessionId !== undefined` keys the cache decision on presence, not truthiness. The downstream `appendAndSaveSession(sessionId, ...)` call already correctly handled the empty string (it generated a new UUID via `createSessionId()` when `id` was falsy), so once the cache stops short-circuiting, the documented "pass empty string for new session" UX path works end-to-end. 10 new tests across the three files: 4 in `codexExecutor.test.ts` (ephemeral present without session, omitted with session, exec-resume sequence with session, cache disabled by empty-string sessionId); 4 in `sessions.test.ts` (dir mode 0o700, file mode 0o600, existing world-readable dir gets tightened, no leftover .tmp files); 2 in `ollamaExecutor.test.ts` (undefined sessionId hits cache, empty-string sessionId bypasses cache).
- **Consequences:** ADR-058's three broken sub-features now actually work end-to-end. Codex users can resume conversations via `sessionId: <thread-id>` parameter on the `ask-codex` MCP tool â€” the second turn correctly references the persisted session instead of dying with "session not found." Ollama session files on macOS/Linux are now owner-only (0o700/0o600), eliminating the cross-user prompt/response disclosure on shared `/tmp/`. Tests skip the permission assertions on Windows (`it.skipIf(process.platform === "win32")`) since POSIX permission semantics don't translate cleanly. The "empty-string sessionId starts a new session" UX path documented in `ask-ollama.tool.ts` and `ask-codex.tool.ts` schemas now actually does what it claims. No tool surface change â€” these are corrections to existing behavior, not new capabilities. Test count: 374 (up from 364, +10). Lint clean across all packages, full build clean. The lstat-based symlink rejection in `ensureSessionDir` is a defense-in-depth measure against a tmp-race attack (attacker creates `/tmp/ask-llm-sessions` as a symlink to `~/.ssh/` before the first server invocation) â€” even if `mkdirSync` doesn't follow the symlink, validating before write protects existing-directory cases.

## ADR-062: Interactive CLI REPL â€” `npx ask-llm-mcp repl`

## ADR-058: Session Continuity Across All Three Providers â€” Native Resume + Server-Side Replay
- **Date:** 2026-04-16
- **Status:** Accepted (partially supersedes ADR-034's `sessionId` schema removal)
- **Context:** The project's primary use case (per `CLAUDE.md`) is "low-friction back-and-forth between Claude and Gemini for validation and debate." That requires multi-turn conversations: ask provider X to review a plan, get pushback, refine, ask again with prior context intact. Today only Gemini's executor accepts a `sessionId` (ADR-021's `--resume`), and ADR-034 deliberately *removed* `sessionId` from the `ask-gemini` MCP tool schema for simplicity, leaving session IDs surfaced in responses but unable to be passed back in. Effectively no provider supports continuation through the MCP surface. The Codex CLI 0.118 has native `codex exec resume <SESSION_ID> [PROMPT]` support â€” the same `thread_id` already returned in `thread.started` events is the resumable id. Ollama has no native session support but its `POST /api/chat` accepts a full `messages` array; conversation state can be maintained server-side and replayed on each call. Decision C from the user's earlier triage locked the design: server-side store at `/tmp/ask-llm-sessions/`, mirroring the existing `chunkCache.ts` pattern, with Codex and Ollama as the providers needing it. Investigation revealed Codex actually has native resume too â€” server-side replay is only required for Ollama. This narrows the blast radius further than the original triage assumed.
- **Decision:** (1) New `packages/shared/src/sessions.ts` mirrors `chunkCache.ts`'s patterns: file-based store at `os.tmpdir()/ask-llm-sessions/<id>.json`, 24-hour TTL (longer than chunkCache's 10 minutes since conversations span longer), 200-file cap (vs chunkCache's 50 since concurrent conversations are common). The `SessionRecord` shape is `{ id, provider, model, messages: SessionMessage[], createdAt, updatedAt }` where each message is `{ role: "user" | "assistant", content: string }`. A `MAX_MESSAGES_PER_SESSION = 40` cap prevents unbounded context growth â€” when exceeded, the OLDEST messages are dropped, keeping the most recent 40 (20 user/assistant pairs). The store rejects unsafe session IDs at both read and write boundaries via `isSafeSessionId()` regex (`/^[A-Za-z0-9_-]{8,128}$/`), preventing path traversal even before the file system gets involved. Public API: `createSessionId()` (UUID via `node:crypto.randomUUID`), `loadSession(id)`, `saveSession(record)`, `appendAndSaveSession(id, provider, model, userContent, assistantContent)` (the high-level helper that loads-or-creates, appends, caps, saves, and returns the assembled state), and `buildPriorMessages(id)` (load helper returning the message array for replay). 18 unit tests cover round-trip, TTL expiry, message-cap trimming, unsafe-id rejection, and the create-or-resume semantics. (2) Codex executor gains `sessionId?: string` option. When provided, `buildArgs` emits `codex exec resume <SESSION_ID> [flags] <prompt>` instead of `codex exec [flags] <prompt>` â€” same flags (`--full-auto`, `--json`, `--ephemeral`, `--skip-git-repo-check`, `-m model`) work with both. Verified via `codex exec resume --help`. The existing `thread.started` event capture means the returned `threadId` is already the resumable id â€” users paste it back as `sessionId` to continue. Response cache is bypassed when sessionId is set (same prompt in different conversations should yield different answers). (3) Ollama executor gains `sessionId?: string`. When provided, `buildPriorMessages(sessionId)` loads the prior history and the executor sends `[...prior, { role: "user", content: prompt }]` to `POST /api/chat` instead of just the new message. After a successful response, `appendAndSaveSession` records the new turn. The response footer includes `[Session ID: <id>]` (and `(new)` if the executor created the session â€” i.e., the user passed an unknown id or none). Cache also bypassed when sessionId is set. (4) Tool schemas: re-added `sessionId: z.string().optional()` to `ask-gemini`, `ask-codex`, `ask-ollama`, AND the orchestrator's `ask-llm` tool. The orchestrator's `ExecutorFn` type signature gained `sessionId?: string` input + `sessionId?: string` / `threadId?: string` output, so it routes the value to the right executor and surfaces the appropriate id in the response footer. This is a partial reversal of ADR-034's "removed sessionId from tool schemas â€” sandbox is a niche feature, sessionId is returned in responses for manual use." The reversal acknowledges that "manual use" means nothing in MCP context where the schema *is* the only interface. The cost is one optional parameter per tool; the value is end-to-end multi-turn capability across all three providers.
- **Consequences:** All three providers now support multi-turn conversations through the MCP surface. Gemini and Codex use native CLI resume (zero replay cost â€” the providers retain state). Ollama uses server-side replay (the prior `messages` array is sent on each turn, costing input tokens proportional to conversation depth â€” but Ollama runs locally with no token bill, and the 40-message cap bounds growth). Tool surface impact: each tool gains one optional parameter (~50 tokens added to each tool's published schema). Aggregate: ask-gemini + ask-codex + ask-ollama + ask-llm orchestrator = ~200 tokens added across the published surface. The orchestrator's surface goes from ADR-056's ~740 tokens to ~790 tokens â€” still 19% smaller than pre-ADR-029 standalone Gemini (973 tokens). ADR-034's "exactly four tools, simplified to 2 params" claim is technically violated for `ask-gemini` (now 3 params), but ADR-034's intent (don't over-expose niche options) is preserved by keeping `sessionId` optional and clearly described. Security: `isSafeSessionId` blocks `../`, slashes, and shell metacharacters at both read and write boundaries â€” the file path can never escape `/tmp/ask-llm-sessions/`. The Ollama session store opts users in to local persistence of conversation history; the description on the tool schema makes this explicit. 269 tests pass (18 new sessions tests in `@ask-llm/shared`), lint clean across all 6 packages, full build clean, no smoke-test failures. Future work: an MCP `resources/list` entry could expose `sessions://list` for users to discover their active sessions â€” deferred to a follow-up ADR.

## ADR-057: Stream-JSON Output for Gemini Executor â€” Live Progressive Content via Progress Notifications
- **Date:** 2026-04-16
- **Status:** Accepted
- **Context:** Gemini calls through the MCP server are inherently slow â€” large-context analyses on `gemini-3.1-pro-preview` routinely take 60â€“180 seconds, and the brainstorm-coordinator's external dispatches can stretch to 10 minutes (ADR-050). The existing executor uses `--output-format json` which buffers the full response and emits it as a single JSON object at the end. During execution, the project's `createProgressTracker` (ADR-039/041) emits keep-alive notifications every 25 seconds with the last 150 chars of raw stdout â€” but raw stdout is unparsed CLI noise, not the model's text, so the notifications display gibberish like `"text":"...","delta":true},{"type":"messa` mid-token. Users perceive long calls as unresponsive even when the model is actively producing tokens. Gemini CLI 0.37+ supports `--output-format stream-json` which emits a JSONL event stream: `{"type":"init",...}` / `{"type":"message","role":"assistant","content":"...","delta":true}` per chunk / `{"type":"result","status":"...","stats":{...}}` at end. Switching to this format unlocks two things: (1) the `--include-directories` analysis can stream prose to the user as the model produces it; (2) the existing `onProgress` plumbing (already wired through `executeCommand` â†’ `createProgressTracker.updateOutput`) already accepts text â€” we just need a parser that extracts assistant content from JSONL events and forwards it. The MCP client (Claude Code, Claude Desktop, etc.) renders progress notifications inline, so the user sees real model output unfolding instead of a frozen spinner. Codex and Ollama have their own streaming protocols (Codex JSONL is already collected at end, Ollama HTTP `stream: true` would be an HTTP body refactor) â€” they're explicitly out of scope for this ADR; one provider at a time keeps blast radius small and lets us verify the pattern before generalizing.
- **Decision:** (1) Added `STREAM_JSON: "stream-json"` to `OUTPUT_FORMATS` in `packages/gemini-mcp/src/constants.ts` and switched `buildArgs` to emit it in place of `JSON`. (2) New `parseGeminiStreamJsonl(raw, model, durationMs, fellBack): GeminiExecutorResult` function in `geminiExecutor.ts` that splits raw stdout by newlines, parses each JSON line, and aggregates: `init.session_id` â†’ sessionId; `message` events with `role: "assistant"` â†’ assembled content (delta:true appends, non-delta replaces); `result.stats` â†’ token usage; `result.status === "error"` or standalone `error` events â†’ throw. ## ADR-055: Add MCP `outputSchema` to `get-usage-stats` and Expose Live Snapshot as a Resource
- **Date:** 2026-04-16
- **Status:** Accepted (extends ADR-054)
- **Context:** ADR-054 promoted `get-usage-stats` to a first-class tool returning a markdown-formatted summary as `content[0].text`. That works for human consumers but is hostile to programmatic ones â€” any client that wants to read individual fields (e.g., to drive a dashboard, enforce a per-session token budget, or feed totals into a parent agent's context) has to parse markdown. The MCP spec since 2025-06-18 supports `outputSchema` on tools, where a Zod schema declares the shape of `structuredContent` returned alongside the text content. Clients that introspect `structuredContent` get the data structurally; clients that don't ignore the field and read the text. SDK 1.27.1 supports both `outputSchema` in `registerTool` config and `registerResource` for static URI-keyed resources. Separately, MCP Resources (distinct from Tools) are the spec-blessed surface for "pollable state" â€” a tool call implies side effects, a resource read implies inspection. The session usage snapshot is a textbook fit: it is read-only, idempotent, and a resource read costs zero against the tool-token budget that ADR-029 established as a load-bearing invariant.
- **Decision:** (1) Extended `UnifiedTool` with an optional `outputSchema?: ZodTypeAny` field and a `ToolResult = string | { text: string; structuredContent: Record<string, unknown> }` return type for `execute`. Tools that don't opt in keep returning a plain string and behave identically to before this ADR. Tools that opt in return both representations from a single `execute` call â€” a single source of truth, no risk of text/structured drift. `registerTools` in `@ask-llm/shared/serverFactory.ts` detects the union variant at runtime and emits the appropriate `CallToolResult` (`{content, structuredContent}` when structured, `{content}` when plain string). The orchestrator (`llm-mcp/src/index.ts`), which has its own bespoke registration path, applies the same branch inline. (2) `createUsageStatsTool` factory now returns a tool with `outputSchema: sessionUsageSnapshotSchema` (a Zod definition mirroring `SessionUsageSnapshot`: totals, per-provider buckets, per-model buckets) and an `execute` that returns both the markdown text and the raw `SessionUsageSnapshot` cast to `Record<string, unknown>`. Six new unit tests verify schema/structured-content round-trips and the per-instance closure isolation. (3) New `registerSessionUsageResource(server, sessionUsage)` helper in `serverFactory.ts` registers an MCP Resource at the URI `usage://current-session` with `mimeType: application/json`. The read callback returns the live `JSON.stringify(sessionUsage.snapshot(), null, 2)` â€” re-reading at any time yields the current totals, no caching. All four servers wire it in alongside the tool registration. (4) Explicitly out of scope: `outputSchema` for `ask-*` tools (their responses are free-form LLM text â€” structuring is artificial and would require a parsing layer that doesn't exist today); `models://<provider>` resources for model discovery (would require new provider model-list APIs and per-provider availability handling â€” separate scope, deferred).
- **Consequences:** Programmatic consumers can now read structured `SessionUsageSnapshot` data via either path: a `tools/call` of `get-usage-stats` returns `structuredContent` (clients that don't understand the field gracefully ignore it and use the markdown), or a `resources/read` of `usage://current-session` returns canonical JSON. Both surfaces share the same `sessionUsage.snapshot()` source â€” no duplication or drift risk. Tool-token-budget impact is bounded: `outputSchema` adds approximately 95 tokens to the `get-usage-stats` registration (the JSON schema for the snapshot) â€” present once per server. The orchestrator surface goes from ADR-054's ~422 tokens to ~517 tokens (still 47% smaller than the pre-ADR-029 standalone Gemini surface of 973 tokens). MCP Resources sit in their own `resources/list` namespace and contribute zero to the `tools/list` budget. The tool-count claim from ADR-054 is unchanged (Gemini 5, others 3) â€” adding `outputSchema` doesn't add a tool. Resources are an entirely new published surface for this project, but they're the spec-blessed home for inspection-only state and ADR-029's token-efficiency principle is preserved by construction. The pattern is now in place for future `outputSchema` opt-ins on other tools when the data shape genuinely warrants it. 217 tests pass, lint clean.

## ADR-052: Rewrite workspace:* â†’ * in Published MCP Packages to Fix npm 9 EUNSUPPORTEDPROTOCOL
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** User report: `npx -y ask-llm-mcp` from Claude Desktop (which ships with Node 18 / npm 9.7.1) fails immediately during install with `npm ERR! code EUNSUPPORTEDPROTOCOL â€” Unsupported URL Type "workspace:": workspace:*`. The server never boots. Investigation revealed that all four published MCP packages (`ask-gemini-mcp@1.5.5`, `ask-codex-mcp@0.2.5`, `ask-ollama-mcp@0.2.5`, `ask-llm-mcp@0.2.5`) have `"@ask-llm/shared": "workspace:*"` (and for `ask-llm-mcp`, also the three other sibling packages) literally in their published `dependencies` field. The bug had been latent in the published packages for a long time â€” it only surfaced because of a subtle distinction between two install paths: (1) `npm install tarball.tgz` reads the local tarball directly, respects `bundledDependencies`, and uses the bundled `node_modules/` inside the tarball â€” succeeds on npm 9; (2) `npm exec`/`npx -y package` fetches the package MANIFEST from the npm registry FIRST (literally the published `package.json`), computes the dependency tree from that manifest via the arborist, and only THEN downloads the tarball. npm 9's arborist doesn't recognize the `workspace:` URL type in a registry manifest and fails before it ever sees the bundled deps. `bundledDependencies` does not help on the manifest-fetch path. Confirmed empirically: installing the `ask-llm-mcp@0.2.5` tarball with `npm install /path/to/tarball.tgz` on Node 18.15.0 / npm 9.7.1 succeeds, but `npx -y ask-llm-mcp` on the same machine fails with `EUNSUPPORTEDPROTOCOL`. The "works for some users, fails for others" pattern was explained by npx cache: once an extraction existed at `~/.npm/_npx/<hash>/`, successive runs reused it and never re-ran the manifest-fetch path. A new version bump invalidated the cache and cold-installs failed.
- **Decision:** Added `scripts/prepack-bundle.mjs` â€” a small Node helper that each MCP package calls from its `prepack` script. The helper does three things: (1) backs up the package's `package.json` â†’ `package.json.bak` so postpack can restore it; (2) copies the requested workspace sibling packages (`packages/shared/dist` + `package.json`, etc.) into `./node_modules/` under the npm-package names (`@ask-llm/shared`, `ask-gemini-mcp`, etc.) â€” this is the existing `bundledDependencies` bundling mechanism; (3) rewrites `workspace:*` â†’ `*` in the top-level `package.json` AND in every bundled nested `package.json`. The `*` semver range is universally parseable by every npm version, and at install time npm honors `bundledDependencies` by using the bundled folder instead of going back to the registry. Added `scripts/postpack-restore.mjs` that runs after `npm pack` (including after `npm publish`) to restore `package.json` from the backup, so the developer's working tree still has `workspace:*` and yarn's workspace resolution continues to work at dev time. Each MCP package's `prepack` was replaced with `node ../../scripts/prepack-bundle.mjs <deps>` â€” gemini/codex/ollama just pass `shared`, llm-mcp passes all four (`shared gemini-mcp codex-mcp ollama-mcp`). Each MCP package also got a `postpack: node ../../scripts/postpack-restore.mjs` entry. Added `packages/*/package.json.bak` to `.gitignore` as defensive cleanup for aborted packs. Rejected alternatives: (a) rewriting `workspace:*` â†’ a concrete version like `^0.2.6` â€” `@ask-llm/shared` is private (not published), so the registry would 404 on resolution; (b) using `file:./node_modules/...` paths â€” more complex and no better than `*` with bundledDependencies; (c) switching to `yarn npm publish` â€” Yarn Berry's publish rewrites workspace protocol automatically, but `yarn pack` ignores `bundledDependencies` entirely (per the explicit comment in `.github/workflows/release.yml`), so switching would regress the bundled-dep shipping mechanism; (d) fixing this at install time with a `preinstall` hook â€” runs too late, the arborist parses the manifest before any install hooks fire.
- **Consequences:** Published `ask-llm-mcp@0.2.7`, `ask-codex-mcp@0.2.7`, `ask-ollama-mcp@0.2.7`, `ask-gemini-mcp@1.5.7` install cleanly under npm 9 (Claude Desktop's Node 18) as well as npm 10/11. End-to-end verified: built tarball, inspected extracted contents, installed under Node 18.15.0 / npm 9.7.1, executed the bundled CLI. The developer working tree still has `workspace:*` after `npm publish` completes (postpublish restores from `.bak`), so yarn workspace resolution continues to work unchanged for dev. Bumped `@ask-llm/shared` to 0.2.7 for consistency even though it isn't published.
- **2026-04-10 correction â€” `postpack` vs `postpublish`:** The first release attempt (1.5.6 / 0.2.6) registered the restore as a `postpack` hook. This produced a subtle split-brain: the published *tarball* had the rewritten `"*"` (correct), but the published *registry manifest* still had `"workspace:*"` (broken). Root cause: npm's publish pipeline reads `package.json` metadata for the manifest **AFTER** `postpack` runs (visible via the `npm warn publish auto-corrected` log line appearing after postpack output). So `postpack` restoring the working tree undid the rewrite before npm read it for the manifest. Moving the restore from `postpack` to `postpublish` fixes it â€” `postpublish` runs after the manifest has been uploaded to the registry, so the manifest reflects the rewritten `"*"`. Verified by running `npm publish --dry-run` and observing the ordering: `prepack â†’ pack â†’ npm warn publish (manifest read, still rewritten) â†’ Publishing â†’ postpublish (restore)`. The 1.5.6 / 0.2.6 versions are now *tombstones* on npm â€” their tarballs are usable via `npm install tarball.tgz` but `npx -y` still fails on them because the registry manifest is broken. **Users must install 1.5.7 / 0.2.7 or later to get the working fix.** Trade-off: `postpublish` only runs on `npm publish`, not on `npm pack`, so `npm pack` alone leaves the working tree rewritten â€” a developer running `npm pack` manually must `git checkout packages/X/package.json` to restore. This is acceptable because `npm pack` is rarely run outside CI, and the failure mode is visible (working tree shows `"*"` instead of `"workspace:*"`) rather than silent.

## ADR-051: Smoke Tests Skip-With-Warning on Quota/Rate-Limit Errors
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** ADR-043 introduced a Husky pre-push hook that runs real smoke tests against locally-installed CLIs (Gemini, Codex, Ollama) on every push. The intent was a "developer hygiene" gate that catches broken provider integrations before they reach `main`. In practice, the design has a self-defeating feedback loop: the smoke test itself makes a real Gemini API call, which counts against the user's quota window. When a developer pushes multiple commits in a single working session (as happened during the #22/#23/Stop-hook release sequence on 2026-04-10), the smoke test on push N can fail with HTTP 429 simply because pushes 1..N-1 burned the quota. Empirical evidence from the 2026-04-10 session: smoke tests passed at 12:19, 12:27, 14:22; failed with `rateLimitExceeded` at 13:01 and 14:31. The 14:22 â†’ 14:31 gap was only 9 minutes â€” well within the same rate-limit window â€” so the success-then-failure pattern is direct evidence that the smoke test is sabotaging itself. The failing test is in `packages/gemini-mcp` integration tests; it has nothing to do with most pushes (e.g., docs-only pushes that don't touch gemini-mcp at all), yet it blocks them all the same.
- **Decision:** Modified `scripts/smoke-test.sh` to wrap each provider's smoke test in a `run_smoke` function that captures both the exit code and the test output, and applies a quota-detection escape hatch: if the test exits non-zero AND the captured output contains any known quota/rate-limit signal (`rateLimitExceeded`, `RESOURCE_EXHAUSTED`, `TerminalQuotaError`, `exhausted your capacity`, `code=429` â€” the same patterns the gemini-mcp executor already uses for its own quota fallback in ADR-044), the smoke test is treated as skip-with-warning rather than a hard failure. The pre-push hook continues for the remaining providers and the push proceeds. Real failures (non-quota errors) still abort the push as before. Added a `FORCE_SMOKE=1` env var that disables the escape entirely, so a developer who genuinely needs to verify quota-bearing tests can opt back into hard failure mode. Used POSIX-clean shell idioms (`set -o pipefail` + `|| rc=$?`) so the script works under bash 3.2 / `sh` POSIX mode on macOS without bash 4 features.
- **Consequences:** The rate-limit-self-defeating loop is broken â€” multi-push sessions no longer fail on the second or third push of the day. Pushes that don't touch `gemini-mcp` source code are no longer blocked by upstream Gemini quota issues. The trade-off is that a real Gemini regression occurring at the same moment as a quota outage would be silently downgraded to a warning instead of a hard failure â€” but this is an extremely narrow window, and `FORCE_SMOKE=1` provides the explicit escape when it matters. Other providers (Ollama, Codex) get the same treatment for symmetry; Ollama runs locally so quota errors there are essentially impossible, and Codex has its own quota fallback chain (gpt-5.4 â†’ gpt-5.4-mini per ADR-028) that the smoke test exercises. Future maintainers will see the warning prominently in pre-push output when the escape fires, so the fact that "tests were skipped" is never silent. This is a pragmatic narrowing of ADR-043's hard-gate semantics â€” the gate is still there, just resilient to one specific class of upstream failure.

## ADR-050: Fix Sub-Agent Background Job Lifecycle Bug in brainstorm-coordinator
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** Issue #23. The `brainstorm-coordinator` agent dispatched external providers (Gemini, Codex, Ollama) via background bash jobs and then waited for Claude Code notifications to wake it up. This was broken: sub-agents cannot own background processes that outlive their turn. When the coordinator stopped issuing foreground tool calls, its turn ended, Claude Code tore down the sub-agent's shell context, and SIGKILLed every in-flight provider process. Codex at `reasoning=high` was especially vulnerable because it can take several minutes to produce a response â€” during that time the coordinator has no foreground work left, the turn ends, and Codex dies silently. The failure mode was invisible: 0-byte output files, no error, no exit code. User evidence trail from a real session: coordinator agent `ab5ea281ebabe67e0`, Codex output `/private/tmp/full-review-codex-response.md` (0 bytes), Codex stderr (86 lines, session header only, no model response). The user's sketched fix in the issue used `(cmd &) && wait` which contains a subtle subshell bug â€” parentheses spawn a subshell that detaches the child, so `wait` returns immediately and the "fix" produces the same orphan-and-SIGKILL behavior as the original. Discovered alongside issue #22 (Claude as first-class participant) â€” the two issues interact: #22's "Phase 3B runs in parallel with Phase 3A" framing is impossible to honor inside a sub-agent when #23 forbids background jobs.
- **Decision:** Restructured Phase 3 of `packages/claude-plugin/agents/brainstorm-coordinator.md` to run sequentially: Phase 3B (Claude Opus research, foreground) runs first, then Phase 3A (external provider dispatch) runs second as a SINGLE blocking foreground Bash tool call. Phase 3A uses direct backgrounding in the outer shell (`cmd > out 2>&1 & pid=$!`) â€” NOT subshells â€” and per-PID `wait` calls so every provider is a direct child of the outer bash and exit codes are captured independently. Required `timeout: 600000` (10 minutes, the Bash tool maximum) on the dispatch call â€” the default 2-minute timeout kills Codex at high reasoning effort mid-response, recreating the silent-failure class. Added per-provider stdout AND stderr capture so Phase 4 can distinguish "provider ran, returned nothing meaningful" from "provider was killed". Added a new "Critical: Sub-Agent Background Job Lifecycle" section in the agent prompt that documents the failure mode, the three anti-patterns to avoid (`run_in_background: true`, `(cmd &) && wait`, multi-call dispatch), and the two required patterns, with explicit reasoning. The sequential ordering also reinforces ADR-049's blindness constraint: Phase 3B completes before Phase 3A dispatches anything, so Claude cannot anchor on external responses because they do not exist yet. Rejected alternatives from the issue: (B) moving dispatch to the main conversation context â€” requires restructuring the skill-to-agent handoff and loses encapsulation; (C) using MCP tools directly instead of shelling out â€” the MCP tools are in the agent's frontmatter tools list but the user reported they did not surface in their session; shelling out is the reliably available path.
- **Consequences:** `/brainstorm` and `/brainstorm-all` no longer silently drop provider responses when the sub-agent turn ends. Codex at high reasoning effort now completes normally within the 10-minute cap. Per-provider exit codes and stderr are captured and surfaced in Phase 4, so failures are loud instead of silent. Total wall clock for a brainstorm run is now `time(Phase 3B) + time(slowest external provider)` â€” a modest increase over the previous (broken) "parallel" behavior, and bounded by the 10-minute Bash timeout cap. Providers exceeding the cap will still be killed, but cleanly with a foreground timeout error instead of a silent SIGKILL. Future maintainers have an explicit, in-prompt warning that documents the sub-agent background-job lifecycle gotcha â€” a Claude Code platform behavior that is not otherwise documented in the plugin codebase. Closes issue #23.

## ADR-048: Remove Stop Hook from Claude Code Plugin
- **Date:** 2026-04-10
- **Status:** Accepted
- **Context:** The Claude Code plugin shipped a `Stop` hook (`scripts/stop-review.sh`) that ran `gemini -p` against `git diff HEAD` on every Claude Code `Stop` event. Two problems surfaced in practice. (1) **Wrong trigger semantic:** the `Stop` event fires every time Claude finishes responding to a user turn, not once at the end of a session. In an iterative conversation this meant 5â€“20 Gemini invocations reviewing overlapping diffs of the same work-in-progress, burning quota and blocking the UI for up to 60s per turn. (2) **Structural coverage gap:** `git diff HEAD` excludes untracked files, so any session that created new files (the majority of feature work) received no review at all â€” the hook silently claimed coverage it did not provide. Both issues were documented in `docs/BUGS.md` but could not be fixed independently: widening the diff capture made the latency bug worse, and there is no alternative trigger semantically equivalent to "end of session" exposed to plugin hooks today. The plugin already provides an explicit on-demand alternative via the `/gemini-review` slash command, and the `PreToolUse` pre-commit hook still catches the meaningful integration boundary (code about to be committed).
- **Decision:** Removed the Stop hook entirely. (1) Deleted `packages/claude-plugin/scripts/stop-review.sh`. (2) Removed the `Stop` block from `packages/claude-plugin/hooks/hooks.json`, keeping `PreToolUse` for the pre-commit review. (3) Updated `packages/claude-plugin/README.md`, root `README.md`, and `apps/docs/plugin/overview.md` to remove Stop hook references from feature tables and prose. (4) Marked the two Stop-hook entries in `docs/BUGS.md` as resolved via removal. (5) Annotated `docs/ROADMAP.md` to record the removal. (6) Bumped `@ask-llm/plugin` to 0.2.9 and updated the marketplace manifest. Rejected alternatives: (a) switching to a `SessionEnd` event â€” no such plugin-level event exists in the current Claude Code schema; (b) adding debounce/state-file gating â€” band-aid over the wrong-trigger problem; (c) widening `git diff HEAD` to include untracked files â€” compounds the per-turn latency bug.
- **Consequences:** Plugin users no longer get automatic per-turn advisory reviews. Users who want a session review now invoke `/gemini-review` explicitly â€” this restores user agency over quota spend and privacy (diffs no longer silently leave the machine on every turn). The pre-commit hook remains as the automated safety net at the real integration boundary. Existing installs pick up the removal on next plugin update; the change is backwards-compatible in the sense that nothing breaks â€” users simply stop seeing unsolicited reviews. If a future Claude Code version exposes a true `SessionEnd` event, the feature can be reintroduced cleanly on a correct trigger.

## ADR-047: Shell PATH Resolution for macOS GUI Apps
- **Date:** 2026-04-08
- **Status:** Accepted
- **Context:** macOS GUI applications (Claude Desktop, Cursor, etc.) do not inherit the user's shell environment â€” `.zshrc`/`.zprofile` are never sourced. This means nvm, fnm, Volta, and Homebrew paths are missing from PATH. When our MCP server spawns CLI tools like `gemini` or `codex`, those CLIs resolve to the wrong Node version (e.g., system v18 instead of user's v24), causing ES2024 incompatibility crashes. This affects all macOS users who install Node via version managers.
- **Decision:** Added `shellPath.ts` to `@ask-llm/shared` with a 3-tier PATH resolution strategy: (1) `ASK_LLM_PATH` env var override if set, (2) Extract real PATH from user's login shell via `execFileSync(shell, ["-ilc", "echo $PATH"])`, (3) Heuristic fallback scanning `~/.nvm/versions/node/`, `~/.volta/bin`, `/opt/homebrew/bin`, `/usr/local/bin` for Node v20+. The resolved PATH is cached once at startup and passed to all `spawn()` calls via `getSpawnEnv()`. Also updated `llm-mcp`'s `isCommandAvailable()` to use the resolved PATH.
- **Consequences:** CLI tools spawned by our MCP servers now use the user's real PATH regardless of how the parent process was launched. The login shell extraction adds ~100ms to first command execution (cached thereafter). Windows is unaffected (GUI apps inherit PATH correctly). Users with non-standard setups can override via `ASK_LLM_PATH` env var in MCP config.

## ADR-024: MCP vs Skill vs Subagent Context Comparison Experiment
- **Date:** 2026-03-01
- **Status:** Informational
- **Context:** To understand how different Claude Code invocation methods affect Gemini's code review quality, we ran the same code review task through three approaches: MCP tool call (structured template with explicit instructions), Skill (markdown skill file with checklist), and Subagent (general-purpose agent with natural language prompt). - **Consequences:** When optimizing for speed/cost, prefer structured prompts with explicit output format instructions. When optimizing for thoroughness on novel problems, less-structured prompts may explore more reasoning paths. No code changes from this ADR.

## ADR-007: Utils Audit â€” Add Process Timeout and Windows Compatibility

## ADR-011: Update Default Models to Gemini 3.1 Pro / 3 Flash
- **Date:** 2026-02-23
- **Status:** Accepted
- **Context:** The project defaulted to `gemini-2.5-pro` with `gemini-2.5-flash` as the quota fallback. Google has released Gemini 3.1 Pro (`gemini-3.1-pro-preview`) as the latest Pro model and Gemini 3 Flash (`gemini-3-flash-preview`) as the latest Flash model. No 3.1 Flash exists yet. The upstream community requested this update (PR #54, Issue #51). Additionally, the quota fallback logic matched a hardcoded model-specific string (`"Quota exceeded for quota metric 'Gemini 2.5 Pro Requests'"`) which would fail to trigger for any other model generation.
- **Decision:** Update `MODELS.PRO` to `gemini-3.1-pro-preview` and `MODELS.FLASH` to `gemini-3-flash-preview`. Fix quota detection to match on `RESOURCE_EXHAUSTED` (the generic gRPC status code) instead of a model-specific quota metric string, so fallback works regardless of which model the user ran. Remove model generation names from status messages (e.g., "Gemini Pro quota exceeded" instead of "Gemini 3 Pro quota exceeded") so they stay accurate as models evolve.
- **Consequences:** Users get the latest Gemini models by default. Quota fallback now works for any model generation (2.5, 3.0, 3.1, etc.). Users on older CLI versions who explicitly pass `gemini-2.5-pro` or `gemini-3-pro-preview` still get Flash fallback on quota errors.

## ADR-015: Revert to `-p` Flag for Gemini CLI v0.29+
- **Date:** 2026-02-24
- **Status:** Accepted (supersedes ADR-006)
- **Context:** Gemini CLI v0.29.5 changed behavior again. The `--` separator (positional argument) now launches **interactive mode**, which expects stdin input. Since our MCP server spawns Gemini with `stdio: ["ignore", ...]`, stdin is closed, causing exit code 42: "No input provided via stdin." The `-p`/`--prompt` flag deprecation from v0.23 has been reversed â€” `-p` is now the correct way to trigger **non-interactive (headless) mode**.
- **Decision:** Revert `CLI.FLAGS.SEPARATOR = "--"` back to `CLI.FLAGS.PROMPT = "-p"` in `src/constants.ts`. Update both main and fallback code paths in `geminiExecutor.ts` to use `CLI.FLAGS.PROMPT`. This undoes ADR-006's change while keeping all other improvements from that audit (removed broken `@` quoting, etc.).
- **Consequences:** Fixes exit code 42 errors for users on Gemini CLI v0.29+. Users on older CLI versions (v0.23â€“v0.28) that still reject `-p` may need to upgrade their Gemini CLI. The `-p` flag is the officially supported non-interactive invocation method going forward.

## ADR-018: Claude Code Plugin â€” Hybrid Multi-Entrypoint Architecture
- **Date:** 2026-02-25
- **Status:** Accepted (implemented in ADR-027)
- **Context:** We want a Claude Code plugin for automated Gemini code reviews (pre-commit hook, Stop hook, on-demand skill, isolated subagent). The core question was whether to reuse code from the existing MCP server. Hooks run as shell commands and have no MCP client, so they can't call MCP tools directly.

- **Consequences:** Zero logic duplication. Hooks get the same model fallback, quota handling, and chunking as the MCP server. Plugin is mostly markdown/config (~3 files). The `ask-gemini-run` binary also serves as a standalone CLI for users who want Gemini access without MCP.

## ADR-019: Structured JSON Output via --output-format json
- **Date:** 2026-02-25
- **Status:** Accepted
- **Context:** The Gemini CLI supports `--output-format json` which returns `{ response, stats, error }` instead of raw text. Our MCP server was parsing raw text output, losing token usage stats and getting unstructured error messages. Since we always invoke the CLI programmatically, structured JSON is strictly better.
- **Decision:** Always pass `--output-format json` to the Gemini CLI. Parse JSON in `geminiExecutor.ts`, extract the `response` text, and append a one-line stats summary (input/output tokens, model). Fall back to raw text if JSON parsing fails (backward compat with older CLI versions). No schema changes â€” the improvement is transparent to MCP clients.
- **Consequences:** Users get token usage visibility. Errors are structured and more informative. Older Gemini CLI versions that don't support `--output-format` degrade gracefully to the current behavior.
- **Follow-up (2026-02-25):** Added `extractJson()` to handle CLI warning/debug lines before JSON object, and improved error handling for error responses that have only a code (no message).
- **Design doc:** [docs/plans/2026-02-25-structured-json-output-design.md](plans/2026-02-25-structured-json-output-design.md)

## ADR-017: Smithery CJS Compatibility â€” createSandboxServer Export
- **Date:** 2026-02-24
- **Status:** Accepted
- **Context:** Smithery bundles MCP servers with esbuild in CJS mode for capability scanning. Our ESM code uses `createRequire(import.meta.url)` to read `package.json` at runtime, but in CJS bundles `import.meta` is empty (undefined), causing the module to crash on load. This prevented Smithery from scanning tool/prompt schemas.
- **Decision:** (1) Wrap `createRequire(import.meta.url)` in a try/catch with hardcoded fallback values (`name: "ask-gemini-mcp"`, `version: "0.0.0"`). (2) Export a `createSandboxServer()` function that returns a configured `McpServer` with tools/prompts registered via stub handlers â€” Smithery's recommended pattern for servers that need special handling during scanning. (3) Separate CLI entry point (`src/cli.ts`) from library module (`src/index.ts`) â€” `index.ts` no longer auto-executes `main()`, preventing stdio conflicts when Smithery imports the module. `package.json` `bin` now points to `dist/cli.js`.
- **Consequences:** The module loads cleanly in both ESM (normal operation) and CJS (Smithery bundling) contexts. Smithery can scan tool schemas without invoking real Gemini CLI or triggering stdio transport. The fallback version "0.0.0" is only used in CJS bundles, not in production stdio operation.
